二、背景（Background）

TCP 自 1980 年代以来一直是互联网事实上的可靠传输协议。尽管经过数十年研究，TCP 在两个关键场景下仍表现不佳：(1) 由短流主导的服务（如 Web 搜索、电商），TCP 的「flow startup problem」使得大多数流在找到最佳拥塞窗口之前就结束了——保守的静态初始窗口（IW = 2、4 或 10）让短流浪费多个 RTT；(2) 由长流主导的服务（视频流、大文件下载），手工静态配置的拥塞控制（CC）方案不能适应时变的网络状况。

论文报告了一个真实数据：在百度移动搜索服务中，IW=10 时仍有 80% 以上的 TCP 流在会话结束时仍处于 slow start 阶段，平均流大小约 120 KB。Pantheon 等综述研究也表明，14 种主流 CC 方案（Tahoe、Reno、Cubic、BBR、PCC Vivace、Copa、Indigo 等）在 288 个合成网络条件上没有任何一种能始终最优——不同网络条件（带宽 1-200 Mbps，RTT 10-200 ms，丢包率 0-10%）下，最优 CC 都不一样。因此，工业界普遍采用的"一 CC 配到底"的实践是次优的。

在此之前，业界研究主要集中在三方面：(a) 设计新的 CC 算法；(b) 增大 IW（如 Google 把 IW 从 2-4 改成 10）；(c) 客户端测量网络状态做预测。但这些都没解决"如何根据时变网络条件动态选 IW 和 CC"这一根本问题。论文于是提出用强化学习（RL）在服务端侧做自适应配置。

三、目的（Purpose / Problems Solved）

  • 痛点 1 — 服务端无法直接测量 TCP 性能：传统 Web 服务器无法获得流的传输时间、丢包率等关键 RL 奖励信号（需客户端配合）。

    • 解决：修改 Linux 内核和 Nginx，实时采集数十亿条 TCP 流性能记录（传输时间、吞吐、丢包率、RTT）。

  • 痛点 2 — RL 在多变非连续网络条件下失效：网络条件随时空剧烈变化，细粒度用户（如 IP）样本量过少无法满足 RL 的"上下文连续性"假设。

    • 解决：按子网、ISP、省份等网络特征，自底向上聚合用户到最细粒度又能满足样本量和连续性的"用户组"，在组粒度上跑传统 model-free RL。

  • 痛点 3 — RL 在大决策空间中收敛慢：IW 取值 2-100+，CC 至少 14 种，brute-force 探索要等网络条件变了都还没找到最优。

    • 解决：对 IW 问题，根据"TCP 性能与 IW 的关系"先验设计 sliding-decision-space；对 CC 问题，线下训练神经网络模型，线上直接前向推理。

  • 痛点 4 — 短流与长流最优配置机制不同：短流结束在 slow start，IW 是关键；长流 IW 影响不大，CC 是关键。

解决：把 IW 与 CC 分成两条独立流水线分别处理。

四、核心原理（Principles）

TCP-RL 的系统总览可概括为"两套独立 RL 流水线在服务端并行运行"：

  • 短流侧（IW 配置）：在用户组（subnet/ISP/省份 + 城市）粒度上跑 Contextual Bandit/RLinucb 风格算法。状态 s = 组内最近窗口的网络条件（带宽、RTT、丢包率），动作 a = 候选 IW 值（2-100+），奖励 r = 该 IW 下该组短流的负平均传输时间。决策时用 sliding-decision-space 把搜索空间裁剪到当前最优 IW 附近的小窗口。
  • 长流侧（CC 配置）：在流粒度上跑 deep RL。线下训练一个神经网络，输入是网络条件（带宽、RTT、丢包率），输出是 14 种 CC 的预测吞吐/延迟分数；线上将 14 种候选 CC 在该流上各试一次，取短期表现最好的作为本期 CC，并通过 change-point detector 适应网络状况漂移。

关键数学：RL 收益建模

把 RL 形式化为一个上下文 bandit 过程。设用户组 g 在 t 时刻观测到状态 s_t（即网络条件），从动作空间 \mathcal{A} 选 a_t（IW 值），得到奖励 r_t = -T_t（T_t 为该流平均传输时间）。目标是最大化累计折扣奖励： \max_{\pi} \ \mathbb{E}\left[ \sum_{t} \gamma^t r_t \mid s_t, a_t=\pi(s_t) \right]

长流侧使用深度 Q 网络： Q^\pi(s,a) = \mathbb{E}\left[ r + \gamma \max_{a'} Q^\pi(s',a') \right] 用 replay buffer 和 target network 训练，避免不稳定。

与现有技术的差异：(1) 与 Remy/Indigo 等生成新 CC 的方法不同，TCP-RL 不发明新 CC，而是选已有 CC；(2) 与 Google 简单改 IW 静态值不同，TCP-RL 实时个性化 IW；(3) 之前 RL 用于视频 ABR/前端选择的工作是离散的、可枚举的，TCP-RL 面对的是大动作空间（IW 连续可达 100+）和非平稳网络条件，需要专门的 sliding-decision-space 技巧。

五、算法详解（Algorithm）

1. 输入 / 输出

  • 输入：服务端持续采集的 TCP 流记录（每条：五元组、传输时间、丢包率、RTT、字节数、起止时间戳）；以及与流对应的用户子网/ISP/省份。
  • 输出：
  • 对短流：用户组→最优 IW 的映射表。
  • 对长流：每个长流选择的具体 CC 方案（Cubic / BBR / Copa …）。

2. 核心模块

  • 内核级 TCP 测量模块：在 Linux kernel 中 hook TCP 状态转换（ESTABLISHED→FIN_WAIT1 等），把 (Tstart, Tend, bytes, rtt, loss) 写进 ring buffer；Nginx 侧加 module 异步消费。
  • 用户组聚合模块：bottom-up 聚类，按 subnet（/24）→ ISP → 省 → 国家逐级合并，保留满足样本量 ≥ N 且方差低于阈值的最小粒度。
  • Group-based RL for IW：在每个组内跑 RLinucb：$\hat{r}_{t,a} = \langle \theta_a, \phi(s_t) \rangle + \alpha \sqrt{\phi(s_t)^\top A_a^{-1} \phi(s_t)}$。
  • Deep RL for CC：线下训练 MLP，输入 5 维网络条件 + 14 维 CC one-hot（也可作为输出），输出预测吞吐/延迟。线上选择 top-1。
  • Sliding-decision-space：当前最优 IW 估计 $\hat{a}^_t$ 后，把 $\mathcal{A}$ 收缩到 $[\hat{a}^_t - k, \hat{a}^*_t + k]$。
  • Change-point detector：监控每组的奖励分布，CUSUM 检测漂移，触发重训。

3. 伪代码（短流 IW 配置主循环）

# Server side, for each user group g:
group_data = collect_tcp_flows_last_window()  # from kernel hook
state = extract_features(group_data)  # bandwidth, RTT, loss

# Sliding decision space centered at last best
if last_best_IW[g] is not None:
    action_space = clip(candidates, last_best_IW[g]-k, last_best_IW[g]+k)
else:
    action_space = full_candidates  # e.g. [2..100]

# RLinucb action selection
best_a = None; best_ucb = -inf
for a in action_space:
    ucb = theta[a] @ phi(state) + alpha * sqrt(phi(state) @ inv_A[a] @ phi(state))
    if ucb > best_ucb:
        best_ucb, best_a = ucb, a

# Set IW for next short flow in group g
set_group_IW(g, best_a)
last_best_IW[g] = best_a
# After flow finishes, observe reward and update RLinucb
reward = -flow_transmission_time
update_RLinucb(g, state, best_a, reward)

4. 关键数学

  • Linear UCB：$\hat{r}_{t,a} = x_t^\top \theta_a + \alpha \sqrt{x_t^\top A_a^{-1} x_t}$，其中 $A_a = \sum x_t x_t^\top$。
  • Bandit 后悔界：$\text{Reg}(T) \le O(\sqrt{d T \log T})$。
  • Deep RL 的 DQN 损失：$L(\theta) = \mathbb{E}\left[\left(r + \gamma \max_{a'} Q(s',a';\theta^-) - Q(s,a;\theta)\right)^2\right]$。

5. 复杂度分析

  • 训练复杂度：用户组 RLinucb 每次更新 $O(d^2)$；deep RL 模型离线训练 $O(N \cdot d_{nn})$（N = 样本数，$d_{nn}$ = 网络参数量）。
  • 推理复杂度：sliding decision space 裁剪到 $\pm k$ 个候选，每组每秒百万级流决策时间 < 1ms。
  • 存储：内核采样 + 定期聚合后落盘，全网每月约 TB 级。

6. 训练与推理

  • 训练目标：最大化累计吞吐/最小化传输时间。
  • 损失函数：上文 RLinucb 的 ridge regression 闭式解 + DQN 的 TD 损失。
  • 推理流程：内核采样 → Nginx 异步聚合 → 特征计算 → RL 选动作 → 注入到 socket option（TCP_NAGLE_OFF、SO_SNDBUF 调整）→ 下一条流沿用。

7. 示例

考虑一个 ISP=某省联通、子网 /24 的用户组，最近 5 分钟 RTT=40 ms、丢包率 0.5%。RLinucb 当前 θ 估计最优 IW=12，但置信区间宽，UCB 倾向于试 IW=24。试验 1000 条流后观测到传输时间比 IW=12 短 8%，更新 θ 后，sliding window 收缩到 IW ∈ [10, 30]。再观察 1 天，发现 5G 用户比例上升，RL 自动把最优 IW 推高到 22，整体平均传输时间下降 23%。

八、关键创新点（Key Innovations）

  • + 首次把 RL 同时用于 IW 和 CC 配置：之前 RL 在视频 ABR 等场景的部署只选离散的服务/算法，TCP-RL 第一次将其用于细粒度 TCP 参数（IW 连续值、CC 离散枚举），并已在一家顶级搜索引擎实生产部署超过一年。
  • + 自底向上用户组聚合满足 RL 上下文连续性：提出了从 /24 子网→ISP→省份逐级合并的策略，使样本量和连续性两个互相制约的目标同时满足，相比 Google 简单的全局 IW=10 显著更精细。
  • + Sliding-decision-space 加速 RL 收敛：利用"TCP 性能与 IW 关系是单峰"的先验，把大动作空间裁剪到当前最优附近的窗口，使得 RLinucb 可以在快速漂移的网络条件下仍然跟得上。
  • + 服务端纯测量 + 离线/在线混合训练：内核 + Nginx 双层采样使得奖励信号不需要客户端配合；对长流 CC 配置则采用"线下训练 + 线上推理"避免对生产网络做危险探索。
  • + 实测效果：相比 IW=10 的业界标准，短流平均传输时间下降 23%–29%；对 288 个网络条件上的 14 种 CC，长流性能 top-5 占比 85%，相比最优 CC 退化 ≤ 12% 的概率 90%。

九、实验与结果（Experiments）

  • 数据集：来自百度移动搜索服务 1 周的 TCP 流日志（约 120 KB/流，80%+ 在 slow start），以及测试床 288 个合成网络条件（带宽 1-200 Mbps，RTT 10-200 ms，丢包率 0-10%）。
  • Baseline：IW 方面：固定 IW=4、IW=10；CC 方面：14 种主流 CC（Tahoe、Reno、Cubic、BBR、PCC Vivace、Copa、Indigo 等）。
  • 主要指标：短流平均传输时间（ms）；长流吞吐 / 公平性 / 退化幅度。
  • 关键结果数字：
  • 短流：相比 IW=10，平均传输时间 降低 23%（线上 A/B 测试）至 29%（testbed）。
  • 长流：在 288 个网络条件下，TCP-RL 选出的 CC top-5 排名命中率 85%，相对该网络条件下最优 CC 退化 < 12% 的概率 90%。
  • 收敛性：sliding-decision-space 相比 brute-force RLinucb 在用户组样本数 < 200 时收敛速度提升约 3-5 倍。
  • 消融实验：
  • 关掉 sliding-decision-space：性能退化明显，尤其在用户组样本少时。
  • 把 group aggregation 改用 IP 粒度：很多组样本数不足 100，RL 几乎不收敛。
  • 把 deep RL 替换为随机选 CC：吞吐下降 30%+。
  • 效率分析：内核采样 overhead < 3% CPU（百度线上 A/B 测试），Nginx 异步聚合内存约 2 GB。

十、应用场景（Use Cases）

  • 大型 Web 服务 / 搜索引擎：百度移动搜索已落地，业内可直接借鉴部署。
  • CDN 边缘节点 / 视频点播：在 CDN 边缘按用户子网聚合跑 group RL，缩短首包时间。
  • 企业内部 SaaS 服务：把 IW/CC 自适应部署在公司网关出口，对来自不同运营商用户的请求差异化调参。
  • 云厂商的负载均衡前端：可作为前端的"网络栈增强"组件发布。
  • 5G / 高速光纤接入场景：IW=10 显然低估了高带宽用户的容量，TCP-RL 可将 IW 推高到 20-30。

十二、术语表（Glossary）

  • IW (Initial Window)：TCP 慢启动开始时发送方一次性可发的最大字节数，对应初始拥塞窗口 cwnd。
  • CC (Congestion Control)：TCP 拥塞控制算法，决定后续数据发送速率。
  • CWND：当前拥塞窗口大小。
  • RTT (Round-Trip Time)：往返时延。
  • Slow Start：TCP 慢启动阶段，每 RTT 指数级增长 cwnd。
  • Bandwidth-Delay Product (BDP)：带宽 × RTT，决定最优 cwnd。
  • RLinucb：Linear Upper Confidence Bound，一种 model-free 的 contextual bandit 算法。
  • Change-point Detection：检测数据分布发生突变的统计方法。
