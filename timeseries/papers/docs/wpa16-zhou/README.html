二、研究背景

2.1 物理活动研究的价值

人们每天大量时间在物理世界中活动、与他人互动。这些活动数据包含用户行为、偏好、态度和兴趣的丰富信息，对用户和企业都有巨大价值。具体应用包括：基于位置的社交网络、机会网络、内容分发、教育研究、流行病防控等。

2.2 校园环境的优势

大学校园是研究人群活动的天然实验场：

  • 高密度 WiFi 覆盖：清华校园 ~2700 个 Cisco 企业 AP，覆盖 114 栋建筑
  • 大规模用户群体：~45000 名学生 + ~12000 教职工
  • 高度活跃：每日 60000+ 独立设备，峰值 20000 同时连接
  • 多元场景：宿舍、教室、食堂、图书馆、体育馆等多种类型建筑

2.3 既有工作的不足

既有研究主要使用单源数据：

  • 仅 WiFi 数据（如 LiveLabs [14]）：缺乏用户属性（年级、性别）和细粒度行为（手机使用模式）
  • 仅移动 App 数据（如 StudentLife [22]）：规模小，缺乏位置广度和全人群覆盖

论文核心论点：把大规模 WiFi 数据与大规模手机 App 数据结合，才能估计"整校人群"的物理活动。这是 MobiCamp 的核心差异化。

2.4 核心挑战

  1. 移动轨迹覆盖率：传统 WiFi 关联数据在设备未连接 / 使用 rogue AP / 漫游时丢失大量位置信号
  2. MAC 地址 ↔ 用户身份映射：现代 Android/iOS 屏蔽 App 访问硬件标识符

三、论文目的与解决的问题

3.1 研究目的

构建一个校园级测试平台，融合大规模 WLAN 数据与大规模移动 App 数据，实现对校园人群物理活动的全面观测和分析。

四、核心原理（含数学公式）

4.1 四种 SNMP 数据源

数据源描述覆盖范围
trap 关联事件AP 关联/断开等实时事件已连接设备
probe RSSI设备探头扫描请求的 RSSI（即使 WiFi 未开启也可能发送）最广
rogue RSSIrogue AP 听到的设备数据包 RSSI未连接设备
connected RSSI已关联 AP 的数据包平均 RSSI已连接设备

4.2 AP Closeness 定义

AP 之间"距离"通过 scan group 中共现概率衡量：

$$C(\text{AP}_i, \text{AP}_j) = \frac{|\{sg \mid \text{AP}_i \in sg \land \text{AP}_j \in sg \land sg \in SG\}|}{|\{sg \mid \text{AP}_i \in sg \land sg \in SG\}|}$$

其中：

  • \(SG\) 是所有历史 scan group 集合
  • \(sg\) 是设备的连续扫描包 RSSI 记录集合
  • \(C(\text{AP}_i, \text{AP}_j)\) 表示给定 AP_i 已看到某设备时 AP_j 也能看到该设备的概率
  • 当 \(C = 0\) 表示两 AP 几乎不出现在同一 scan group

AP closeness 的计算不依赖任何人工标注的 AP 位置坐标或指纹图，完全从数据中自动学习。

4.3 Mobility Detection 算法原理

论文的定位算法分为两步：

  1. Fingerprint 快照生成：滑动窗口处理原始数据，剔除"远离"的 AP 记录
  2. 相似快照合并：当 75% 以上的 AP pair 是 close 的时，合并为一个大 fingerprint

两个 RSSI fingerprint 被判定为相似的条件：

$$\text{相似} \iff \frac{|\{(\text{AP}_a, \text{AP}_b) \mid C(\text{AP}_a, \text{AP}_b) > \text{threshold}\}|}{|\{(\text{AP}_a, \text{AP}_b)\}|} \ge 0.75$$

设备被认为"停留"（pause）在某个区域当且仅当其 RSSI fingerprint 在 ≥5 分钟内保持稳定。停留区间定义为该 merged fingerprint 内记录的最早和最晚时间。

4.4 设备分类规则

论文将非访客设备分为三种类型：

$$\text{type}(d) = \begin{cases} \text{Stationary} & \text{if } \text{single\_loc}(d) \ge 7\text{hrs} \land \text{ratio}_{\text{days}} \ge 60\% \\ \text{Laptop} & \text{if } \text{mobile}(d) \land \text{never\_transient}(d) \land \text{ratio}_{\text{days}} \ge 60\% \\ \text{Phone} & \text{otherwise} \end{cases}$$

访客（visitor）定义为 MAC 地址出现 < 5 天的设备。

4.5 Cross-Correlation 分析

论文使用交叉相关（cross-correlation）衡量两个建筑人流时间序列的相似性：

$$\rho_{XY}(\tau) = \frac{E[(X_t - \mu_X)(Y_{t+\tau} - \mu_Y)]}{\sigma_X \sigma_Y}$$

其中 \(\tau\) 是时延（lag），论文考虑 \(|\tau| \le 30\) 分钟。建筑 \(X\) 和 \(Y\) 的 flow similarity 定义为：

$$\text{FlowSimilarity}(X, Y) = \max_{|\tau| \le 30\text{min}} \rho_{XY}(\tau)$$

基于 flow similarity 矩阵，使用 hierarchical clustering（complete linkage）对所有 114 栋建筑进行聚类。

6.1 Mobility Detection 算法

6.1.1 伪代码

算法：Mobility Detection — 生成设备停留区间
输入：设备的 scan group 列表 SG,
      AP closeness 矩阵 C[AP][AP],
      滑动窗口长度 W = 5min,
      相似度阈值 θ_sim = 0.75,
      停留时长阈值 θ_pause = 5min
输出：停留区间列表 [(t_start, t_end), ...]

1: snapshots ← []
2: for each scan group sg in SG do
3:     // 生成 fingerprint 快照
4:     snap ← ∅
5:     for each record r in sg where r.rssi != null do
6:         // 如果 AP 不远离当前已知 AP，加入 fingerprint
7:         if ¬is_far_away(r.ap, current_aps, C, θ_close) then
8:             snap ← snap ∪ {(r.ap, r.rssi, r.time)}
9:         end if
10:    end for
11:    if snap ≠ ∅ then
12:        snapshots ← snapshots ∪ [snap]
13:    end if
14: end for
15: // 第 2 步：合并相似快照
16: merged ← []
17: cur ← snapshots[0]
18: for each snap in snapshots[1:] do
19:     if is_similar(cur, snap, θ_sim, C) then
20:         cur ← merge(cur, snap)
21:     else
22:         merged ← merged ∪ [cur]
23:         cur ← snap
24:     end if
25: end for
26: merged ← merged ∪ [cur]
27: // 第 3 步：提取停留区间
28: intervals ← []
29: for each fingerprint f in merged do
30:     duration ← f.t_end - f.t_start
31:     if duration ≥ θ_pause then
32:         intervals ← intervals ∪ [(f.t_start, f.t_end)]
33:     end if
34: end for
35: return intervals

辅助函数 is_far_away(ap_i, ap_set, C, θ_close):
    for each ap_j in ap_set:
        if C[ap_i][ap_j] < θ_close:
            return True
    return False

辅助函数 is_similar(f1, f2, θ_sim, C):
    count_close ← 0
    count_total ← |f1| × |f2|
    for each (ap_a, rssi_a) in f1:
        for each (ap_b, rssi_b) in f2:
            if C[ap_a][ap_b] ≥ θ_close:
                count_close ← count_close + 1
    return (count_close / count_total) ≥ θ_sim

8.5 应用 1：建筑异质性分析

论文分析了两栋有代表性建筑的人流模式：

  • ZJ17（宿舍公寓）：午间和夜间人员聚集，白天人员离开
  • 6Jiao（教学楼）：每天 5 节课时段有人流高峰

层次聚类结果显示：

  • 宿舍自然聚类（ZJ14-18 vs ZJ19-23 区分了博士生和留学生）
  • 食堂自然聚类（用餐时段有独特的人流高峰）
  • 高频教室（3Jiao, 4Jiao, 5Jiao, 6Jiao）共享相似人流模式

8.6 应用 2：弱社交关系与手机使用

基于共现网络定义社交关系强度：

  • 共现频次高 → Strong Tie
  • 共现频次低 → Weak Tie

关键发现：

  • 短会话（≤30s）：弱关系 vs 强关系的手机使用频率分布有明显差异
  • 弱关系下：手机使用更突发（bursty），表示用户分心更多
  • 强关系下：手机使用更均匀，表示用户更多参与社交互动

这一发现表明，手机使用的突发性可以作为同类社交关系强度的一个重要指标。

十二、局限性与未来工作

12.1 局限性

  1. 室外覆盖有限：WLAN 信号主要在室内，设备离开建筑物时丢失追踪

  3. MAC 随机化趋势增强：iOS/Windows 等系统日益普及 MAC 随机化，削弱追踪能力
  4. App 样本偏差：2,300 主动贡献者可能不代表全校 45,000 学生的整体分布
  5. 设备分类规则较简单：基于阈值规则，对边界情况（如工作站）区分能力有限
  6. 11 周观察期：尚未涉及学期内不同阶段、考试周等特殊时期的对比

12.2 未来工作方向

  1. 跨学期长期研究：分析学生行为随学期进展的变化
  2. 室外定位改进：融合蜂窝网络数据弥补 WLAN 室外覆盖
  3. MAC 随机化对抗：利用行为模式识别同一设备（即使 MAC 变化）
  4. 隐私保护机制：差分隐私、数据脱敏等
  5. 更多应用场景：疫情传播建模、校园安全、节能减排

十三、总结与启示

13.1 论文总结

MobiCamp 是当时已知最大规模之一的校园人群物理活动研究测试平台。它通过创新的 4 类 SNMP 数据融合和免标注 mobility detection 算法，实现了比既有工作更广的覆盖范围（+71%），并通过 WLAN 轨迹与移动 App 数据的关联首次实现"整校人群"物理活动的估计。

13.2 关键启示

  • 多数据源融合至关重要：单一数据源（联事件）覆盖率不足，利用 probe 等辅助数据大幅提升
  • 免标注定位可行：AP closeness 算法在 AP 密集部署的校园场景中有效
  • 跨数据源关联是关键工程：现代 OS 对硬件标识符的访问限制需要创造性地解决
  • 人群移动模式与建筑功能高度相关：建筑聚类结果与真实功能一致性高
  • 手机使用行为反映社交关系：弱关系下手机使用更突发，可以作为社交强度的指标

13.3 对 AIOps 和可观测性的启示

  • 利用现有基础设施（SNMP）进行大规模数据采集，最小化额外部署
  • 数据融合策略：多种覆盖不完整的信号源可以互补形成完整视图
  • 免标注算法降低了运维成本
  • 设备和用户的关联（MAC-账号映射）是可观测性领域的常见挑战
  • WLAN 数据可用于网络性能优化（WifiSeer）

14.1 SNMP 数据采集架构

MobiCamp 通过 Cisco WLC（Wireless LAN Controller）的 SNMP 接口拉取数据，使用 snmpwalk + 自定义 Python 解析器。关键 OID：

  • 1.3.6.1.4.1.9.9.599.1.3.1.1.1 — cLApIfReceivedPowerLevel（Connected RSSI）
  • 1.3.6.1.4.1.9.9.599.1.3.1.1.3 — cLApDot11IfForeignPowerLevel（Probe RSSI）
  • 1.3.6.1.4.1.9.9.599.1.3.1.1.2 — cLApDot11IfRoguePowerLevel（Rogue RSSI）
  • 1.3.6.1.4.1.9.9.599.1.3.1.1.5 — cLApIfTrap（SNMP Trap 关联/解关联事件）

采集频率为 5 秒一次，全校 2,786 个 AP 同时轮询，单次 snmpwalk 耗时约 3 分钟。为避免 WLC 过载，论文使用 8 台采集服务器并行。

14.2 TUNet App 的 MAC 查询 Web API

Android 6.0+ 和 iOS 屏蔽普通 App 访问 getMacAddress()。TUNet 在校园 IT 后端暴露一个 Web API：

Response:

  "device_id": "tsinghua-device-uuid",
  "mac_address": "AA:BB:CC:DD:EE:FF",

App 在注册时调用此接口，校园 IT 后端从 RADIUS 计费系统查询该 token 对应的设备 MAC。论文期间 8,600 Android + 6,500 iOS 设备成功绑定。

十五、AP Closeness 算法细节补充

15.1 Scan Group 构建

一个 scan group 是同一时间窗（5 秒）内多个 AP 同时"听到"同一 MAC 的事件集合。构建算法：

def build_scan_groups(events, window=5):
    """events: [(mac, ap, rssi, ts)]"""
    groups = []
    by_mac = groupby(events, key=lambda e: e.mac)
    for mac, evs in by_mac:
        evs = sorted(evs, key=lambda e: e.ts)
        i = 0
        while i < len(evs):
            j = i
            while j < len(evs) and evs[j].ts - evs[i].ts <= window:
                j += 1
            aps = set(e.ap for e in evs[i:j])
            if len(aps) >= 2:  # 至少 2 个 AP 听到
                groups.append(ScanGroup(mac, aps, evs[i].ts, evs[j-1].ts))
            i = j
    return groups

15.2 AP Closeness 矩阵计算

论文 Eq. 1 中分母 $\sum \mathbf{1}\{\text{AP}_i \in sg\}$ 实际是 AP_i 出现的 scan group 数，分子是两 AP 同时出现的 scan group 数。得到 C(i,j) 是对称矩阵，论文取 0.5 阈值判定 close。

十六、Mobility Detection 详细步骤

  1. 对设备 d 的每条 scan group，构建 RSSI fingerprint $F = \{(\text{AP}, \text{RSSI})\}$。
  2. 滑动 5 分钟窗口，把时间连续且 RSSI 相似（>75% AP pair close）的 fingerprint 合并。
  3. 合并后的 fingerprint 视为同一"位置事件"。
  4. 若位置事件持续 $\ge 5$ 分钟，判为 pause interval（停留）；否则为 transient（经过）。
  5. 输出所有 pause interval，作为设备移动轨迹的"锚点"。

十九、应用案例 2：弱关系与手机使用

19.1 共现网络构建

两设备 d1, d2 在同一 pause interval 内同时出现 $\ge N$ 次（论文 N=5），视为有共现关系。关系强度 $W(d1,d2) = \sum_{interval} \text{duration}$。$W \ge T$ 视为 strong tie，$W < T$ 视为 weak tie。

19.2 手机使用模式

对每个设备的 App 启动时间序列，计算"burstiness"：

$$ B = \frac{\sigma_\tau - \mu_\tau}{\sigma_\tau + \mu_\tau} $$

其中 $\tau$ 是连续 App 启动间隔。B 越接近 1 表示越突发（bursty），接近 -1 表示越均匀。

19.3 关键发现

  • 弱关系共现（停留 ≤30s）：burstiness 更高（更短促的手机使用）。
  • 强关系共现（停留 >5min）：burstiness 更低（更连续的手机使用，可能在共同看屏幕）。
  • 暗示：与陌生人短暂相遇时人倾向"刷手机缓解尴尬"，与熟人长时间相处时反而放松。

三十八、思考题解答提示

Q1: AP closeness 与 AP 物理距离的关系

AP closeness 基于 scan group 共现频次，反映了"信号重叠覆盖"的程度。理论上物理距离越近，closeness 越高，但实际：

  • 物理近但中间有墙 → RSSI 衰减大，closeness 中等。
  • 物理远但走廊相通 → RSSI 仍可接收，closeness 反而高。
  • 同楼层不同区域 → closeness 中等。

closeness 实际反映"信号可达性"，比物理距离更适合定位。

Q2: 5 分钟 pause 阈值的合理性

5 分钟基于人类行为学：

  • 路过教室门口：< 30s
  • 教室听课：≥ 45 分钟
  • 咖啡厅买咖啡：1-3 分钟
  • 食堂吃饭：≥ 15 分钟
  • 等电梯：30s-1 分钟

5 分钟可过滤"短暂停留"，保留"真实活动"。论文 §3.2 给出选择理由。

Q3: complete linkage vs single linkage

建筑聚类目标是"识别相似人流模式"，complete linkage 倾向于发现紧凑球形簇；single linkage 倾向于拉长链状簇。建筑间相似度基于 cross correlation 矩阵，论文发现 complete linkage 聚类结果更稳定。

Q4: 推广到企业园区或医院

需要：

  • BLE/UWB 信标补充 RSSI；
  • 门禁系统关联人-身份；
  • OA/HR 系统关联人-属性；
  • 考虑隐私合规（GDPR、HIPAA）。

Q5: iOS 14+ Private WiFi Address

iOS 14+ 默认每 SSID 随机化 MAC，TUNet 的 MAC 查询 API 不再生效。解决：

  • 让用户主动关闭 Private Address（牺牲隐私）；
  • 改用 BLE 5.0 信标（不受 MAC 随机化影响）；
  • 用 iOS 设备指纹（屏幕尺寸、字体列表）识别，但稳定性差。

  • 免标注算法是工程落地的关键：AP closeness 不需要 3D 坐标或 fingerprinting 标注。

四十四、附录：关键算法的复杂度分析

44.1 AP closeness 矩阵计算

设 $N$ 为 scan group 总数，$M$ 为 AP 总数。算法对每对 AP 统计共现的 scan group 数，复杂度为 $O(N \cdot M^2)$。论文中 $N \approx 3.6 \times 10^8$，$M = 2786$，$M^2 \approx 7.7 \times 10^6$，因此 $N \cdot M^2 \approx 2.8 \times 10^{15}$，单台服务器无法在 1 小时内完成，必须用 Spark 分布式并行：

  • 按 AP 分区，每个 partition 处理一部分 AP；
  • 使用 aggregateByKey + mapValues 优化 Spark 任务；
  • 使用 bitmap 加速 AP 是否在同一 scan group 的查询；
  • 论文实际：50 台 Spark worker 节点 1.5 小时完成。

44.2 Mobility detection

单设备 $d$ 复杂度 $O(S_d)$，$S_d$ 是设备 $d$ 的 scan group 数。整体 $O(\sum S_d) = O(N)$。论文中 $N = 3.6 \times 10^8$，单机 1 小时内可处理。

44.3 建筑聚类

设 $B$ 为建筑数（论文 114）。两两 cross correlation 复杂度 $O(B^2 \cdot L)$，$L$ 是时间序列长度。论文 $L = 30 \times 48 = 1440$，$B^2 = 12996$，因此总复杂度 $O(1.9 \times 10^7)$，可单机完成。Hierarchical clustering 额外 $O(B^3) = 1.5 \times 10^6$，可忽略。
