二、背景（Background）

随着 ICT 与车载传感技术发展，智能网联车（IoV）成为汽车产业新标准。每辆智能车都搭载大量传感器（ADAS、BMS、DCU 等），持续采集车辆运行、驾驶行为、道路信息，并通过 5G 等无线技术上传到云端。云端数据库为 TSP（Telematics Service Provider）、整车厂、自动驾驶厂商、政府监管等多方提供数据管理能力——当车辆出故障时，TSP 可远程通过云端数据库查询运行数据做诊断。

IoV 数据本质是多维时序数据，与 IoT 和 DevOps 指标类似，但有三个独特之处：

  1. 单设备指标规模差异巨大：DevOps 单服务器几十到几百指标，IoT 设备单设备几百指标，IoV 单车 2500+ 指标（ADAS、BMS、DCU 等子组件多）。
  2. 写吞吐极高：B 公司一天可达 80TB、平均 1GB/s，A 公司约 721MB/s。
  3. "update-style writing"：单车不同子组件独立上传数据，timestamp 在数据库侧对齐到秒/分钟级；当某秒已有数据，本次写入是"在该行追加新指标"而不是"插入新行"。

传统数据库的痛点：

  • InfluxDB（时序库）：单指标管理；查询大量指标需要逐个独立检索 → I/O 爆炸。
  • HBase（宽列库）：每个数据点视为一个 key-value；查询大量指标 → CPU 大量花在 key 比较上。
  • MongoDB（文档库）：把同一 timestamp 的所有指标存为文档，看似解决多指标问题；但 update-style 写入需要"读出文档 → 修改 → 写回"，小指标子集查询要 fetch 整篇文档 → 读放大严重。

Lindorm-UWC 是阿里云在 Lindorm 多模数据库之上专门为 IoV 场景设计的"超宽列"存储引擎，从 2019 年起已在阿里云生产环境上服务多家 IoV 企业客户，管理数十 PB 数据、千万级 QPS。

三、目的（Purpose / Problems Solved）

论文从 empirical study 中明确总结三大挑战：

  • 挑战 C1：极致的写入吞吐。IoV 系统数据流量 >1GB/s、80TB/天，对数据库写吞吐要求极高。Lindorm-UWC 用 LSM-tree + append-only + 多指标合并为单列，让一次写入承载尽可能多的指标。
  • 挑战 C2：单车辆数千指标 + update-style 写入。一次写入不会包含所有指标，且多次写入"按时间对齐后更新同一行"。Lindorm-UWC 放弃 update-in-place，每次写入都是 append 到当前行对应时间桶的最新状态，避免 HBase/MongoDB 那种"先读再改"的代价。
  • 挑战 C3：多样化查询模式。查询可能涉及大量指标（综合分析）也可能仅几个指标（远程故障诊断），需要数据库既支持"少量指标快速查"也支持"全部指标批量拉"。Lindorm-UWC 同时维护行存+列存双格式，让不同查询模式走最优路径。

四、核心原理（Principles）

Lindorm-UWC 整体架构包含三层：

  • 分布式层：按"车辆 ID + 时间"对数据进行分区；支持自动负载均衡。
  • 超宽列存储引擎：每个数据分区用独立的 LSM-tree 引擎。两条关键设计：
    • 多指标合并为单列：一次写入请求中的多个指标值合并到一个列里，避免在写路径上对不同指标分别建索引和分组。
    • append-only：每次写请求以 append 方式追加到对应车辆-时间桶的最新数据单元，不做 in-place update。
  • 冷热数据分离：把访问频率低的数据 offload 到廉价存储介质（OSS、HDD），热数据留在高性能介质（NVMe SSD），降低单位存储成本。

读路径：

  • 双格式存储：磁盘上同时维护行存 + 列存两种格式，根据查询的"指标数 vs 时间跨度"选择最合适的读路径：
    • 少量指标 + 长时间跨度 → 列存（只读需要的列，I/O 最小）。
    • 大量指标 + 短时间跨度 → 行存（一次读多列）。
  • 减少读放大：通过双格式让"读不必要数据"最小化。

与现有方案差异：

  • vs InfluxDB：本方法按"车辆"管理多指标，而不是按"单指标"管理，查询大量指标不再需要 N 次独立检索。
  • vs HBase：本方法把多指标合并到一列，写入时不需要为每个数据点建 KV 条目；查询时也不需要大量 key 比较。
  • vs MongoDB：本方法 append-only + 双格式，避免"读整篇文档再修改"；少量指标查询可直接走列存，不读全文档。

数学上，论文没有特别强调新颖的数学公式，更多是工程架构创新。其核心数学度量：

  • 写吞吐：writes/s，Lindorm-UWC 在 100–5000 metrics/write 范围内均保持最高吞吐。
  • 查询延迟：p99 延迟在高并发下仍可控。
  • 存储压缩比：相对基线的压缩率提升。

五、算法详解（Algorithm）

1. 输入 / 输出

  • 输入：IoV 设备上传的 (vehicle_id, timestamp, metrics) 三元组。
  • 输出：低延迟写入、灵活的 (vehicle_id, time_range, metric_set) 查询。

2. 核心模块

  • Distributed Coordinator：分布式协调；按 vehicle_id + 时间分区。
  • Storage Engine (Ultra-Wide-Column)：基于 LSM-tree 的单分区内存储引擎。
  • Multi-metric Merge：把多指标值合并为单列存储。
  • Append-only Writer：append-only 写，不做 update-in-place。
  • Dual-format Storage：行存 + 列存双格式。
  • Cold/Hot Tier：冷热数据分离存储。
  • Query Router：根据查询的指标数与时间跨度选择行/列路径。

    # 避免按指标分别建索引

# === Append-only 写入 ===
def append_write(partition, vehicle_id, timestamp, metrics):
    # 找到 vehicle_id 对应时间桶
    bucket = partition.get_or_create_bucket(vehicle_id, timestamp)
    # 直接 append，不做 in-place update
    return bucket.append(merge_metrics_to_single_column(metrics))

# === 双格式读取 ===
def read(partition, vehicle_id, time_range, metric_set):
    if len(metric_set) <= THRESHOLD or time_range is wide:
        # 少量指标 / 长时间 → 列存
        return columnar_read(partition, vehicle_id, time_range, metric_set)
    else:
        # 多指标 / 短时间 → 行存
        return row_read(partition, vehicle_id, time_range, metric_set)

4. 关键数学

  • 写吞吐（Table 4）：

$$\mathrm{WriteThroughput} = \frac{\#\mathrm{writes}}{\mathrm{second}}$$

论文在 #Metrics=100 时 Lindorm-UWC 写吞吐 ≥200K writes/s（论文实测），MongoDB 85K、HBase 199K、InfluxDB 156K。

  • 查询延迟：

$$L_{p99} = \mathrm{P99\ of\ query\ latency}$$

  • 存储压缩比：

$$R = \frac{\mathrm{raw\ size}}{\mathrm{compressed\ size}}$$

5. 复杂度分析

论文未给严格算法复杂度，更多关注 I/O 路径优化：一次写入的 I/O 量与"指标数 × 写入频次"相关；查询的 I/O 量与"返回指标数"相关。Lindorm-UWC 的双格式让两种查询模式都能保持 I/O 与"实际需要的数据量"成正比，避免读放大。

6. 训练与推理

这是数据库系统论文，没有"训练"环节。系统服务相关参数：

  • 分布式分区数：自动扩展。
  • 列存/行存选择阈值：可配置。
  • 冷热数据迁移策略：基于访问频率。

八、关键创新点（Key Innovations）

  • + 第一次 IoV 工作负载系统化 empirical study：从三家国内主要 IoV 企业（A/B/C）拿到月级写日志 + 周级查询日志，揭示单车 2500+ 指标、update-style writing、单车辆查询为主等关键特征，让后续的数据库设计有数据支撑。
  • + 超宽列（Ultra-Wide-Column）存储引擎：突破"每指标一列"的传统时序库设计，把一次写入的多个指标合并为单列存储，避免写路径上对不同指标分别建索引和分组。
  • + Append-only 写入策略：放弃 update-in-place，每次写都是"追加到当前车辆-时间桶的最新数据"，避免 HBase/MongoDB 那种"读出-修改-写回"的开销；与 LSM-tree 的天然 append 特性契合。
  • + 行存+列存双格式按需读取：根据查询的"指标数 vs 时间跨度"自动选择读路径，让"少量指标快速查"和"大量指标批量拉"两种模式都能保持低 I/O。
  • + 冷热数据自动分层：热数据留 NVMe、冷数据 offload 到廉价介质，降低单位存储成本。论文部署在阿里云 Lindorm 上，已管理数十 PB 数据。
  • + 79% 到数量级写吞吐提升：Table 4 显示 Lindorm-UWC 在 #Metrics=100 时写吞吐 200K+/s，相对 MongoDB 85K、HBase 199K、InfluxDB 156K 都有显著优势，特别是 metric 数增加时优势更明显（#Metrics=5000 时领先一个数量级）。

九、实验与结果（Experiments）

  • 数据集 / Workload：基于三家 IoV 公司（A/B/C）的月级写日志、周级查询日志构建的 benchmark suite。
  • Baseline：MongoDB、HBase、InfluxDB。
  • 关键结果数字：
    • 写吞吐（Table 4）：
      • #Metrics=100：Lindorm-UWC > 200K writes/s；HBase 199K、InfluxDB 156K、MongoDB 85K。
      • #Metrics=5000：Lindorm-UWC 仍保持优势（具体数字看 Table 4 后续行），相对基线领先 79%–一个数量级。
    • 数据吞吐：A 公司 560–950MB/s（平均 721MB/s）；B 公司 >800MB/s（平均 907MB/s，峰值 1.68GB/s）；C 公司 ~55MB/s。
    • 查询性能：单车辆查询为主（A 100%、B 100%、C 99.79%）；Lindorm-UWC 在少量指标和大量指标两种查询模式下均保持高并发低延迟。
    • 存储成本：通过冷热分层，存储成本显著低于全 SSD 方案。
    • 多样性查询：A 的查询指标数 ≤130（占总数 2999 的 <4.4%），验证双格式设计的合理性。
  • 消融实验：论文未严格做 ablation（架构性工作），但 Table 4 通过变化 #Metrics 隐式验证多指标合并设计的优势。
  • 效率分析：写吞吐随 #Metrics 增加的优势扩大（双格式 + 合并列的扩展性）。
  • 工业部署：2019 年起服务阿里云 IoV 企业客户，管理数十 PB 数据、千万级 QPS。

十、应用场景（Use Cases）

  • 智能网联车数据管理：单车 2500+ 传感器、1GB/s 写吞吐的工业场景核心数据库。
  • 车队远程诊断：TSP 工程师按车辆 ID + 时间段 + 几个特定指标快速查询数据。
  • 驾驶行为分析：按车辆 ID + 长时间跨度 + 全部指标的批量分析（行存路径）。
  • 故障回溯：按工单号查某辆车的多个系统在故障时刻前后的指标数据。
  • 政府监管数据上报：批量导出某时间段某区域所有车辆的运行数据。
  • 云原生时序存储底座：Lindorm-UWC 的架构思想可推广到任何"高维度 + 高并发写 + 灵活查询"的工业场景。
