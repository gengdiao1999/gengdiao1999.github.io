二、研究背景

2.1 WiFi 延迟的重要性

WiFi 延迟是影响现代实时交互应用（如在线游戏、网页浏览、直播协作）的关键因素：

  • 20ms 的最后一跳延迟可导致约 2-3 秒的网页加载时间
  • 这超出了 47% 用户的期望
  • WiFi 最后一跳延迟往往是端到端延迟的主导因素

2.2 测量 WiFi 延迟的挑战

  • WiFi 延迟不在常见无线产品的报告数据中
  • 基于嗅探器的测量方法成本过高
  • 修改 AP 固件/软件需要较长时间才能被商业采纳
  • 标准 ping 方法受设备节能模式影响

三、论文目的与解决的问题

WiFiSeer 旨在解决以下问题：

  • 大规模 EWLAN 中准确测量 WiFi 延迟
  • 建立 WiFi 延迟与无线因素之间的复杂关系模型
  • 提供三种实际应用：理解因素影响、诊断高延迟、选择低延迟 AP
  • 部署实际优化方案改善 WiFi 延迟

四、核心原理

4.1 ping2 测量原理

ping2 从服务器向用户设备发送两次连续 ping：

  • 第一次 ping：唤醒设备（WiFi 芯片 + CPU）
  • 第二次 ping：在设备活跃模式期间测量真实 WiFi 延迟

关键假设：设备默认活跃模式时间（Android ~200ms, iPhone ~100ms）通常大于 RTT。

六、关键算法详解

6.1 随机森林训练（用于 AP 选择）

算法：随机森林训练（应用3）
输入：WiFi 延迟数据集 D, WiFi 因素特征 X
输出：随机森林分类器

1: 标签化：将延迟分为 slow/EX slow 和 fast/EX fast
2: 欠采样处理类别不平衡
3: for t = 1 to T do
4:     D_t ← BootstrapSample(D)
5:     tree_t ← BuildDecisionTree(D_t, m=sqrt(|features|))
6: end for
7: 返回 {tree_1, ..., tree_T}

6.2 AGE AP 选择算法

算法：AGE AP 选择
输入：设备扫描到的 AP 列表, SNMP 因素数据
输出：推荐关联的 AP（或 Null）

1: for each AP in AP_list do
2:     factors ← GetSNMPFactors(AP)
3:     prediction ← RandomForest.predict(factors)
4:     if prediction == fast/EX fast then
5:         AP.is_good ← True
6:         AP.likelihood ← prediction_probability
7:     else
8:         AP.is_good ← False
9:     end if
10: end for
11: if current_AP.is_good == False then
12:     best_AP ← AP with highest likelihood among good APs
13:     return best_AP
14: else
15:     return Null  // 无需切换
16: end if

复杂度分析：O(|AP_list| × T × depth)，其中 T 是树的数量，depth 是平均深度。对于典型场景（5-10 个 AP，100 棵树），推理时间在毫秒级。

1. ping2 方法的详细技术分析

ping2 是 WiFiSeer 的核心创新，通过双 ping 机制克服设备节能模式的干扰。

  • 第一次 ping 作为唤醒信号，避免 WiFi 芯片和 CPU 的唤醒延迟
  • Android 默认活跃时间约 200ms，iPhone 约 100ms
  • 97% 的 RTT 小于 100ms，因此第二次 ping 通常在活跃窗口内完成
  • 有线部分延迟可忽略（99分位 <1ms）
  • 实现为 C++ 多线程，支持 15000+ 并发设备
  • 服务器开销极低：<10% CPU, <1MB/s 带宽
  • 电池消耗在极端 24 小时测试中为 7-10%
  • 实际使用中电池消耗更低，因为用户设备本身经常被 App 流量唤醒

2. 决策树模型中关键阈值的工程意义

决策树模型揭示了多个对 WiFi 延迟有显著影响的阈值，这些阈值可以直接指导网络优化决策。

  • CU > 47.5% 是重负载问题的标志，EX slow 延迟仅在此条件下出现
  • CU > 47.5% 定量验证了之前建议的'信道利用率应低于 50%'的经验法则
  • 设备数 > 35.5（低 CU 下）表示本地竞争问题
  • SNR 在 21.5-25.5 dB 是决策树中的关键分裂点
  • 这些阈值来自数据驱动的分析，而非理论推导
  • 不同环境可能需要不同的阈值

3. 5 GHz 频段利用不足的原因分析

尽管 5 GHz 提供更低的延迟，但大量双频设备仍然连接到 2.4 GHz。

  • 设备默认 AP 选择基于 RSSI，5 GHz 信号衰减更快
  • Cisco 的延迟探测响应机制试图引导设备到 5 GHz
  • 但设备仍可通过监听信标发现 2.4 GHz
  • 厂商之间缺乏协作导致机制失效
  • 解决方案：添加 5 GHz 专用 SSID

4. AGE 系统的详细设计决策

AGE 的设计在多个方面做了权衡，以平衡效果和用户体验。

  • 分类阈值设为 0.6（高于默认 0.5），以减少误切
  • 使用 0.6 阈值基于 10-fold 交叉验证的最佳结果
  • 仅当当前 AP 为慢速且存在快速 AP 时才推荐切换
  • 用户需要确认重关联，保持控制权
  • 不使用 ping2 进行在线测量（省电），仅用 SNMP 因素预测

扩展分析 1：WiFi 延迟分布的深度分析

WiFi 延迟的多模态分布

WiFi 延迟不是简单的单峰分布，而是呈现多模态特征：

  • 快速模式（EX fast/fast）：低 CU、少设备、高 SNR
  • 慢速模式（slow）：中等 CU、适中的设备数
  • 极慢模式（EX slow）：高 CU、多设备、低 SNR 或高干扰

这种多模态分布使得简单的统计模型（如回归）难以准确建模。

长尾分布的工程影响

长尾分布意味着：

  • 虽然大多数时候延迟可接受，但偶尔的延迟爆发严重影响用户体验
  • 75% 的设备每周至少经历一次超过 1 分钟的 EX slow 延迟
  • 约 20% 的延迟属于 slow 或 EX slow 类别
  • 这些长尾延迟是用户抱怨的主要来源

扩展分析 2：ping2 方法的理论分析

ping2 的数学建模

设设备节能模式的唤醒时间为 \(T_w\)，两次 ping 的间隔为 \(T_i\)，RTT 为 \(T_r\)。

ping2 测量准确的充分条件：\(T_w > T_r\)

在清华校园网中：

  • Android 设备 \(T_w \approx 200ms\)
  • iPhone 设备 \(T_w \approx 100ms\)
  • 98% 的 RTT < 200ms
  • 97% 的 RTT < 100ms

因此条件在绝大多数情况下满足。

ping2 的误差分析

ping2 的测量误差来源：

  • 有线部分延迟（99分位 <1ms，可忽略）
  • 第二次 ping 仍可能受节能模式影响（概率极低）
  • ping 包的处理延迟（通常 <1ms）

总体误差估计 < 2ms，对于 WiFi 延迟测量来说足够精确。

扩展分析 3：5 GHz vs 2.4 GHz 的详细对比

5 GHz 的技术优势

  • 更多可用信道（中国：36/40/44/48/52/56/60/64/149/153/157/161/165）
  • 更少的干扰源（微波炉、蓝牙等不使用 5 GHz）
  • 更低的信道利用率（更少的设备使用 5 GHz）
  • 支持更宽的信道带宽（80/160 MHz vs 20/40 MHz）

5 GHz 的劣势

  • 信号衰减更快（穿墙能力弱）
  • 覆盖范围较小
  • 设备兼容性问题（老旧设备不支持 5 GHz）
  • RSSI 通常低于 2.4 GHz（导致默认 AP 选择偏向 2.4 GHz）

WiFiSeer 的发现

  • 5 GHz 平均延迟比 2.4 GHz 低 35%
  • 双频设备中 50%+ 仍连接 2.4 GHz
  • RSSI 偏好的 AP 选择是主要原因
  • 添加 5 GHz 专用 SSID 后连接数持续增长
