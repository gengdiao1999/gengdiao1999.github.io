二、背景

大规模软件系统的故障日益频繁且相互关联，日志是最直接、最细粒度的故障证据。然而日志数量庞大、语义模糊、噪声多，传统 RCA（根因分析）方法难以跟上系统演化的速度。基于规则的方法（Ladra、LogRule）需要人工维护规则，覆盖不全；基于机器学习的方法（Log3C、LogRCA）依赖浅层特征或服务指标，难以泛化；基于知识图谱的方法（LogKG）能给出结构化因果链，但实体抽取依赖固定规则，容易遗漏新模式。LLM 出现后，LogPrompt、LogGPT、LogRAG 等方法带来新希望，但直接调用存在幻觉、缺乏领域知识等问题。

三、目的

  • 痛点 1：LLM 提取的实体存在大量语义冗余 → 方案：在实体表示层做嵌入聚类+规范化去重。
  • 痛点 2：KG 依赖固定规则抽取，可能遗漏关键故障实体 → 方案：在线阶段用 LLM 总结日志+实体抽取并通过嵌入相似度召回。
  • 痛点 3：纯 LLM 方法缺乏结构化因果推理 → 方案：用 KG 检索结果构造 context-enhanced prompts。
  • 痛点 4：评估缺乏工业级数据集 → 引入 2 个真实工业数据集共 2,764 故障案例。

四、核心原理

AetherLog 由离线 KG 构造 + 在线 RCA 推理两条管线组成。离线阶段用 LLM 在弱监督下抽取实体与关系，然后做嵌入聚类与规范化，得到紧凑、语义一致的 KG。在线阶段 LLM 抽取日志摘要与实体，调用嵌入检索找到最相似节点，注入到 context-enhanced prompt 中，输出可解释的根因。

关键概念：

  • Label-Aware Prompting：把 root cause label 作为弱监督。
  • BigLog Embedding：使用专门在大型日志语料上预训练的编码器。
  • Density-Based Clustering：使用 DBSCAN，$\epsilon=0.5$ 由 k-distance elbow 确定。
  • Context-Aware Recall：在线阶段用嵌入相似度召回最相关节点。
  • Context-Enhanced Prompt：将召回的实体关系拼入 LLM prompt。

聚类中心：

$$\hat{e}_j=\arg\min_{e\in c_j}\|v_e-\mu_j\|_2,\quad \mu_j=\frac{1}{|c_j|}\sum_{e\in c_j}v_e$$

八、关键创新点

  • 语义实体聚合：把 LLM 抽取的实体嵌入到 BigLog 空间做 DBSCAN 聚类去冗余。
  • 上下文感知实体召回：在线阶段用嵌入相似度检索最相关 KG 节点。
  • Label-Aware Prompting：把 root cause label 作为弱监督信号引导抽取。
  • LLM+KG 协同范式：降低幻觉并提升解释性。
  • 工业规模评估：在 2 个真实数据集上验证 F1 提升 6%-8%。

九、实验与结果

  • 数据集：阿里巴巴（2,671 故障）、中国移动（93 故障）。
  • Baseline：LogKG、LogRule、LogPrompt、LogRAG。
  • 主要指标：F1-score、Precision、Recall。
  • 关键数字：AetherLog F1 0.93 与 0.97，相对最佳基线提升 6% 与 8%。
  • 消融：去掉语义聚合、context-aware recall 各带来明显 F1 下降。
  • 效率：检索阶段主要开销是 LLM 调用，KG 查询近似 ANN。
