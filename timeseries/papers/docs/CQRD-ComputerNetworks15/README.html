二、研究背景

2.1 数据中心网络流量特征

现代数据中心网络（DCN）承载搜索、Web 零售、社交等实时交互业务。根据对真实运营数据中心流量的测量分析 [Alizadeh et al., SIGCOMM 2010]，DCN 流量呈现以下典型特征：

  • 短流占绝大多数：约 90% 的流大小小于 100KB（定义为短流/小流），它们对延迟高度敏感。这些短流通常在 1 个 RTT 内即可完成传输，但受限于 TCP 的慢启动机制，往往无法充分利用可用带宽。
  • 少量长流/巨流占据带宽：约 3% 的流为巨流（>1MB），如备份、复制等，是"带宽贪婪型"。这些长流在拥塞避免阶段会持续占据 buffer 和链路资源。
  • 流完成时间（FCT）严重退化：短流的 FCT 可比理想值高出 10 倍以上。这种退化直接影响用户体验和业务营收。

流量分布可以用对数正态分布（Log-normal）来刻画。设流大小为 $X$，则其概率密度函数为：

$$ f(x) = \frac{1}{x \sigma \sqrt{2\pi}} \exp\left(-\frac{(\ln x - \mu)^2}{2\sigma^2}\right), \quad x > 0 $$

在真实 DCN 工作负载中，$\mu$ 和 $\sigma$ 的取值使得大部分概率质量集中在较小的 $x$ 值处。论文 Fig. 3 展示了这种分布：超过 80% 的流小于 10KB，而巨流（>1MB）仅占约 3%。这种极度偏斜的分布是 DCN 流量区别于传统 Internet 流量的关键特征。

论文还指出，在数据中心挖掘（data mining）场景中，超过 80% 的流小于 10KB [pFabric, SIGCOMM 2013]。这意味着绝大多数流在 slow-start 阶段就已完成传输，根本没有机会通过拥塞控制找到最佳发送速率。

2.2 流干扰的根因

论文定义了以下关键概念：

  • Output Contending（输出竞争，OC）：两流在同一交换机共享同一输出端口。
  • Switch Path（交换路径）：同一交换机的输入-输出对。
  • Path Contending（路径竞争，PC）：两流在同一交换路径上时间重叠。
  • OC-PC：output contending but not path contending——论文重点优化的"大多数短流"场景。这类短流虽然与长流竞争同一输出端口，但由于来自不同输入端口，可以通过 CQ 的独立 buffer 完全隔离。

在传统 OQ（输出队列）交换机中，所有 OC 流共享输出 buffer，采用 tail-drop 策略。这导致三个问题：

  1. 长流包占据 buffer 大部分空间，短流包排队延迟增加。
  2. Buffer 溢出时 tail-drop 丢弃所有流的包，短流丢包率与长流相同。
  3. 短流的 RTT 和丢包率均显著增加，FCT 急剧恶化。

论文使用了一个 8 端口交换机的玩具示例（Fig. 1）：7 个流（5 短流 + 2 长流）全部到输出 O8。在 OQ 中，长流在数毫秒内占满 36KB buffer，短流 FCT 升到数百毫秒（理想值仅数十微秒）。在 CQ 中，短流 FCT 比 OQ 低 3 个数量级、比 HCF 低 1 个数量级。这个 toy example 清楚地展示了队列管理粒度对短流性能的巨大影响。

在 G/D/1/K 排队模型下，OQ 的丢包率随利用率呈指数增长。当长流使到达率接近 1 时，短流丢包率急剧上升，这是 OQ 方案的根本缺陷。

2.3 已有解决方案的局限

2.3.1 传输层速率控制

DCTCP [Alizadeh et al., SIGCOMM 2010]、HULL [Bai et al., SIGCOMM 2012]、D2TCP [Vamanan et al., SIGCOMM 2012] 等方案通过 ECN 调速尝试把队列维持在低位。然而：

  • 精确速率控制在突发流量下极具挑战。由于 DCN 流量的突发性，精确控制发送速率以在保持高利用率的同时维持低 buffer 是非常困难的。即使采用了 DCTCP，buffer 仍可能因为突发流量而积压。
  • 在 OQ 交换机中，短流包仍可能排在长流包后面。即使 DCTCP 尝试保持低 buffer，但 burst 仍然会导致短流包被阻塞在长流包之后。
  • 需要修改端主机协议栈，部署困难。网络运营商通常不控制端主机协议栈（如公有云场景），一些高性能应用实现自己的传输协议绕过内核，导致部署更加困难。
  • 某些方案还需要修改 NIC 硬件或交换机硬件（如 HULL），进一步增加部署难度。

2.3.2 流调度

pFabric [Alizadeh et al., SIGCOMM 2013]、PDQ [Hong et al., NSDI 2012] 等方案尝试实现最优流调度以最小化短流 FCT。但：

  • 全局最优调度是 NP-hard 问题，在大规模多级 DCN 中更是如此。
  • 调度延迟对短流过长。对于微秒级短流，调度决策本身的延迟就可能超过流完成时间。
  • 需要大量端/网元改动，几乎属于 clean-slate 方案，部署困难。
  • 即使采用了流调度，非最优调度和长调度延迟仍然可能导致短流在 OQ 交换机中被长流干扰。

2.3.3 HCF（Hashed Credits Fair）

HCF [Noormohammadpour et al., SIGCOMM/CoNEXT 2014] 是当时 DCN 领域最先进的交换机队列管理方案。它设置 HP/LP 两个队列，对 flow hash 到 bin 后按 credit 投放。但 HCF 存在以下不足：

  • Hash 粒度仅到 bin（如 20 个 bin），bin 内部仍可能冲突。不同输入端口的短流可能被 hash 到同一 bin，仍然竞争同一 buffer 空间。
  • 长流消耗 credit 后填满 LP 队列。如果短流不幸与长流 hash 到同一 bin，短流也会受到严重影响。
  • 需要周期性更换 hash 函数以改善公平性，增加了硬件实现成本。
  • 当短流数量增加时，它们各自消耗 credit 并用完后在 LP 队列被丢弃，导致大量误报。

2.3.4 Ethernet 相关技术

IEEE 802.1p 使用 3 bit 优先级区分流量类型，但需要大量端主机/交换机修改来按流大小打标签，部署困难。

2.3.5 Crosspoint Queue 的历史与复兴

经典 CQ（Crosspoint Queue）交换机为每个输入-输出对分配独立 buffer（XB），用 Round-Robin 调度包。CQ 最早在 Internet 路由器设计中由 Katevenis 等人提出，但因半导体工艺限制被认为不可行。直到 2010 年代，[Katevenis et al., 2010] 重新论证了纯 CQ 交换机的可行性。然而，CQ 在 XB 满时采用 tail-drop，大流继续占满小流的 buffer。CQRD 在 CQ 基础上将 tail-drop 改为 random-drop，解决了这一问题。论文声称是第一篇将 CQ 引入 DCN 交换机队列管理的论文。

3.3 设计目标

  1. 细粒度：队列管理粒度比 OQ 细，减少流干扰概率。
  2. 简单：基于成熟硬件原语（CQ、random-drop），易于理解和实现。
  3. 可部署：零端主机修改、零交换机间协调、增量部署。
  4. 高性能：短流 FCT 显著降低，长流 goodput 几乎不受影响。
  5. 可协同：与 DCTCP/ECN 等传输层方案兼容并进一步增强。

4.2 DCN 流量关键观察

论文在 Section 4.1 提出了一个关键的观察（Observation 1）：

长带宽贪婪流仅遍历少量交换路径（switch path），而大多数交换路径正在传输短延迟敏感流。

具体来说，一个 24x24 聚合交换机有 24x23=552 个交换路径（排除自环）。假设 2 个巨流穿越 2 条交换路径，同时有数百个短流穿越其他 550 条路径。在 OQ 中，这 2 个巨流通过 2x23=46 条路径上的数百个短流造成干扰。CQRD 通过为每条独立交换路径分配独立 buffer 来消除这种干扰。

Fig. 6 展示了仿真中的流干扰统计：在单交换机场景下，巨流干扰了超过 80% 的流；在多级 DCN 场景下，由于路径更长、交叉更多，干扰比例更高。其中 OC-PC 占被干扰流的大多数，PC 占少数。

4.3 关键数学推导

4.3.1 OC-PC 短流的 RTT 分析

考虑一个 $N \times N$ 交换机，每个 XB 容量为 $L$ packets。设有 $s$ 个短流和 $l$ 个长流到同一输出 $O$。对于 OC-PC 短流 $F_{s1}$（来自输入 $I_{s1}$，去往输出 $O$），其在 OQ 中的 RTT：

$$ RTT_o = \bar{L}_o \times t = \frac{N \times L}{2} \times t $$

其中 $\bar{L}_o = \frac{N \times L}{2}$ 是 OQ buffer 在稳态下的平均队列长度（基于 TCP 锯齿特性 [Morris, 1999]），$t$ 是调度一个包出 buffer 的基本时间单位。在 CQRD 中：

$$ RTT_c = \bar{N}' \times \bar{L}_c \times t $$

其中 $\bar{N}'$ 是平均同时有 packet 的 XB 数（对应输出 $O$），$\bar{L}_c$ 是短流 XB 的平均长度。由于短流不与长流共享 XB，$\bar{L}_c < \frac{L}{2}$，且 $\bar{N}' \leq N$，因此：

$$ RTT_c = \bar{N}' \times \bar{L}_c \times t \leq N \times \frac{L}{2} \times t = RTT_o $$

即 CQRD 中 OC-PC 短流的 RTT 严格小于 OQ。这个结论的直观解释是：CQRD 通过独立 buffer 消除了长流对短流的排队影响。

4.3.2 PC 短流的 RTT 分析

对于 path contending 短流 $F_{s2}$，其与长流共享同一 XB，在 CQRD 中：

$$ RTT_c = \bar{N}' \times (\bar{L}_c - \bar{D}) \times t \leq N \times \frac{L}{2} \times t = RTT_o $$

其中 $\bar{D}$ 是被 random-drop 丢弃的 packet 平均数。由于 $\bar{D} \geq 0$，PC 短流的 RTT 同样不超过 OQ。在 OQ 中，PC 和 OC-PC 短流的 RTT 相同（均排在长流之后）；CQRD 中 PC 短流因与长流共享 XB 而 RTT 高于 OC-PC 短流，但仍优于 OQ。

4.3.3 丢包率分析

对于 OC-PC 短流，由于短流独占 XB，到达率 $\lambda$ 低。在 G/D/1/K 排队模型下，丢包率随利用率指数增长：

$$ Loss_c < Loss_o $$

对于 PC 短流，设 $P_{Loss_o}$ 为 OQ 的总体丢包率，$P_{Loss_c}$ 为 XB 的总体丢包率，$L_{s2}$ 为短流 $F_{s2}$ 在 XB 中的 packet 数，则：

$$ Loss_c = P_{Loss_c} \times \frac{L_{s2}}{L} < Loss_o = P_{Loss_o} $$ $$ \text{当} \quad \frac{P_{Loss_c}}{P_{Loss_o}} < \frac{L}{L_{s2}} \quad \text{时即成立} $$

由于短流 $L_{s2}$ 占 buffer 极少（$L/L_{s2}$ 很大），此条件通常满足。因此 CQRD 中 PC 短流的丢包率也低于 OQ。

4.3.4 长流吞吐保持

最极端情况：2 个长流共用同一 XB（来自同一输入端口），需要 BDP 缓冲保证输出容量饱和。生产 DCN 典型 BDP：

$$ BDP = 1 \text{ Gbps} \times 200 \mu s = 25 \text{ KB} $$

48x48 CQRD 需约 58 MB on-chip memory（48x48 x 25KB）。当前 ASIC 工艺可支持 128x128（25KB XB）或 48x48（250KB XB，适配 100Gbps x 20us BDP）。

更常见的情况是长流来自多个输入端口（$l_n$ 个端口），则 CQRD 仅需总和 BDP（$l_n \times BDP$）即可保证 100% 输出利用率。根据 tiny buffer 模型 [Appenzeller et al., 2004]，每个 XB 只需数 KB 即可保证 80-90% 输出利用率，进一步降低了 CQRD 的 memory 需求。

4.3.5 DCTCP/ECN 集成：Random-Mark

当 XB 长度超过 ECN 阈值时，CQRD 用 random-mark 替代 random-drop：按长度比例随机挑一个整包打 ECN 标记。短流被误标记概率：

$$ P_{\text{mark\_short}} = \frac{L_{\text{short}}}{L_{\text{total}}} \ll P_{\text{mark\_OQ\_short}} = 1 $$

在 OQ 中，所有包（含短流包）在超过阈值时都被标记；CQRD 中短流包被标记的概率正比于其在 buffer 中的占比，远低于 OQ。但 random-mark 不保证所有短流都获益——某些被连续标记的短流 FCT 可能仍然较高。论文明确承认了这一局限性。

4.4 Work-Conserving 与调度机制

CQRD 保证 work-conserving：只要 XB 中有包，输出就不会空闲。每个时间槽包含两阶段：

  1. Arrival phase：新 packet 到达输入 $I_i$ 的 $XB_{ij}$。若有空间直接入队；若满，按长度比例随机找 victim 丢弃，循环直到有空间。
  2. Departure phase：每个输出独立 RR 选择非空 XB，调度 head-of-line packet。

Random-drop 可以流水线化实现：一个 $C$ cells 的包到达消耗 $C$ 个时间槽，这 $C$ 个时间槽足够完成 $C$ 轮 random-drop（每次丢弃至少 1 cell，因为最小包大小大于 1 cell）。因此不引入额外延迟。

        self.queue = []                       # 包列表，每项为 cells 数
        self.length = 0                       # 当前占用的 cells 数
        self.ecn_threshold = ecn_threshold    # ECN 标记阈值

    def arrival(self, pkt_cells, is_ecn_mode=False):
        """包到达处理：入队或随机丢弃/标记"""
        # 情况 1：XB 有空间，直接入队
        if self.length + pkt_cells <= self.cap:
            self.queue.append(pkt_cells)
            self.length += pkt_cells
            return True                        # 入队成功

        # 情况 2：XB 满，执行 Random-Drop / Random-Mark
        while self.length + pkt_cells > self.cap:
            if is_ecn_mode and self.length > self.ecn_threshold:
                # ECN 模式：Random-Mark（不丢弃，仅标记）
                victim_idx = weighted_random_choice(self.queue)
                self._mark_packet(victim_idx)  # 打 ECN 标记
            else:
                # 非 ECN 模式：Random-Drop（丢弃整包）
                victim_idx = weighted_random_choice(self.queue)
                victim_cells = self.queue.pop(victim_idx)
                self.length -= victim_cells
                if victim_idx == len(self.queue):
                    return False               # 新包被丢弃

        # 空间足够后入队新包
        self.queue.append(pkt_cells)
        self.length += pkt_cells
        return True

    def departure_round_robin(self, output_port, xb_column):
        """输出调度：Round-Robin 选择非空 XB 的 head-of-line"""
        for xb in xb_column:
            if xb.queue:
                pkt = xb.queue.pop(0)
                xb.length -= pkt
                return pkt
        return None


def weighted_random_choice(queue):
    """按 packet 长度比例随机选择一个 packet 索引"""
    total = sum(queue)
    if total == 0:
        return random.randint(0, len(queue) - 1)
    r = random.random() * total
    cumsum = 0
    for i, pkt_size in enumerate(queue):
        cumsum += pkt_size
        if cumsum >= r:
            return i
    return len(queue) - 1


class CQRDSwitch:
    """N x N CQRD 交换机"""
    def __init__(self, n_ports, xb_size_cells):
        self.N = n_ports
        # 创建 N x N 的 XB 矩阵
        self.xbs = [[CQRDCrosspointBuffer(xb_size_cells)
                     for _ in range(n_ports)]
                    for _ in range(n_ports)]

    def process_arrival(self, in_port, out_port, pkt_cells, ecn=False):
        """处理包到达"""
        return self.xbs[in_port][out_port].arrival(pkt_cells, ecn)

    def process_departure(self, out_port):
        """处理包离开（RR 调度）"""
        col = [self.xbs[i][out_port] for i in range(self.N)]
        for xb in col:
            pkt = xb.departure_round_robin(out_port, col)
            if pkt is not None:
                return pkt
        return None

    """
    在 Broadcom BCM56842 上配置 8x8 逻辑 CQRD 交换机。
    BCM56842: 32x32 10Gbps 交换机, 9MB shared buffer
    """
    # Step 1: 启用 8 个 priority queue 每 output，等大小
    buffer_per_queue = 9_000_000 / 82  # 约 140 KB
    for output_port in range(n_ports):
        switch.enable_priority_queues(
            port=output_port,
            n_queues=n_ports,
            size_per_queue=int(buffer_per_queue)
        )
        # Step 2: 配置 WRR 调度，等权重（等效于 RR）
        switch.set_scheduling(
            port=output_port,
            mode='WRR',
            weights=[1] * n_ports
        )

    # Step 3: 配置 ICAP 分类规则（ingress）
    # 把不同 input port 的包映射到不同 priority queue
    for input_port in range(n_ports):
        switch.set_icap_rule(
            input_port=input_port,
            match_field='input_port',
            match_value=input_port,
            action='redirect_to_priority_queue',
            priority_queue_index=input_port
        )

    # Step 4: 启用 RED 模拟 Random-Drop
    for output_port in range(n_ports):
        for pq in range(n_ports):
            queue_size = switch.get_priority_queue_capacity(output_port, pq)
            switch.set_red(
                port=output_port,
                queue=pq,
                min_threshold=int(0.8 * queue_size),
                max_threshold=queue_size,
                drop_probability=1.0
            )

    # Step 5: 验证配置
    switch.verify_config()
    return True

7.2 工程优化

流水线化 random-drop：包到达消耗 $C$ 个时间槽，这 $C$ 个时间槽足够完成 $C$ 轮 random-drop，不引入额外延迟。这是 CQRD 能够在 line rate 下工作的关键工程保证。

Cell-based 内部处理：交换机内部以 cell（最小包大小，通常 64 bytes）为单位处理。包由多个 cell 组成，SoP（Start-of-Packet）cell 标记包长度。这种设计使得 XB 可以以固定大小的 cell 为单位管理空间，简化了硬件实现。

WRR 替代 RR：在 BCM56842 上用 Weighted Round-Robin 替代纯 Round-Robin，等权重时等效。BCM56842 不支持纯 RR 模式，但等权重 WRR 在功能上完全等效。

RED 近似 random-drop：RED 在队列超过阈值后随机丢包。虽然与 CQRD 的精确 random-drop 略有差异（RED 在阈值之后连续随机丢弃后续包，而非针对 XB 满时的整包丢弃），但统计上按入队比例丢弃，能够近似 CQRD。

Pipeline 流水线：arrival phase 和 departure phase 在同一时间槽内可异步执行，最大化吞吐。这种设计确保了 CQRD 不会成为转发路径上的瓶颈。

ICAP 规则简化：输入端口到 priority queue 的一一映射避免了复杂的 TCAM 查表，降低了配置复杂度和处理延迟。

十二、局限性与未来工作

12.1 局限性

  1. XB 容量限制：小 XB 在 incast 场景（大量短流同时到同一输出）下可能不足。论文指出 incast 在 DCN 中发生频率低。
  2. 长流 goodput 轻微下降：由于损失了统计复用，长流 goodput 最多下降约 5%。
  3. Random-drop 不保证公平性：某些短流可能被连续丢弃（bad tail），统计上整体改善但个案仍受影响。
  4. 假设短流占多数：CQRD 的设计假设 DCN 流量中短流占大多数；对其他流量模式不适用。
  5. 大规模 ASIC 未实际流片：仅在 BCM56842 上 8x8 配置验证，大规模 ASIC 实现仍需工程验证。
  6. 硬件成本：48x48 CQRD 需 58MB on-chip memory，比 OQ 的 5MB 多一个数量级。

12.2 未来工作

  • 将 CQRD 与更先进的传输层方案（如 BBR、TIMELY）结合。
  • 探索动态 XB 容量分配——根据流量模式自适应调整 XB 大小。
  • 研究 CQRD 在 incast 场景下的特殊优化。
  • 在更大规模测试床上验证（如 48x48 交换机，100Gbps 链路）。
  • 研究 CQRD 对 RDMA/RoCEv2 流量的影响。
  • 将 random-drop/mark 扩展到 programmable switch 平台（如 P4 编程交换机）。
  • 研究 CQRD 与 INT（In-band Network Telemetry）遥测技术的结合。

十三、总结与启示

CQRD 的核心思想简洁而深刻：通过细粒度队列管理（per-input-output XB）+ 随机丢弃（random-drop）解决 DCN 流干扰问题。其关键启示包括：

  1. 细粒度隔离是根本：OQ 的粗粒度共享是流干扰的根因；CQ 的输入-输出对粒度直接解决了 OC-PC 场景。
  2. 随机丢弃优于尾部丢弃：在共享 XB 场景中，random-drop 按长度比例丢弃给小流"入场"机会。
  3. 零端主机改动：所有逻辑在交换机内部完成，不需要修改 TCP 栈或应用程序。
  4. 可增量部署：50% ToR 部署即可获得显著收益，避免了全面硬件替换。
  5. 跨领域借鉴：将 Internet 路由器的成熟技术（CQ）引入 DCN，用现代 ASIC 技术实现。
  6. 与传输层互补：CQRD + DCTCP 组合效果最优，证明交换机级方案与传输层方案不矛盾而是互补。
  7. 工程可实现：在现有 commodity 交换机上通过配置即可实现。

15.1 算法时间复杂度详细分析

对核心算法的每个步骤进行详细的复杂度分析有助于工程实现优化。我们分别讨论训练阶段和推理阶段的复杂度。

15.1.2 推理阶段复杂度

推理阶段的时间复杂度直接影响系统的响应性能。对于实时系统，推理时间需要控制在毫秒级甚至微秒级。

  • 单样本推理：$O(d^2)$ 或 $O(d \log d)$（取决于模型类型）
  • 批处理推理：$O(B \cdot d^2)$，其中 $B$ 是批大小
  • 流式推理：$O(d)$ 每样本，利用增量计算

15.2 内存优化策略

在生产环境部署时，内存优化是关键问题。常用的优化策略包括：

  1. 模型压缩：通过剪枝、量化、知识蒸馏等技术减少模型大小。
  2. 缓存优化：使用 LRU 缓存最近访问的数据，减少重复计算。
  3. 分批处理：将大批量数据分批处理，避免内存爆炸。
  4. 稀疏存储：对于稀疏特征，使用稀疏矩阵存储。
  5. 内存映射：对于大文件，使用 mmap 技术，避免一次性加载。

十五、附加技术深度讨论

15.1 排队论模型详细推导

15.1.1 G/D/1/K 模型基础

G/D/1/K 排队模型描述一个有限缓冲队列系统：

  • G（General）：到达过程为一般分布（包括泊松、突发等）。
  • D（Deterministic）：服务时间是确定的（每个包消耗固定时间 $t$）。
  • 1：单个服务器（一个输出端口）。
  • K：队列容量上限（这里 $K = N \times L$）。

该模型的稳态丢包率 $P_{\text{loss}}$ 与利用率 $\rho = \lambda / \mu$ 的关系：

$$ P_{\text{loss}} \approx \frac{(1-\rho) \rho^K}{1 - \rho^{K+1}} $$

当 $\rho \to 1$（高负载）时，$P_{\text{loss}}$ 接近 $\frac{1}{K+1}$。当 $\rho$ 略小于 1 时，$P_{\text{loss}}$ 仍然较大。

15.1.2 OQ vs CQRD 的丢包率对比

在 OQ 中，所有流共享同一个队列，长流使 $\rho \to 1$，导致 $P_{\text{loss}}$ 接近 $\frac{1}{K+1}$，所有流都受影响。

在 CQRD 中，OC-PC 短流独占 XB，$\rho \ll 1$，$P_{\text{loss}} \to 0$。PC 短流虽然与长流共享 XB，但 random-drop 按比例丢弃，短流被丢弃的概率与其在 XB 中的占比成正比。

15.1.3 BDP 与 XB 容量的关系

带宽-时延积（BDP）决定了 TCP 拥塞控制能维持的最大窗口：

$$ BDP = B \times RTT $$

其中 $B$ 是链路带宽，$RTT$ 是往返时延。对于 1 Gbps 链路 + 200us RTT，$BDP = 25$ KB。CQRD 的 XB 容量设计原则：

  • 若仅有 1 个长流通过 XB：需要至少 BDP 大小。
  • 若有多个长流通过 XB：可以共享 BDP（统计复用），平均每 XB 只需 BDP / N 大小。
  • tiny buffer 理论 [Appenzeller et al., 2004]：只需 BDP / sqrt(N) 即可达到 80-90% 利用率。

15.2 与 SDN/NFV 的协同

CQRD 作为数据平面方案，与 SDN 控制平面可以协同工作：

  • SDN 控制器：监控全局流量模式，决定哪些交换机部署 CQRD。
  • OpenFlow 扩展：可以通过 OpenFlow Meter 或 Group 表配置 CQRD 队列。
  • NFV：在 SmartNIC 上实现软件 CQRD，与硬件 CQRD 互补。

15.3 安全性考量

CQRD 引入了 random-drop，可能影响安全性：

  • DDoS 攻击：攻击者可能利用 random-drop 的特性，发送大量小流以挤占 XB。但 CQRD 的 buffer 隔离反而限制了攻击影响范围。
  • 侧信道攻击：通过观察 random-drop 的统计特性，可能推断队列状态。但实际利用难度高。
  • QoS 公平性：random-drop 给小流"露脸"机会，对正常用户提升公平性。

15.4 与 RDMA/RoCEv2 的兼容性

RDMA over Converged Ethernet (RoCEv2) 对 lossless fabric 有强需求。CQRD 的 random-drop 与之矛盾。解决方案：

  • 在 RoCEv2 流量上禁用 random-drop，改用 PFC（Priority Flow Control）。
  • 使用 random-mark 模式（基于 ECN），不实际丢包，让 DCQCN 调整速率。
  • 对不同流量类型采用不同的 XB 策略。

15.5 Programmable Switch 实现

使用 P4 编程交换机（如 Tofino）可以更灵活地实现 CQRD：

// P4-16 伪代码
control CQRDIngress(inout headers hdr, inout metadata meta) {
    apply {
        meta.xb_idx = hdr.ingress_port * NUM_PORTS + meta.egress_port;

15.6 部署经验总结

  1. 选择合适的 XB 大小：根据链路带宽和典型 RTT 计算 BDP，XB 大小应至少为 BDP 的几分之一。
  2. 合理设置 ECN 阈值：建议设为 XB 容量的 50-70%，与 DCTCP 推荐配置一致。
  3. 监控关键指标：包括短流 FCT 分布、长流 goodput、buffer 占用率、丢包率等。
  4. 渐进式部署：从 ToR 层开始，逐步扩展到 Agg 层，最后 Core 层。
  5. 与传输层协同：建议同时部署 DCTCP，发挥最大效果。

  • 70% 负载：CQRD 短流 FCT 比 OQ 低 15%+；99 分位低 5%+。

随着负载升高，所有方案性能趋同，但 CQRD 始终最优。这是因为高负载下更多 XB 同时有包，RR 调度的公平性更难保证。

22.2 被巨流干扰的短流 vs 未被干扰的短流

论文将短流分为"被巨流干扰"和"未被干扰"两类，分别分析：

流类型OQ 平均 FCTCQRD 平均 FCT下降比例
被巨流干扰（OC-PC）基准大幅下降>30%
被巨流干扰（PC）基准中等下降>20%
未被巨流干扰基准几乎相同~0%

这验证了 CQRD 的核心设计：通过 buffer 隔离和 random-drop 专门改善被巨流干扰的短流。

23.3 测试床实验结果

  • ICAP 分类规则正确映射输入端口到 priority queue。
