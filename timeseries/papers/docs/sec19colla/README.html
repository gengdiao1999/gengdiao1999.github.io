2. 研究背景

随着智能手机、笔记本电脑等终端设备的普及，用户行为预测（如位置变化、应用启动、广告点击等）已成为系统与机器学习社区的重要研究课题。传统方法采用"先训练后部署"（learn-then-deploy）的范式：在云端收集大量混合用户数据，训练一个全局模型，再部署到各终端设备上进行推理。这种经典范式存在三大缺陷：

  • 数据异构性：不同用户的行为模式差异巨大（如儿童、大学生、公司职员的移动模式截然不同），单一全局模型无法适配所有用户。
  • 资源限制：为了在混合数据集上获得良好性能，模型容量必须足够大，从而增加了终端设备的资源消耗。
  • 无法适应数据漂移：训练完成后模型不再更新，无法利用运行时的新数据，导致模型随时间推移性能下降。

另一方面，直接在终端设备上训练模型也面临困境：本地数据量有限、计算能力和能源预算极为受限。

针对上述矛盾，本文提出Colla——一种云-端协同学习框架。核心洞察在于：尽管每个设备的数据有限且分布不均，但用户行为之间存在内在的共性模式（如所有设备上的Yelp应用倾向于在午餐时间被启动），这使得云端与多个终端设备可以相互补充、共同学习。

3. 论文目的与解决的问题

本文旨在回答一个核心问题：每个终端设备能否独立学习自身的预测模型并持续改进？

具体而言，Colla要解决以下挑战：

  1. 有效且高效的本地训练：终端设备资源有限，如何在其上完成有意义的模型训练？
  2. 知识迁移：如何实现云→设备和设备→设备之间的知识传递？
  3. 冷启动问题：新设备加入时没有本地数据，如何快速获得可用的预测模型？
  4. 过拟合防范：本地数据量少，如何避免过拟合？

本文以智能手机位置预测为具体案例，验证Colla框架的可行性与优势。

4. 核心原理（含数学公式）

4.1 问题形式化

定义序列 \(S^d\) 为设备 \(d\) 上的采样序列 \(q_{d1}, q_{d2}, ..., q_{dn}\)，其中 \(q_{di} = (t_i, v_i)\)，\(t_i\) 为时间戳，\(v_i \in V\) 为值（如位置ID）。序列预测问题即预测下一个数据点 \(q_{dn+1}\)。

4.2 多特征RNN（M-RNN）

M-RNN模型从轨迹序列中提取6种特征：位置ID、小时、分钟、星期、停留时长、间隔时间。每种特征通过嵌入层转换为稠密向量，拼接后输入LSTM：

\[ h_n, c_n = ext{LSTM}(X_n, (h_{n-1}, c_{n-1})) \]

\[ ext{location}_{n+1} = f(h_n) \]

其中 \(f\) 是全连接层，输出维度为 \(|V|\)（可能的位置数量）。

4.3 知识蒸馏损失函数

知识蒸馏的核心公式：

\[ L = \lambda L(y, p) + (1 - \lambda)L(q, p) \]

其中 \(L(st,st)\) 为损失函数（交叉熵或KL散度），\(p\) 为学生模型输出概率，\(q\) 为教师模型输出概率。

4.4 客户端蒸馏

\[ \mathcal{L}_{d_i} = \lambda_d L(y_i^d, p_i^d) + (1 - \lambda_d)L(q^c, p_i^d) \]

4.5 云端蒸馏

\[ \mathcal{L}_c = \lambda_c L(y^c, p^c) + (1 - \lambda_c)L\left( rac{1}{N}\sum_{i\in N} q_i^d, p^c ight) \]

4.6 模型压缩

云端使用大模型（LSTM单元128个，嵌入维度32），设备端使用小模型（LSTM单元16个，嵌入维度4）。参数数量分别为128,796和11,868。

4.7 模型分组

使用Affinity Propagation聚类算法，将N个客户端模型分为K组，每组生成一个分组云模型。最佳K值通过Silhouette系数确定。

6. 关键算法详解

6.1 双蒸馏算法

算法1: Colla双蒸馏更新

输入: 云模型 M, 客户端模型集 {m_i}, 本地数据集 D_i, 云数据集 D_c
输出: 更新后的 M 和 {m_i}

1: for 每个更新周期 do
2:    for 每个设备 i in 并行 do
3:        // 客户端蒸馏
4:        从云端拉取最新 M
5:        在 D_i 上计算硬标签损失 L(y_i^d, p_i^d)
6:        用 M 在 D_i 上生成软标签 q^c
7:        计算蒸馏损失 L(q^c, p_i^d)
8:        L_d = λ_d · L(y_i^d, p_i^d) + (1-λ_d) · L(q^c, p_i^d)
9:        反向传播更新 m_i
10:       上传 m_i 参数至云端
11:   end for
12:   // 云端蒸馏
13:   收集所有设备参数 {m_i}
14:   在 D_c 上计算所有 m_i 的软标签 {q_i^d}
15:   L_c = λ_c · L(y^c, p^c) + (1-λ_c) · L(avg(q_i^d), p^c)
16:   反向传播更新 M
17:   // 模型分组
18:   用每个 m_i 在 D_c 上推理，获得特征向量
19:   计算特征向量间的交叉熵距离
20:   Affinity Propagation聚类，用Silhouette系数选K
21:   每组内进行云端蒸馏，生成分组云模型 M^k
22: end for

操作 时间复杂度 空间复杂度
客户端蒸馏（单设备） O(E·B·H²) 其中E=epoch数，B=batch大小，H=隐藏层维度 O(H² + |V|·d) 模型参数

7. 实现细节

云端实现：Microsoft Azure云平台，Intel Xeon CPU@2.60GHz，128GB内存，3TB存储，8块NVIDIA Tesla P100 GPU。模型结构存储在.json文件，权重存储在.npy文件。初始训练数据以HDFS格式存储在Azure HDInsight。使用Keras (v1.0.0)和Theano (v1.0.0)进行模型训练。通过RESTful API实现云-端通信。

客户端实现：Android智能手机，使用Pydroid 3（支持离线Python 3.6解释器和内置C/C++编译器）。通过pip安装Keras、numpy、scipy和Theano。由于缺少移动GPU训练编程模型，模型训练在CPU上完成。

训练配置：损失函数使用交叉熵，λ_c = λ_d = 0.5（经超参数搜索确定），batch size = 32，Adam优化器，学习率0.1。训练/验证/测试集比例16:4:5。早停机制：验证集损失连续20个epoch不下降则停止训练。

8. 实验设计与结果

8.1 数据集

WiFi AP扫描记录收集自某大型大学校园，2016年3月至6月共4个月。总计120,624,600条扫描记录，涉及201,041台设备和2,890个Cisco企业AP（部署在116栋建筑中）。高峰时段约20,000台设备同时连接校园WLAN。每天唯一设备数超过60,000台。每台设备平均每天扫描4.89个AP。

将2,890个AP聚类为368个位置簇（|V|=368）。使用10分钟时间窗口提取轨迹序列。过滤掉每月活跃天数少于10天的设备，最终保留12,540台设备。

9. 消融实验

9.1 模型分组的影响

在无分组（COLLA-1）情况下，虽然预测结果随更新次数提升，但增益逐渐减小。加入分组（COLLA-2）后，持续改进效果显著。针对Top-100设备，聚类为2组时获得最佳整体准确率，与Silhouette系数的排名一致。

9.2 自定义FC层的影响

比较Customized FC（每台设备动态扩展FC层大小）与Unified FC（所有设备使用相同FC层架构）：自定义FC严格优于统一FC，因为新位置更可能在早期周期中出现。

9.3 三种更新策略对比

策略 上传参数 推理FLOPS 微调时间

Same-FC 仅FC层 高 中等
Customization 仅FC层（动态扩展） 减半 最短（2.42分钟平均）

Colla已在Microsoft Azure云和Android智能手机上完全实现。云端使用Keras+Theano进行模型训练和推理。客户端使用Pydroid 3作为Python运行环境。通信开销低至每次更新520KB（模型架构4-16）。一次夜间训练（约1小时）可处理典型智能手机一年的数据量，能耗约2000mWh。

12. 局限性与未来工作

  • 初始数据质量：当前设计从固定量的初始数据开始学习，初始数据的质量（如多样性）对通用模型的影响有待研究。
  • 额外数据上传：探索启动后额外数据上传的收益，如新设备可一次性推送一定量数据。
  • 模型架构自适应：固定客户端模型架构可能不适合增量学习场景，需要动态调整架构的能力。
  • 选择性聚合：选择性地使用设备模型进行云端更新，可提高泛化能力并防止被对抗输入破坏。
  • 隐私保护：模型参数可能泄露训练集信息，需要设计隐私保护的Colla版本。
  • 扩展到CNN：将Colla扩展到卷积神经网络是未来的研究方向。

13. 总结与启示

Colla提出了一个创新的云-端协同学习框架，通过双蒸馏机制实现了知识在云和设备之间的双向传递。核心启示包括：

  1. 终端设备可以且应该主动参与学习：在资源受限的设备上，通过轻量级模型和蒸馏技术，本地学习是可行的。
  2. 群体智能优于个体智慧：利用云端聚合多设备知识，显著提升单个设备的预测性能。
  3. 个性化与通用性的平衡：通过模型分组和自定义FC层，实现了在全局知识共享前提下的设备级个性化。
  4. 效率与效果的兼顾：通过模型压缩和参数选择上传，在保证预测精度的同时将通信和计算开销控制在可接受范围内。

2a. 研究背景深入分析

2a.1 用户行为预测的发展历程

用户行为预测的研究可以追溯到20世纪90年代，早期方法主要基于马尔可夫链模型。Asahara等人[7]提出了混合马尔可夫链模型（Mixed Markov-chain Model），通过构建位置之间的转移概率矩阵来预测未来移动。Gambs等人[8]使用移动马尔可夫链进行下一位置预测，取得了较好的短期预测效果。

随着深度学习的发展，循环神经网络（RNN）因其序列依赖建模能力被广泛应用于行为预测。Liu等人[4]提出了ST-RNN（Spatial Temporal Recurrent Neural Networks），在时间和空间维度上建模移动模式。DeepSense[65]集成了卷积和循环神经网络，为移动感知数据提供了统一的深度学习框架。然而，这些方法大多采用集中式训练范式，忽略了用户间的数据异构性。

2a.2 分布式机器学习综述

分布式机器学习已成为大数据和大模型时代的热门研究方向。模型平均（Model Average）[32-35]是一种简单高效的技术，通过迭代平均本地训练的模型实现参数同步。Allreduce通信[36]用于数据并行设置中的梯度通信。参数服务器[37-39]使用专用服务器同步模型参数并实现优化算法相关的计算逻辑。

联邦学习[19,40,41]尝试利用本地收集的数据训练全局模型，但未考虑设备多样性，不允许模型定制化。与这些方法不同，Colla中的端点不仅是计算设备，还是训练模型的消费者，产生非独立同分布的训练数据。

2a.3 知识蒸馏技术演进

知识蒸馏最早由Bucila等人[15]提出，用于将集成模型或大型教师模型的知识迁移到小型学生模型。Hinton等人[16]系统化地提出了蒸馏框架，使用软标签（soft labels）作为教师模型的知识传递媒介。此后，知识蒸馏被广泛应用于图像分类[16]、神经机器翻译[22,23]和语音识别[16,24]等任务。

双向蒸馏是Colla的核心创新之一。Deep Mutual Learning[27]在相同数据集上同时训练两个模型并使其相互匹配概率估计。Born-Again Neural Networks[28]使一个网络能够自我教学并通过连续蒸馏生成新模型。Colla的双蒸馏范式与这些方法不同：N个不同的客户端模型从各自的本地数据中蒸馏知识，并通过云在彼此之间共享。

2a.4 边缘计算与移动深度学习

研究人员一直在探索各种方法以在计算能力和能源预算有限的移动和边缘设备上实现深度学习。这些工作包括：构建不牺牲太多精度的更小模型[42-44]；利用或构建定制化硬件实现快速学习[45-48]；模型压缩以减少资源消耗[49-51]；系统优化以实现更好的资源-精度权衡[52-54]。

例如，DeepEar[42]提出了在不受约束声学环境中进行智能手机音频感知的特殊模型。DianNao[45]设计了专用ASIC加速器。NestDNN[55]设计了动态框架以在应用资源变化时选择最合适的模型。DeepX[49]利用运行时层压缩和深度架构分解减少资源使用。这些工作都聚焦于单个设备，与Colla的协同学习方法互补。

2a.5 位置预测的可行性与极限

Song等人[9]的研究表明，人类移动性的潜在平均可预测性高达93%。这一发现为位置预测研究提供了坚实的理论基础。然而，不同用户之间的可预测性差异巨大——儿童、上班族和旅行者的移动模式截然不同。这正是Colla提出个性化模型（客户端模型）的原因。

从信息论角度看，位置预测可视为对用户移动轨迹中信息熵的估计问题。在最佳情况下（即完全规则的生活模式），预测准确率接近100%；在最差情况下（完全随机移动），预测准确率趋近于随机猜测。Colla通过利用跨用户的共同模式（云模型）和用户特定的模式（客户端模型），有效地降低了预测问题的信息熵。

4a. 核心原理深入推导

4a.1 M-RNN模型的嵌入层设计

M-RNN模型从轨迹序列中提取六种特征，每种特征通过独立的嵌入层转换为稠密向量：

  1. 位置ID嵌入：使用10分钟时间窗口内的位置ID序列（类似于NLP中的句子），预训练嵌入模型将位置ID转换为稠密向量。嵌入维度在大模型中为32，在小模型中为4。
  2. 小时嵌入：24个离散值（0-23），捕捉一天中不同时段的行为模式差异。
  3. 分钟嵌入：60个离散值（0-59），提供更精细的时间粒度。
  4. 星期嵌入：7个离散值（周一至周日），区分工作日和周末的行为模式。
  5. 停留时长嵌入：144个离散值（每10分钟一个），表示用户在某位置停留的时间长度。
  6. 间隔嵌入：144个离散值，表示两个相邻不同位置之间的时间间隔。

在推理时，输入给定时间tn+1，通过Gap特征预测任意给定时间的位置值vn+1。六个嵌入向量拼接后形成LSTM的输入Xn。

4a.2 双蒸馏的理论分析

双蒸馏机制的核心思想可以从信息论角度理解。客户端蒸馏中，云模型（教师）提供了关于数据全局分布的先验知识，客户端模型（学生）在此基础上结合本地数据进行学习。这等价于：

\[ p( ext{local knowledge}) \propto p( ext{global knowledge}) imes p( ext{local data} | ext{global knowledge}) \]

云端蒸馏则可以理解为贝叶斯模型平均（Bayesian Model Averaging）的一种近似：通过聚合多个客户端模型的输出概率，云端模型获得了对整个数据分布的更准确估计。

在客户端蒸馏中，参数λ_d控制全局知识与本地数据的权衡。当λ_d=1时，模型完全依赖本地数据（退化为DT）；当λ_d=0时，模型完全追随云模型（退化为CT-ws）。λ_d=0.5的设定在实验中表现最佳，说明两者等权重结合是最优策略。

4a.3 模型分组的数学基础

模型分组使用Affinity Propagation聚类算法[20]，其核心是消息传递机制。给定N个客户端模型，每个模型在云数据集上的输出特征向量作为该模型的表示。两个模型之间的距离使用输出概率分布的交叉熵度量：

\[ d(m_i, m_j) = H(p_i, p_j) = -\sum_k p_i^{(k)} \log p_j^{(k)} \]

其中p_i和p_j分别是模型i和j在云数据上的输出概率分布。交叉熵距离比欧几里得距离更适合度量概率分布之间的差异。

聚类数K的选择使用Silhouette系数[21]：

\[ s(i) = rac{b(i) - a(i)}{\max\{a(i), b(i)\}} \]

其中a(i)是样本i到同簇其他样本的平均距离，b(i)是样本i到最近其他簇样本的平均距离。Silhouette系数范围为[-1,1]，值越大表示聚类效果越好。实验中从{2,3,5,10}中选择Silhouette系数最大的K值。

4a.4 通信效率分析

Colla的通信开销主要来自两个方面：

  1. 下行通信（云→设备）：云模型参数传输。小模型仅11,868个参数，约88KB。
  2. 上行通信（设备→云）：客户端模型参数上传。使用Customization策略时，仅上传FC层参数，约520KB（130,000×4字节）。

相比之下，CT方法需要上传所有本地数据（可能数GB），且推理时需要下载大模型（503KB+运行开销）。Colla的通信效率远优于传统方法。

7a. 实现细节深入

7a.1 嵌入层训练策略

位置ID嵌入采用预训练策略：使用10分钟时间窗口内的位置ID序列（类似NLP中的句子），通过word2vec风格的训练获得位置ID的稠密向量表示。这种方法使空间上相近的位置在嵌入空间中也更接近，为LSTM提供了更丰富的位置语义信息。

其他五种时间相关特征的嵌入则在端到端训练过程中联合学习。每个时间特征有独立的嵌入查找表，嵌入向量在训练中自适应更新。

7a.2 自适应FC层设计

Colla的一个重要设计是自适应FC层。由于不同设备访问的位置集合不同，每个设备的输出层维度|V|也应该不同。具体实现方式为：

  • 初始时，设备从云端获取压缩后的基础模型，FC层大小为其当前访问过的位置数。
  • 当设备发现新的位置ID时，FC层动态扩展：添加新的输出节点，初始化权重，然后通过微调适应新的输出空间。
  • 这种设计确保了每个设备只维护自己需要的输出空间，减少了参数量和计算开销。

7a.3 早停机制与正则化

客户端蒸馏中使用早停机制（Early Stopping）防止在有限本地数据上过拟合。具体策略为：连续20个epoch验证集损失不下降则终止训练。这一机制在数据量少的设备上尤为重要，因为过拟合的风险更高。

此外，云模型的软标签（soft labels）本身就起到了正则化的作用。与硬标签（one-hot编码）不同，软标签包含了类别间的相似性信息，使模型在微调时能够利用这些先验知识，避免过拟合到少量本地样本的噪声上。

7a.4 异步更新机制

在实际部署中，云端和设备的模型更新是异步进行的。设备根据自己的数据收集速率和资源约束（如电池电量）决定何时拉取云模型和上传本地参数。这种异步设计有以下优势：

  • 适应不同设备的数据收集速率差异——活跃设备更新更频繁。
  • 支持设备在充电时进行训练，避免影响正常使用。
  • 减少云端服务器的并发压力。

但异步更新也可能导致客户端模型与云模型的版本不一致，需要通过版本控制机制确保兼容性。

8a. 实验结果深入分析

8a.1 不同初始数据集大小的影响

实验比较了初始数据集包含10³和10⁴设备时各方法的性能。关键发现：

  • 当初始数据集较小时，基于微调的方法（DT、DT-ws、Colla）优于CT，因为微调允许模型从有限的初始知识出发逐步适应本地数据。
  • 当初始数据集足够大时，CT可以赶上甚至超过DT和DT-ws，但Colla始终保持最佳性能。
  • Colla从更大的初始数据集中获益——预训练云模型质量更高，带动所有设备的预测准确率提升。

8a.2 设备规模的影响

实验分别在Top-100、Top-1000和Top-10000设备上评估了CT和Colla的性能：

  • CT的性能随设备数增加而提升，因为更多数据帮助全局模型更好地泛化。
  • Colla的性能同样随设备数增加而提升，但提升幅度更大，因为协同学习能更有效地利用群体知识。
  • 在所有规模下，Colla的中位准确率均高于CT。

8a.3 模型更新周期的分析

默认更新周期T=20天。研究发现：

  • DT-ws在连续更新中几乎无法提升设备预测能力，因为纯依赖本地数据容易过拟合。
  • Colla在每次更新后都有显著提升，主要归功于模型分组带来的知识共享。
  • 更新增益随时间递减——第一次更新的增益最大，后续更新的边际收益逐渐减小。

8a.4 选择性设备模型过滤

过滤掉推理准确率最低的30%设备上传的模型后，端到端中位准确率提升了3.46%。这一发现表明：

  • 低质量设备模型可能包含噪声或对抗性信息，对云模型有负面影响。
  • 选择性聚合不仅提高泛化能力，还能防止云模型被对抗输入破坏。
  • 这为未来的联邦学习系统设计提供了重要启示——并非所有参与方的贡献都是有益的。

8a.5 不同模型架构的影响

比较四种M-RNN架构（4-16, 8-32, 16-64, 32-128）在智能手机上的表现：

  • 模型8-32需要最长的微调时间，虽然其大小排第三，但需要更多训练轮次才能收敛。
  • Customization策略在所有架构中都是最高效的——微调时间最短、推理FLOPS最少、上传参数最少。
  • 推理时间在所有设备上仅约133.5毫秒，使位置预测可以作为第三方应用的组件运行。

# Colla客户端蒸馏伪代码
def client_distillation(local_data, cloud_model, client_model, lambda_d=0.5):
    for epoch in range(max_epochs):
        for batch_x, batch_y in local_data.batches():
            # 硬标签损失
            hard_loss = cross_entropy(client_model(batch_x), batch_y)
            # 软标签（云模型输出）
            soft_labels = cloud_model(batch_x)
            # 蒸馏损失
            distill_loss = kl_divergence(client_model(batch_x), soft_labels)
            # 综合损失
            total_loss = lambda_d * hard_loss + (1 - lambda_d) * distill_loss
            # 反向传播更新客户端模型
            client_model.update(total_loss)
        # 早停检查
        if not validation_loss_improves():
            break
    return client_model

14a.5 Colla的扩展性分析

随着设备数量增长，Colla的扩展性面临以下挑战：

  • 云模型更新开销：设备数N增加时，云端需要聚合更多模型参数，但通过异步更新和批量处理可缓解。
  • 模型分组复杂度：Affinity Propagation的时间复杂度为O(N²)，当N极大时需近似算法。
  • 通信瓶颈：所有设备与云通信可能造成网络拥塞，可通过分层聚合（引入边缘节点）解决。
  • 设备异构性：不同设备的计算能力和数据量差异巨大，需要自适应的训练和上传策略。

附加A. 深度架构剖析

A.1 多特征RNN的张量流

M-RNN模型在每个时间步t接收六维特征输入。设嵌入维度分别为d_loc, d_h, d_m, d_w, d_dur, d_gap（对应位置、小时、分钟、星期、时长、间隔）。在大模型中均为32，小模型中均为4。

输入张量构造过程：

  1. 每个特征通过查找表得到嵌入向量：e_loc ∈ R^{d_loc}, e_h ∈ R^{d_h}, ...
  2. 拼接所有嵌入：X_t = concat([e_loc, e_h, e_m, e_w, e_dur, e_gap]) ∈ R^{6d}
  3. 输入LSTM单元：(h_t, c_t) = LSTM(X_t, (h_{t-1}, c_{t-1}))
  4. 最后时间步的隐状态h_n经过FC层映射到|V|维概率分布

A.2 LSTM单元的内部计算

LSTM的标准计算公式：

\[ f_t = \sigma(W_f \cdot [h_{t-1}, X_t] + b_f) \]

\[ i_t = \sigma(W_i \cdot [h_{t-1}, X_t] + b_i) \]

\[ \tilde{c}_t = \tanh(W_c \cdot [h_{t-1}, X_t] + b_c) \]

\[ c_t = f_t \odot c_{t-1} + i_t \odot \tilde{c}_t \]

\[ o_t = \sigma(W_o \cdot [h_{t-1}, X_t] + b_o) \]

\[ h_t = o_t \odot \tanh(c_t) \]

其中f_t是遗忘门，i_t是输入门，o_t是输出门，c_t是细胞状态。

附加B. 工作流程详解

B.1 启动阶段（Bootstrapping）

启动阶段是Colla系统初始化的关键步骤。云端使用初始数据集训练第一个云模型M_0。这一阶段仅在系统部署时执行一次。

初始数据可以来源于：

  • 匿名用户数据
  • 公开数据集（如word corpus用于键盘应用）
  • 历史归档数据

B.2 首次拉取阶段（First Pulling）

当新设备d_i在时间T_0加入系统时，向云端请求最新模型。云端压缩M_0为小模型并发送给设备。设备使用此压缩小模型作为第一个客户端模型m_{i0}进行推理。

B.3 客户端模型更新阶段

设备d_i在收集到足够数据或经过固定周期T后，执行以下步骤：

  1. 从云端拉取最新云模型M_1
  2. 使用M_1作为教师模型，通过知识蒸馏在本地数据上微调当前客户端模型m_{i0}
  3. 生成新客户端模型m_{i1}
  4. 将m_{i1}的参数推送至云端
  5. 使用m_{i1}进行后续推理至时间T_2

B.4 云端模型更新阶段

云端接收到N个设备模型参数后，通过以下步骤更新自身：

  1. 使用N个设备模型作为教师模型集合
  2. 在云数据集上执行多教师知识蒸馏
  3. 更新基础云模型M_1为M_2
  4. 对N个设备模型进行分组（Affinity Propagation）
  5. 每组生成一个分类云模型M_{2k}
  6. 记录设备到组的映射关系

附加C. 评估方法学

C.1 评估指标详解

预测准确率（top-1 accuracy）：正确预测占总预测的比例。

加权精度（weighted-precision）：

\[ Pr = \sum_{i=1}^{|V|} w_i \cdot \frac{TP_i}{TP_i + FP_i} \]

加权召回率（weighted-recall）：

\[ Re = \sum_{i=1}^{|V|} w_i \cdot \frac{TP_i}{TP_i + FN_i} \]

其中w_i = (i) / Σ_i (i) 是类别i的样本占总样本的比例。

C.2 数据划分策略

数据集按月份划分：

  • 第1月：初始训练集（云端使用）
  • 第2-4月：分为4部分用于移动性预测的循环评估
  • 每次循环：当前部分作训练集，下一部分作评估集
  • 共3个循环用于移动性预测，2个循环用于应用启动预测

C.3 设备选择策略

选择10,000台每月至少有10天活跃的设备进行评估。其中关注最活跃的100台设备（Top-100）。这种选择基于：

  • 活跃设备数据丰富，便于评估个性化效果
  • Top-100代表理想用户群体，性能上限
  • 整体10,000设备代表平均用户，性能基线

云-设备通信使用RESTful API，主要包含以下接口：

  • GET /version: 检查模型版本兼容性

附加G. 理论分析

G.1 收敛性分析

Colla的收敛性可以从以下两个角度分析：

客户端模型收敛：在本地数据集上，使用云模型作为正则项的客户端模型优化是一个标准的带正则化的随机梯度下降问题，在适当学习率下保证收敛。

云端模型收敛：云端使用多教师蒸馏，可视为多任务学习。每个客户端模型贡献一个任务（其对应分组的数据分布），云端学习的是这些任务的综合表示。

G.2 泛化误差界

对于客户端模型，其在本地数据分布D_i上的泛化误差可以用以下界估计：

\[ \mathbb{E}[\text{err}(m_i)] \leq \text{err}_{train}(m_i) + O(\sqrt{(VC(m_i) + \ln(1/\delta))/n_i}) \]

其中VC(m_i)是模型m_i的VC维（与参数量相关），n_i是本地数据量。

知识蒸馏可视为引入了先验信息，等价于减小了有效VC维，从而降低了泛化误差。

G.3 通信复杂度分析

Colla的通信复杂度：

  • 每周期通信量：O(N · |params_client|) = O(N · 130000 · 4) = O(520 KB · N)
  • 相比联邦学习：O(N · |params_full|) ≈ O(2 MB · N)，Colla节省约75%
  • 相比CT：O(N · |data|) ≈ O(GB · N)，Colla节省数千倍
