1. 文档信息速览

Self-Evolution 是一个面向领域问答的轻量级 LLM 自进化微调框架，由中国移动、南开大学、清华大学和微软联合提出，发表于 ISSRE 2024。

核心思想：使用 7B 参数量级的开源模型（Qwen1.5-7B-Chat），通过多轮迭代的自生成指令数据 + IFD 评分筛选高质量数据 + 微调，实现模型能力的逐步进化。

2. 研究背景

2.1 背景

大语言模型在通用问答上表现卓越，但在特定领域常因缺乏领域知识而表现不佳。例如 GPT-4 在回答法律问题时可能引用错误法条。

企业面临双重挑战：1）商业模型 API 涉及数据隐私泄露风险；2）72B 等大模型需要 5 张 V100 才能推理，部署成本极高。

2.2 现有方法局限

  • 指令微调需大量人工标注数据
  • 用大模型生成数据无法保证正确性和多样性
  • 自生成方法局限于模型已有的知识范围
  • 全参数微调资源需求大、容易过拟合

3. 论文目的与解决的问题

目标：用轻量 7B 开源模型通过自动化方法实现领域知识对齐，无需人工标注，保护数据隐私。

  1. 计算资源限制：大模型微调部署成本高
  2. 高质量数据稀缺：人工标注成本高
  3. 多样性和正确性：自生成数据质量参差
  4. 数据隐私：领域数据含敏感信息
  1. 选7B开源模型（单V100可完成全部阶段）
  2. LLM基于文档生成QA对
  3. IFD评分筛选高质量数据
  4. 迭代进化：生成→筛选→微调循环
  • 选择 7B 模型兼顾性能与成本
  • 无需人工标注数据
  • 迭代进化超强基线 72B 模型
  • 中国移动 117 天生产验证

4. 核心原理（含数学公式）

4.1 三阶段框架

Self-Evolution 包含三个核心阶段，循环迭代执行：

4.2 阶段一：QA生成

基于领域文档生成问答对：\( q_{ij} = LLM(\theta_i, t_j) \)，其中 t_j 是第 j 份文档。

问题生成提示强调：简洁性（不要子问题）、可回答性（确保文档包含答案）。

答案生成结合文档：\( a_{ij} = LLM(\theta_i, t_j, q_{ij}) \)，强调完整性和准确性。

4.3 阶段二：数据筛选

使用 IFD 分数筛选高价值数据：

\( s_\theta(A|Q) = -\frac{1}{N}\sum_i \log P(w_i^A|Q, w_1^A, ..., w_{i-1}^A; \theta) \)

\( s_\theta(A) = -\frac{1}{N}\sum_i \log P(w_i^A|w_1^A, ..., w_{i-1}^A; \theta) \)

\( IFD_\theta(Q, A) = \frac{s_\theta(A|Q)}{s_\theta(A)} \)

IFD 分数衡量模型在给定问题条件下回答的困难程度，分数越高表示这条数据对模型越有价值。

4.4 阶段三：迭代微调

每轮使用前 i-1 轮中 IFD 分数最高的 k 条历史数据 + 本轮新生成的 D_i 进行 LoRA 微调。

迭代停止条件：达到预设迭代次数上限或模型性能不再提升。

6. 关键算法详解

6.1 伪代码

算法: Self-Evolution

输入: 初始模型θ_0, 文档集T, 迭代轮数N, IFD阈值
输出: 训练好的领域模型

for i in 1..N:
  // 阶段一: QA生成
  D_i = {}
  for t_j in T:
    q = LLM(θ_i, t_j)  // 生成问题
    a = LLM(θ_i, t_j, q)  // 生成答案
    D_i.append((q, a))

  // 阶段二: 数据筛选
  IFD_scores = {}
  for (q, a) in (D_0 ∪ ... ∪ D_{i-1}):
    score = compute_IFD(θ_i, q, a)
    IFD_scores.append(score)
  top_k = select_top_k(IFD_scores)

  // 阶段三: 微调
  train_data = top_k ∪ D_i
  θ_{i+1} = LoRA_finetune(θ_i, train_data)

return θ_N

策略 描述 效果

9.1 关键发现

  • 历史数据需要筛选而非全量使用
  • 随机选择不稳定，仅早期有效
  • IFD 是有效的数据质量度量
  • 历史数据加入可防止灾难性遗忘

补充: 评分计算

输入: 模型θ, 问题q, 文档t
输出: 条件答案得分

s_cond = -1/N * sum(log P(w_i|q, w_1..w_{i-1}; θ))
s_direct = -1/N * sum(log P(w_i|w_1..w_{i-1}; θ))
IFD = s_cond / s_direct
return IFD

Self-Evolution的系统部署涉及多层架构协同：数据采集层建议使用 OpenTelemetry 或 Prometheus 作为统一采集协议，避免不同厂商私有协议导致字段不一致。采样间隔建议1分钟，过短增加存储压力，过长丢失短时异常。

数据存储层推荐使用列式时序数据库如 InfluxDB 或 TDengine，相比关系数据库可节省5-10倍存储空间。建议按指标+实体+时间三维分区，并设置合理的数据保留策略（如热数据7天、温数据30天、冷数据归档至对象存储）。

预处理层应作为独立微服务便于横向扩展。流水线包括：数据校验→缺失值填充→异常值裁剪→归一化→窗口化。每步应有指标埋点便于排查。使用 Apache Flink 或 Kafka Streams 实现流式处理。

  • 告警去重率>80%
  • 模型重训周期7天
  • 误报率<5%，漏报率<2%
