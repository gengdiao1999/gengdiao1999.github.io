二、背景（Background）

大型在线服务系统需要频繁软件变更。Google SRE 报告约 70% 事故与软件变更相关。变更前经过 code review、unit test、integration test，但生产环境的集群规模、组件交互、资源竞争、OS/库版本、意外负载等差异仍可能让 bug 漏过。部署后监控至关重要。

传统方法多关注业务 KPI 异常（响应时间等），存在三大问题：(1) 等到业务 KPI 异常时已造成损失（MTTD 可达 5+ 天）；(2) 3σ、Holt-winters 通用方法未针对变更场景；(3) 商业产品（Dynatrace、Datadog）也未解决。

论文对某大型商业银行 5 个服务系统两年内工单做了首次大规模实证：约 50.4% 事故由变更引起；坏变更根因：代码缺陷（38%）、配置错误（31%）、版本不兼容（11%）、资源竞争（9%）、其他（12%）。

三、目的（Problems Solved）

  • 多源数据融合：业务 + 机器 + 日志。
  • 早期检测：不等业务 KPI 异常。
  • 复杂模式：无监督学习。
  • 预期变更过滤：知识库区分。
  • 可解释告警：健康分数 + Top-K 异常。
  • MTTD 降低：20.4%~60.7%。

四、核心原理（Principles）

系统总览：4 步——数据准备、多模态异常检测、告警 + 报告、行动决策。

关键概念：Software Change、Bad Change、Expected Change、KPI、Multimodal LSTM、Drain、k-σ、MTTD、Knowledge Base。

数学原理：

  • LSTM 单模态：

\[\]

\(\phi^B = \text{LSTM}(X^B_{t-w:t}),\ \phi^M = \text{LSTM}(X^M_{t-w:t}),\ \phi^L = \text{LSTM}(X^L_{t-w:t})\)


\[\]
  • 多模态融合：

\[\]

\(F(f^B, f^M, f^L) = \text{FC}([f^B; f^M; f^L])\)


\[\]
  • 多模态 LSTM 损失：

\[\]

\(L = \frac{1}{n} \sum_{i=1}^{n} [(P^B_i - X^B_i)^2 + (P^M_i - X^M_i)^2 + (P^L_i - X^L_i)^2]\)

  • k-σ 阈值：

\[\]

\(\text{threshold} = \mu + k \sigma\)


\[\]
  • 健康分数：

\[\]

\(\text{health\_score} = \max(100 - 100 \cdot \frac{x - \min}{\max - \min + \epsilon}, 0)\)

def compute_anomaly_score(model, data):
    return np.abs(model.predict(data) - data)

def alert(score, mu, sigma, k=1, alpha=3):
    thresh = mu + k * sigma
    flags = (score > thresh).astype(int)
    consec = np.convolve(flags, np.ones(alpha), mode='same')
    return (consec >= alpha).astype(int)

def decision(alert_flag, change_op, knowledge_base):
    if change_op in knowledge_base:
        return 'expected', knowledge_base[change_op]
    return 'unknown', None

def health_score(score, train_min, train_max):
    return max(100 - 100 * (score - train_min) / (train_max - train_min + 1e-9), 0)

八、关键创新点（Key Innovations）

  • + 首次大规模实证：5 服务 2 年，揭示 50.4% 事故由变更。
  • + 多模态 LSTM 融合：业务+机器+日志三路 LSTM + 共享 FC。
  • + 连续点告警：避免瞬时噪声。
  • + 健康分数 + Top-K：可解释。
  • + 预期变更过滤：知识库 + StepWise。
  • + MTTD 大幅降低：20.4%~60.7%。

九、实验与结果（Experiments）

  • 数据集：Train-Ticket（30+ 微服务）+ E-commerce；2 数据集 A（121+96）、B（125+98）。
  • Baseline：Gandalf、Funnel、Lumos、单源（Donut/B-LSTM/LSTM-NDT/OmniAnomaly/DeepLog）、多源（M-AE/M-LSTM/Multimodal AE）。
  • 指标：P/R/F1/MTTD/训练/检测时间。
  • 结果：SCWarn A F1=0.93 MTTD=5.1；B F1=0.97 MTTD=2.3。Gandalf 0.79/0.87，Funnel 0.73/0.81，Lumos 0.78/0.82。MTTD 平均 3.7 min，降低 20.4%-63.7%。多模态 LSTM vs M-LSTM p=0.0416，effect size=0.6860；vs Multimodal AE p=0.0019，effect size=0.8388。
  • 消融：单源 0.66-0.93；多源 0.81-0.97。SCWarn 训练 3.43-4.84 min，检测 0.86-0.94 s，比 OmniAnomaly（28.11 min）、DeepLog（50.65 min）快得多。
