1.1 核心贡献

本文针对数据库内部合法用户的异常行为检测问题，提出了一种基于核密度估计的无监督学习方法，能够在无标注的海量历史日志中有效发现简单异常和复杂异常。
  1. 问题创新：聚焦数据库内部合法用户的异常行为，区别于传统的入侵检测（针对外部攻击）
  2. 方法创新：提出基于核密度估计的无监督检测框架，同时支持简单异常（单维度）和复杂异常（多维度）的检测
  3. 特征工程创新：设计了一套针对数据库异常行为的特征工程方案，特征维度不随数据规模膨胀
  4. 实战验证：在真实数据上检测出了未被安全运维人员注意的新的复杂异常

2. 研究背景

2.1 数据库安全的内忧外患

数据库存储了企业的大量敏感数据，涉及金融、财务、电子商务、在线支付等核心业务。数据泄露的危害已严重到企业不可承受之重。

数据泄露的来源主要分为两类：

类型 特点 典型场景
外部攻击 来自企业外部黑客的入侵 SQL注入、暴力破解、漏洞利用
内部合法用户异常 企业内部员工的恶意或不当行为 数据拖取、越权访问、违规导出

研究表明，由内部用户对数据库（包括密级数据）进行合法访问，对整个系统非常熟悉，其恶意滥用权限、偷取数据、泄露数据将造成更致命的攻击和严重的损失。

2.2 现有防护策略的局限

现有的数据库安全防护策略（或工具）主要分两类：

  1. 基于规则与鉴权的方法：能防止用户访问未授权数据，但无法防范合法用户对已授权数据的滥用
  2. 基于机器学习的异常检测：需要有标注的数据集进行模型训练，但面临"三大挑战"使得有监督方法难以应用

2.3 数据库行为异常检测的三大挑战

挑战1：攻击模式不确定

为了规避检测，内部攻击者会将其恶意的行为隐藏在正常的行为中，不易发现。攻击策略没有固定模式，无法事先预知。

挑战2：真实异常样本极少

由于内部员工叛变或黑客冒充合法用户攻击是小概率事件，可获取的真实异常事件样本极少；正负样本数量极不平衡，历史日志中绝大部分都是正常行为的数据。异常样本数占比例极低，不足以准确描绘出异常的特征。

挑战3：人工标注非常困难

真实环境中采集的数据通常没有准确的"正常"或"异常"标记。要将海量无标注的历史日志转化为有准确标注的数据集是非常困难的，需要非常丰富的领域知识和大量的时间、人力投入。

而且，这类标注比图像分类等问题的标注更加复杂、抽象，甚至还有一些是有领域知识也无法覆盖的未知异常，导致人对异常的认识本身存在局限性，难以准确标注。

3. 论文目的与解决的问题

3.1 异常定义

论文对数据库用户行为"异常"的定义：

当用户的操作行为严重偏离其绝大多数正常历史行为的轨迹时，或者当用户的操作行为具有拖取数据、滥用数据等攻击嫌疑时，对数据库的安全构成威胁的，称为"异常"。

3.2 形式化问题描述

假设绝大部分正常行为的数据服从某个未知的分布 \(X\)，并函数 \(y = f(x)\) 表示其未知的概率密度函数。采集的历史数据集 \(D\) 可以认为是由分布 \(X\) 中独立抽取一定数量的样本构成的。

那么用户行为异常检测的问题，等价于判断给定的任意样本是否属于分布 \(X\)。若属于，则判定该样本为正常；若不属于，则判定该样本为异常。

3.3 异常分类

根据异常的复杂程度、检测难易程度，论文将异常分为两类：

异常类型 定义 典型场景
简单异常 通过单维度分析就能够直接判定的异常，主要表现为单个特征指标上出现的偏离绝大多数正常样本的离群值 大量数据拖取异常（"访问数据量"维度极值）
复杂异常 通过单维度分析无法判定的异常，需要将多个特征指标组合分析才能发现，比如"24小时连续访问"或先未知的新异常 异常的时间分布模式、多维度组合异常

3.4 设计目标

  1. 无需历史标注：无需对历史数据进行费时费力的人工标注
  2. 不平衡样本鲁棒：在正负例不平衡（正例数量远大于负例数）、真实异常样本缺失、数据集缺少标注或完全无标注的情况下仍能检测
  3. 个性化检测：能区分不同用户（或用户群体）分别训练模型，且支持定期重训练实现阈值自动更新
  4. 简单与复杂异常一体化：单维度建模检测简单异常，多维度建模检测复杂异常
  5. 真实数据验证：能在真实生产环境数据集上验证有效性

4. 核心原理（含数学公式）

4.1 核密度估计基本原理

核密度估计（Kernel Density Estimation, KDE）是一种用于估计随机变量未知概率密度函数的非参数检验方法。

假设从概率密度为 \(f\) 的未知分布中独立地抽取 \(n\) 个样本 \((x_1, x_2, ..., x_n)\)，则可以通过公式(1)得到该分布概率密度函数 \(f\) 的估计值：

\[ \hat{f}_b(x) = \frac{1}{n} \sum_{i=1}^{n} K_b(x - x_i) = \frac{1}{nb} \sum_{i=1}^{n} K\left(\frac{x - x_i}{b}\right) \tag{1} \]

其中：

  • \(K\) 为核函数，是积分值为1的非负函数
  • \(b\) 是带宽，是一个平滑参数
  • 核密度估计可以简单地理解为将每个样本作为中心建立的核函数的加权和

4.2 核函数选择：高斯核

由于本文关注的是正常行为的整体分布情况，考虑到高斯核函数能够使估计的分布更加平滑，因此实验中核函数 \(K\) 选用高斯核函数：

\[ K(u) = \frac{1}{\sqrt{2\pi}} \exp\left(-\frac{u^2}{2}\right) \]

代入公式(1)：

\[ \hat{f}_b(x) = \frac{1}{nb\sqrt{2\pi}} \sum_{i=1}^{n} \exp\left(-\frac{(x - x_i)^2}{2b^2}\right) \]

4.3 带宽b的影响

带宽 \(b\) 的取值能够直接影响密度估计的效果：

带宽取值 影响
\(b\) 值过大 得到的概率密度估计曲线终将过于平滑，无法反映出局部区域内的变化和差异
\(b\) 值过小 得到的概率密度估计曲线终将过拟合，即曲线中出现过多、过于细微的波动
\(b\) 值适中 能够很好地反映数据的真实分布

带宽 \(b\) 有多种调参方法。实验中采用了python的超参自动搜索模块GridSearchCV对参数 \(b\) 进行调优。

4.4 单维度概率密度模型

单维度概率密度模型用于检测单维度简单异常。简单异常通常是某些重要特征指标在单维度上的离群值。

单维度概率密度模型训练时使用单个特征指标对应的单维度历史数据为训练集，用核密度估计算法训练得到该维度的概率密度分布曲线，通过找出概率密度值极低且数量很少的少数离群样本，实现检测出简单异常的目的。

单维度概率密度判定阈值 \(S_0\) 可以根据可疑异常率 \(a\) 求出。例如，若假定数据集中可疑异常率为 \(a = 1\%\)，计算所有历史样本在该维度的概率密度值并按序排列后，处在前1%位置的样本对应的概率密度即为相应的概率密度判定阈值。

检测时，计算新样本在该维度的概率密度值 \(P\)：

\[ \begin{cases} P < S_0 & \Rightarrow \text{异常告警，进入人工排查与取证追责流程} \\ P \geq S_0 & \Rightarrow \text{正常，不为该简单异常} \end{cases} \]

4.5 多维度概率密度模型

多维度概率密度模型用于检测复杂异常。与单维度概率密度模型不同，多维度同时考察样本的多个维度联合起来分析，考察样本在高维空间的概率密度分布情况；通过找出高维空间中分布稀疏的样本，从而发现一些潜在的复杂异常，其中可能包括未察觉的复杂异常。

设样本第 \(i\) 个特征（即第 \(i\) 个维度）的概率密度函数为 \(d_i(x)\)，样本的维度数为 \(m\)，则该样本的整体概率密度 \(D\) 可由公式(2)求得：

\[ D(x) = \prod_{i=1}^{m} d_i(x) \tag{2} \]

本文通过公式(2)建立的概率密度模型称为多维度概率密度模型。严格地说，公式(2)中的 \(m\) 个特征之间满足条件独立时，样本的整体概率密度等于各个单维度的概率密度值相乘。

由于本文选取的特征彼此之间没有明显的依赖关系，可以利用公式(2)来近似计算样本的整体概率密度。这种近似计算方法比简单、开销更小，有助于快速找出高维空间中概率密度极小的样本。实验中也取得了不错的效果。

4.6 多维度判定阈值

多维度概率密度判定阈值 \(M_0\) 可以同理求得：设可疑异常率为 \(a\)，计算所有历史样本对应的多维度概率密度值并升序排列后，排在前 \(a\) 位置的样本对应的多维度概率密度值即为相应的多维度复杂异常的概率密度判定阈值 \(M_0\)。

检测时：

\[ \begin{cases} P < M_0 & \Rightarrow \text{多维度复杂异常告警} \\ P \geq M_0 & \Rightarrow \text{不为多维度异常} \end{cases} \]

4.7 整体用户模型与单用户模型

本文区分了两种模型策略：

模型类型 训练数据 检测语义 适用场景
单用户模型 单个用户的历史数据 偏离该用户个人行为的异常点 训练样本充足的活跃用户
整体用户模型 所有用户的历史数据 偏离所有用户行为的"离群中的离群" 新用户冷启动、不活跃用户、强级别异常检测

整体用户模型检测出的异常严重程度更高、级别更高。两类模型互为补充。

6.1 特征工程算法

1 用户角色 N/A N/A

特征提取伪代码

算法1: 数据库用户行为特征提取
输入: 原始日志 L = {(uid, sql, timestamp, tables, columns, perm_result)}
      时间窗口大小 W ∈ {day, 2hours}
输出: 用户特征矩阵 X ∈ R^{N×D}, 用户列表 Users

1: // 步骤1: 按用户分组
2: user_logs = group_by(L, key=uid)

3: // 步骤2: 对每个用户提取特征
4: X = []
5: for user, logs in user_logs:
6:     // 按时间窗口W分桶
7:     for window_t in sliding_windows(logs, W):
8:         window_logs = filter(logs, in_window=window_t)
9:
10:        // 提取属性特征
11:        f1 = user.role
12:        f2 = user.work_status  // 在岗/离职审批中/已离职
13:        f3 = count_distinct(window_logs, key=ip_location)
14:
15:        // 提取操作统计特征
16:        f4 = sum([log.data_volume for log in window_logs])
17:        f5 = count_distinct([table for log in window_logs for table in log.tables])
18:        f6 = count(window_logs, filter=perm_failed) / count(window_logs)
19:        f7 = count(window_logs, filter=access_top_secret) / count(window_logs)
20:
21:        sample = [f1, f2, f3, f4, f5, f6, f7]
22:        X.append((user, window_t, sample))

23: // 步骤3: 标准化
24: for d in feature_dimensions:
25:     mean_d = mean(X[:, d])
26:     std_d = std(X[:, d])
27:     X[:, d] = (X[:, d] - mean_d) / std_d

28: return X, Users

聚合粒度选择

本文中按"天"为单位提取特征，即用户一天内所有的日志数据提取生成一个样本，同时还将表1编号4-7的这四个重要特征分解为12个按"两小时"统计的特征值。这样可在不同时间粒度上观察异常行为。

4: grid_search = GridSearchCV(kde, param_grid, cv=5)
5: grid_search.fit(v.reshape(-1, 1))
6: best_b = grid_search.best_params_['bandwidth']

7: // 步骤2: 用最优带宽训练KDE模型
8: model = KernelDensity(kernel='gaussian', bandwidth=best_b)
9: model.fit(v.reshape(-1, 1))

10: // 步骤3: 计算所有历史样本的概率密度
11: log_dens = model.score_samples(v.reshape(-1, 1))
12: dens = np.exp(log_dens)

13: // 步骤4: 按概率密度升序排序，取前a位置作为阈值
14: sorted_dens = np.sort(dens)
15: threshold_idx = int(N * a)
16: S_0 = sorted_dens[threshold_idx]

17: return model, S_0

6.3 多维度KDE训练算法

算法3: 多维度KDE模型训练
输入: 多维度特征矩阵 X ∈ R^{N×m}
      可疑异常率 a = 1%
输出: 多维度模型集合 models, 判定阈值 M_0

1: // 步骤1: 为每个维度训练单维度KDE模型
2: models = {}
3: for i in range(m):
4:     model_i, _ = 单维度KDE训练(X[:, i], a)
5:     models[i] = model_i

6: // 步骤2: 计算每个样本的多维度概率密度
7: D = np.ones(N)
8: for i in range(m):
9:     log_dens_i = models[i].score_samples(X[:, i].reshape(-1, 1))
10:    D *= np.exp(log_dens_i)

11: // 步骤3: 对多维度概率密度取对数（避免数值下溢）
12: log_D = np.log(D + 1e-300)

13: // 步骤4: 按概率密度升序排序，取前a位置作为阈值
14: sorted_log_D = np.sort(log_D)
15: threshold_idx = int(N * a)
16: M_0 = sorted_log_D[threshold_idx]

17: return models, M_0

6.4 在线检测算法

算法4: 实时异常检测
输入: 新样本特征 x ∈ R^m
      单维度模型集合 single_models 和阈值 S_0_list
      多维度模型集合 multi_models 和阈值 M_0
输出: 异常类型列表 anomaly_types

1: anomaly_types = []

2: // 步骤1: 单维度简单异常检测
3: for i in range(m):
4:     if i in [访问数据量, 访问表数, 鉴权失败比例, 绝密列比例]:
5:         log_p = single_models[i].score_samples([[x[i]]])[0]
6:         p = np.exp(log_p)
7:         if p < S_0_list[i]:
8:             anomaly_types.append(('简单异常', feature_name_i))

9: // 步骤2: 多维度复杂异常检测
10: log_D = 0
11: for i in range(m):
12:     log_p_i = multi_models[i].score_samples([[x[i]]])[0]
13:     log_D += log_p_i

14: if log_D < M_0:
15:     anomaly_types.append(('复杂异常', '多维度'))

16: return anomaly_types

7.4 工程优化

  1. 对数空间计算：多维度概率密度连乘易导致下溢，转为对数空间求和
  2. 稀疏存储：用户的稀疏访问模式用稀疏矩阵存储

7.5 数据预处理细节

  • 标准化处理：由于不同的特征维度的度量单位有所不同，在进行模型训练之前需对特征数据进行标准化处理。即对每一维度，先求出其均值和方差，并将该维度的原始数据减去均值后再除以方差。
  • 缺失值处理：用户未在某时间段访问数据库时，相应特征值为0
  • 异常值过滤：去除明显错误的日志记录（如时间戳异常）

8. 实验设计与结果

8.1 实验环境

本文以某大型互联网公司的一个重要业务数据库（数据仓库）作为实验场所。该数据库上线半年左右，共有87名用户（均为具备合法访问权限的内部员工），暂时无法靠人工方法抓取到真实内部攻击的异常案例，且采集的数据集完全没有标注。

考虑到该数据库历史真实攻击事件发生的概率低、安全要求较高，结合安全运维人员根据历史观察和运维经验综合评估后，将可疑异常率设定为1%。

8.4 评估指标

由于数据集完全无ground truth，本文以Precision（查准率）为主要评估指标：

\[ \text{Precision} = \frac{TP}{TP + FP} \tag{3} \] \[ \text{Recall} = \frac{TP}{TP + FN} \tag{4} \]

由于真实环境中存在哪些异常、是否有未知异常等不可准确掌握，FN无法准确评估，因此Recall无法计算。

本文区分两种查准率：

查准率类型 TP定义 说明
严格查准率 只考虑"强显著"异常 有明显攻击嫌疑或高风险威胁、且概率密度极低的样本
宽松查准率 "强显著"+"弱显著"异常都算 有潜在攻击嫌疑或高风险威胁、概率密度低的样本也算

按1%可疑异常率，三种简单异常共检测出60个可疑样本（每类20个）。由安全运维人员对这60个可疑异常进行逐个人工排查后：

大量数据拖取 0.532ms 95% 100%

8.7 多维度异常检测结果

整体用户多维度概率密度（取对数）的累积分布显示，历史数据的整体概率密度值（取对数）大部分集中于(-20,7)，根据实验结果中选取的可疑异常率，求出概率密度判定阈值 \(M_0 = 6.23E-52\)。

检测到的20个多维度可疑样本中，发现整体概率密度值最低的20个样本中：

  • 19个为"24小时连续访问"异常（单维度无法检测的新型复杂异常）
  • 1个为"频繁鉴权失败"异常（属于通过单维度也能检测到的简单异常）

8.8 新发现的复杂异常分析

这20个样本当天内每两小时访问表数显示，除第15个样本外，其它样本在一天内的任意两小时内访问表的操作基本没有间断。

通常情况下，用户一天的访问行为有一定规律性，主要访问集中于白天，在下班休息时间一般极少出现访问行为，深夜加班或临时任务等特殊情况更少见。即便是喜欢在夜间工作的"夜猫子"，一天之内至少有一段休息时间。

因此，24小时连续访问是一种非常可疑的、少见的异常。如果这些异常背后不是人为操作，而是自动化程序在访问数据库，那么这种异常和真实的攻击就十分接近，存在极大的安全风险，将可能导致极为严重的危害。

因此，安全运维人员认定这19个"24小时连续访问"样本具有极大的攻击嫌疑和极高的风险威胁，属于"强显著"的异常，并进行了更加深入的二次调查。

8.9 排查结论

经过详细的调查取证后确认，这19个"24小时连续访问"强显著异常是由某平台管理员采用自动化程序对数据的导入、导出和格式转换等工作所导致。结合该用户的访问详情，并与其它业务行为关联分析后最终确认，该用户的行为没有恶意的动机，不属于恶意的攻击。

然而，通过模型检测出的"24小时连续访问"异常本身具有非常大的意义，且非常必要：

  • 这类异常之前完全没有被察觉和关注
  • 没有预设任何固定规则可以将此类异常进行有效防御
  • 多维度模型找出的新种类的"复杂异常"具有非常的发现价值

12. 局限性与未来工作

12.1 当前局限性

  1. 数据集规模较小：本文实验仅87名用户、1973个样本，需更大规模数据进一步验证
  2. 无ground truth：缺少真实异常标签使得Recall无法准确评估
  3. 特征独立假设：多维度KDE假设特征独立，但实际特征间可能有依赖
  4. 单用户模型样本不足：低活跃度用户（占43%）无法建立单用户模型
  5. 特征工程依赖经验：特征选取依赖安全运维经验，自动特征工程仍待研究
  6. 时间窗口固定：1天和2小时是固定的，无法适应不同攻击模式的时间尺度
  7. 新异常解释难：发现的新异常仍需要人工分析理解

12.2 未来研究方向

方向1：深度无监督学习

  • 引入Variational Auto-Encoder建模用户行为
  • 使用LSTM建模用户的行为序列
  • 探索Transformer在用户行为表示上的应用

方向2：自动特征工程

  • 研究自动从SQL日志中提取特征的方法
  • 结合NLP技术理解SQL语义
  • 动态调整特征集，适应业务变化

方向3：联邦学习

  • 跨数据库联邦训练模型，提升通用性
  • 同时保护各数据库的数据隐私

方向4：增量学习

  • 在线增量更新模型，无需全量重训
  • 对概念漂移自适应

方向5：因果分析

  • 不仅检测异常，还分析异常的根因
  • 结合服务调用关系做更精细的归因

方向6：异常解释自动化

  • 使用LLM对检测出的异常进行自然语言解释
  • 自动生成异常调查报告

13. 总结与启示

13.1 论文核心贡献

  1. 问题创新：聚焦数据库内部合法用户的异常行为检测，这一问题在已有研究中关注不足
  2. 方法创新：提出基于核密度估计的无监督检测框架，巧妙结合单维度和多维度建模
  3. 工程创新：特征工程方案使特征维度固定，不随数据规模膨胀，具备良好的工程实用性
  4. 实证创新：在真实生产数据上验证有效性，并发现了之前未被察觉的新型复杂异常

13.2 方法学启示

启示1：无监督学习的实用价值

在真实场景中，标注成本极高甚至不可能。本文证明：合适的无监督方法（如KDE）配合精心的特征工程，可以达到接近有监督方法的效果。

启示2：简单方法的强大

KDE是一个相当"传统"的方法（1956年Rosenblatt提出），但本文证明：

  • 不必追求最复杂的深度学习方法
  • 针对问题特性的简单方法往往更有效
  • 可解释性、低计算开销也是重要价值

启示3：分层异常的设计思想

将异常分为"简单异常"和"复杂异常"，分别用不同维度建模：

  • 简单异常用单维度KDE快速检测
  • 复杂异常用多维度KDE发现潜在威胁
  • 这种分层设计为其他领域提供了借鉴

启示4：发现未知比检测已知更有价值

本文的最有价值发现不是检测出已知的数据拖取异常，而是发现了"24小时连续访问"这一全新的异常模式。这告诉我们：

机器学习异常检测的核心价值，不仅在于自动化已有的规则检测，更在于发现人类未察觉的新模式。

13.3 对数据库安全的启示

启示 说明
内部威胁是数据库安全的重大风险 合法用户的恶意行为难以用传统访问控制防范
行为分析是必要的补充 不依赖规则，直接学习用户行为模式
事后追溯与事前防范并重 异常检测既支持事后追责，也能辅助事前预警
无ground truth的挑战是常态 无监督方法应作为数据库安全的基础工具

13.4 产业价值

  • 金融行业：银行、证券、保险等对数据安全要求最高，本方法可直接应用
  • 互联网公司：用户数据、商业数据的内部访问监控
  • 政府机构：敏感数据库的访问审计
  • 医疗行业：患者隐私数据的访问监控
  • 云数据库服务商：作为云数据库的增值安全服务
