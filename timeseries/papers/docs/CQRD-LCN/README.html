二、研究背景

2.1 数据中心网络的挑战

现代数据中心承载着越来越多的互联网应用，从搜索引擎、电子商务、社交网络到云计算服务。这些应用对数据中心网络的性能提出了越来越高的要求：

  • 低延迟：实时交互式应用要求毫秒级响应
  • 高吞吐：大数据传输需要充分利用带宽
  • 稳定性：服务可用性需要保证99.99%以上
  • 可扩展性：网络规模需要支持数万到数十万服务器

2.2 数据中心流量的分类

流量类型定义占比典型应用性能要求
短流/小流≤ 100KB~90%搜索查询、Web请求、RPC调用低延迟（毫秒级FCT）
大流/长流> 100KB~10%文件传输、数据分析高吞吐量
巨流> 1MB~3%文件备份、数据复制、批处理高带宽利用

2.3 流干扰问题

在数据中心交换机中，当两个流来自不同输入端口但去往同一输出端口时，它们会争夺输出端口的队列内存和链路容量。论文定义了两种流干扰：

2.3.1 输出争用（Output Contending）

当两个流在同一交换机的同一输出端口在时间上重叠时，称为输出争用。这是最常见的争用形式。

2.3.2 路径争用（Path Contending / Switch Path Contending）

当两个流共享同一交换机路径（同一输入输出端口对）时，称为路径争用。路径争用是输出争用的特殊情况，会带来更严重的性能影响。

2.4 现有解决方案的分类

解决方案类别代表工作核心思想优势劣势
传输层速率控制D-CTCP、HULL、D²TCP、D³基于ECN或截止时间调整发送速率不改硬件需要修改主机协议栈
抢占式流调度PDQ、pFabric基于流优先级抢占调度性能最优需要彻底改造硬件和协议栈
基于交换机的队列管理DRR、SFQ、HCF在交换机内进行细粒度队列管理不需修改主机HCF不够细粒度
本论文（CQRD）—交叉点队列 + 随机丢弃简单、有效、易部署仍有小幅大流吞吐下降

2.5 主流方案HCF的局限

HCF（Hashed Credits Fair）是当时最先进的数据中心队列管理方案，但存在以下问题：

  • 使用哈希将流划分到bin中，可能多个小流被哈希到同一bin
  • 每个输出有HP（高优先级）和LP（低优先级）两个队列
  • 当大流耗尽信用后被切换到LP队列，可能阻塞小流
  • 不够细粒度，仍然存在流干扰问题

2.6 经典CQ交换机的优势与局限

经典CQ（Crosspoint-Queued）交换机为每对输入输出端口分配独立的缓冲区，具有以下优势：

  • 来自不同输入端口的流完全隔离
  • 使用简单的轮询调度即可实现公平性
  • 调度决策完全分布式，可扩展性好

但经典CQ交换机的局限在于：

  • 需要 $N^2$ 个独立缓冲区，存储开销大
  • 不能处理路径争用（同一输入输出对）
  • 不适合直接用于数据中心场景

三、论文目的与解决的问题

3.1 研究目标

CQRD的核心目标是设计一个简单、有效、易部署的数据中心交换机队列管理方案，解决流干扰问题，同时保持以下特性：

  1. 无需修改主机：不改变传输层或主机端协议栈
  2. 无需交换机协调：不需要多个交换机之间的协调
  3. 简单实现：对现有交换机的修改最小化
  4. 显著性能提升：明显降低短流的FCT
  5. 不影响大流吞吐：大流性能下降可控

3.2 解决的关键问题

问题挑战CQRD解决思路
如何隔离不同流？OQ无法隔离使用交叉点队列分离不同输入
如何处理同一路径上的流？CQ无法处理路径争用使用随机丢弃替代尾部丢弃
如何避免大量存储？纯CQ需要N²×L大缓冲使用小容量交叉点缓冲（4-8KB）
如何保证公平性？避免饥饿轮询（RR）调度
如何与现有系统兼容？避免大改仅需修改交换机队列管理逻辑

3.3 设计原则

CQRD的设计哲学是：从根本上解决流干扰问题（粗粒度队列管理），同时保持队列管理粒度与开销/成本之间的良好平衡。理想情况下，为每个流分配独立缓冲区可以完全消除干扰，但开销过大。CQRD通过交叉点队列实现交换机路径分离，通过随机丢弃缓解路径内干扰。

四、核心原理

4.1 基本概念定义

4.1.1 性能指标

  • 流完成时间（FCT, Flow Completion Time）：流从开始传输到完成传输所用的总时间，是短延迟敏感流的关键指标
  • 吞吐量（Goodput）：流大小除以FCT，等于流的平均传输速率，是大带宽贪婪流的关键指标

4.1.2 流的分类

沿用相关工作的惯例：

  • 小流/短流：< 100KB
  • 大流/长流：> 100KB
  • 巨流：> 1MB（大流的特例）

4.1.3 争用定义

在给定交换机上，定义两种争用：

  • 输出争用（Output Contending）：两个流去往同一输出端口且在时间上重叠
  • 交换机路径争用（Switch Path Contending）：两个流在同一交换机上具有相同的输入-输出端口对

定义 交换机路径（Switch Path） 为同一交换机上的输入-输出端口对。对于穿越多个交换机的流，如果在任一交换机上路径争用，则称为路径争用。

4.2 流干扰的根本原因

论文通过深入分析指出流干扰的根本原因是商品DCN交换机使用的粗粒度（输出队列OQ或OQ）队列管理方案。具体来说：

  1. 大流快速占用共享内存和输出链路容量
  2. 小流的数据包必须排在大流数据包后面等待
  3. 这导致小流不必要的延迟
  4. 当输出争用持续时，输出缓冲区被填满，开始丢弃所有短流的数据包
  5. 整体性能严重下降

4.3 OQ vs CQ 玩具示例

论文使用一个8端口OQ交换机的玩具示例说明问题：

  • 每个端口具有10Gbps双向链路速率，单向链路延迟4μs
  • 每个端口的小输出队列大小为36KB（共288KB）
  • 服务器1-8通过端口1-8连接
  • I6和I7有巨流去往O8（100MB文件传输使用TCP）
  • I1-I5有延迟敏感任务，5ms后每个发送10KB TCP流到O8

仿真结果

方案大流吞吐量小流FCT小流吞吐量
理想（无争用）~10Gbps~10μs~1Gbps
OQ~10Gbps~100ms~1Mbps
HCF~10Gbps~1ms~10Mbps
CQ (4.5KB/交叉点)~10Gbps~100μs~1Gbps

结果表明：

  • OQ场景下，小流FCT达到百毫秒级，性能极差
  • HCF在OQ基础上有所改进，但仍丢失短流数据包
  • CQ提供了完全的流分离，性能最佳

4.4 CQ的不足与CQRD的提出

虽然CQ在玩具示例中表现出色，但有以下不足：

  1. 需要大量缓冲：每个交叉点需要分配缓冲
  2. 不能处理路径争用：同一输入输出对的流仍在同一缓冲中竞争
  3. 不适合多级网络：在实际DCN中，路径争用很常见

为此，论文提出 CQRD（Crosspoint-Queue with Random Drop），结合了：

  • 交叉点队列：解决输出争用（不同输入间的隔离）
  • 随机丢弃：解决路径争用（同一输入间的公平性）

4.5 CQRD的核心思想

4.5.1 交换机路径分离

CQRD为每对输入输出端口分配独立的交叉点缓冲区（XB），具有相同的容量。这样：

  • 来自不同输入端口的流不会在同一缓冲区中竞争
  • 使用轮询（RR）调度从不同XB中选择数据包
  • 不同交换机路径的流享受几乎平等的链路带宽

4.5.2 随机丢弃

当交叉点缓冲区满时，CQRD采用随机丢弃方案：

  • 不是简单丢弃尾部数据包（如经典CQ）
  • 如果XB没有足够空间，随机选择一个数据包丢弃
  • 数据量大的流被丢弃的概率更高（因为缓冲中数据包多）
  • 数据量小的流即使在缓冲满时也有合理机会进入缓冲区

4.6 数学分析

4.6.1 流大小与丢弃概率

假设交叉点缓冲区中的数据包总数为 $N$，某个流 $f$ 的数据包数为 $n_f$，则在随机丢弃方案下，流 $f$ 的丢弃概率为：

$$P_{drop}(f) = \frac{n_f}{N}$$

这意味着大流的丢弃概率与其数据包占比成正比。对于一个占据90%缓冲的巨流和占据10%缓冲的小流：

  • 巨流的丢弃概率：90%
  • 小流的丢弃概率：10%

这样大流承担了大部分丢弃，小流得到保护。

4.6.2 与尾丢弃的对比

在传统尾丢弃方案下，所有新到达的数据包都被丢弃：

$$P_{drop}^{tail}(f) = 1, \quad \text{when buffer is full}$$

这意味着即使是小流的新数据包也会被丢弃，造成不公平。

4.6.3 交叉点缓冲大小的设计

令 $N$ 为交换机端口数，$M$ 为总片上内存。CQRD每个交叉点的缓冲大小为：

$$L_{XB} = \frac{M}{N^2}$$

对于24端口交换机，总内存5MB，每个XB约为9KB。

4.6.4 流隔离效果

设有两个流 $f_1$（小流）和 $f_2$（大流）来自不同输入但去往同一输出。在OQ中，两流共享同一队列：

$$P_{conflict}^{OQ}(f_1, f_2) = 1$$

在CQRD中，两流分别在不同的XB中：

$$P_{conflict}^{CQRD}(f_1, f_2) = 0$$

完全消除了它们之间的输出争用。

4.6.5 随机丢弃下的路径争用分析

当两个流 $f_1$（大）和 $f_2$（小）来自相同输入去往相同输出（路径争用），其数据包比例为 $r = n_{f_2}/n_{f_1}$，则：

$$P_{enter}(f_2) = \frac{1}{1 + 1/r} = \frac{r}{1+r}$$

当 $r << 1$（小流远小于大流）时：

$$P_{enter}(f_2) \approx r$$

这与流大小成正比，相对公平。

六、关键算法详解

6.1 CQRD数据包入队算法

算法描述

CQRD的核心是数据包入队算法，包括交叉点缓冲区定位、空间检查、随机丢弃决策等。

伪代码

算法1：CQRD数据包入队

输入：
    - packet：到达的数据包
    - input_port：输入端口号
    - XB[1..N][1..N]：交叉点缓冲区数组
    - L_XB：每个XB的容量（信元数）

输出：
    - success：入队是否成功

function CQRDEnqueue(packet, input_port, XB, L_XB):
    output_port = packet.destination_port
    target_xb = XB[input_port][output_port]

    // 1. 检查空间
    if target_xb.size() < L_XB:
        // 有空间，直接入队
        target_xb.enqueue(packet)
        return True

    // 2. 缓冲区满，执行随机丢弃
    n = target_xb.size()
    random_index = uniform_random(0, n-1)

    // 3. 选中的数据包被丢弃
    dropped_packet = target_xb.remove_at(random_index)

    // 4. 新数据包入队
    target_xb.enqueue(packet)

    // 5. 记录丢弃统计
    update_drop_stats(dropped_packet.flow_id)

    return True  // 新包入队成功

6.2 CQRD调度算法（轮询）

算法描述

CQRD使用轮询（Round-Robin）调度算法，每个输出端口的调度器轮流从其列中的各个XB中选择数据包发送。

伪代码

算法2：CQRD轮询调度

输入：
    - output_port：输出端口号
    - XB[1..N][output_port]：输出端口对应的所有XB
    - last_input[output_port]：上次选中的输入端口

输出：
    - selected_packet：选中的数据包

function CQRDRoundRobinSchedule(output_port, XB, last_input):
    N = number_of_input_ports
    start = (last_input[output_port] % N) + 1

    // 循环扫描N个XB
    for offset from 0 to N-1:
        current_input = ((start + offset - 1) % N) + 1
        target_xb = XB[current_input][output_port]

        if not target_xb.is_empty():
            // 找到非空XB
            selected_packet = target_xb.dequeue()
            last_input[output_port] = current_input
            return selected_packet

    // 所有XB都为空
    return None

// 主调度循环
function MainSchedulingLoop():
    last_input = [0] * (N + 1)

    while True:
        for output_port from 1 to N:
            packet = CQRDRoundRobinSchedule(output_port, XB, last_input)

            if packet is not None:
                transmit_packet(packet, output_port)

6.3 随机丢弃决策的优化实现

算法描述

为了减少随机丢弃的硬件实现复杂度，可以采用以下优化：

  1. 不需要严格的随机选择，可以使用简单的哈希函数
  2. 可以使用指针追踪而非物理移动数据
  3. 可以批量处理多个丢弃决策

伪代码

算法3：优化的随机丢弃

输入：
    - target_xb：目标交叉点缓冲区
    - new_packet：要入队的新数据包

输出：
    - 是否成功入队

function OptimizedRandomDrop(target_xb, new_packet):
    if target_xb.size() < target_xb.capacity:
        // 直接入队
        target_xb.append(new_packet)
        return True

    // 生成伪随机索引（使用LFSR）
    random_index = generate_pseudo_random_index(target_xb.size())

    // 使用指针操作避免数据移动
    target_xb.invalidate(random_index)
    target_xb.append(new_packet)

    // 定期压缩
    if target_xb.invalidated_count() > THRESHOLD:
        target_xb.compact()

    return True

// 硬件友好的伪随机数生成
function generate_pseudo_random_index(n):
    // 使用LFSR (Linear Feedback Shift Register)
    static lfsr = 0xACE1  // 初始值

    feedback = (lfsr & 1) ^ ((lfsr >> 2) & 1) ^
               ((lfsr >> 3) & 1) ^ ((lfsr >> 5) & 1)
    lfsr = (lfsr >> 1) | (feedback << 15)

    return lfsr % n

6.4 多级交换机协调（无需协调）

CQRD的一个重要优点是 不需要交换机间协调。每个交换机独立运行CQRD算法，自动获得以下好处：

  • 本地路径分离：在每个交换机上隔离不同输入
  • 整体路径独立性：流的路径在每个交换机上独立处理
  • 故障隔离：单个交换机的故障不影响其他

十二、局限性与未来工作

12.1 当前局限性

  1. 存储开销：N²个交叉点缓冲对于大端口数交换机存储开销大
  2. 随机丢弃的副作用：随机丢弃可能丢弃重要数据包（如TCP SYN）
  3. 大流吞吐量下降：5-10%的大流吞吐量下降可能影响某些应用
  4. 无优先级支持：不能为高优先级流提供保证
  5. 无截止时间感知：不能区分有截止时间的流

12.2 未来工作方向

12.2.1 算法改进

  • 智能随机丢弃：避免丢弃控制包（SYN、ACK）
  • 优先级支持：为不同流提供不同优先级
  • 截止时间感知：结合D²TCP的截止时间信息
  • 自适应缓冲：根据流量动态调整缓冲大小

12.2.2 架构演进

  • 分层CQRD：将端口分组，减少交叉点数量
  • 共享缓冲CQRD：在不同交叉点间共享部分缓冲
  • 多平面CQRD：使用多个交换平面提升容量

12.2.3 应用扩展

  • SDN集成：与SDN控制器深度集成
  • NFV场景：在网络功能虚拟化场景中的应用
  • 5G核心网：在移动网络中的应用

十三、总结与启示

13.1 主要贡献

  1. 提出了简单有效的CQRD方案，解决数据中心流干扰问题
  2. 通过交叉点队列实现交换机路径分离，为不同输入的流提供隔离
  3. 通过随机丢弃缓解同一路径内的流干扰，保护短流
  4. 实现短流FCT显著降低（单机20-44%，多级8-30%）
  5. 不需要修改主机协议栈，部署简单

13.2 设计哲学启示

CQRD的成功体现了"从根本解决问题"的设计哲学。通过直接解决流干扰的根本原因（粗粒度队列管理），而非通过传输层补救，CQRD实现了简单而有效的解决方案。这启示我们：好的系统设计应该追求简洁性和根本性，而非堆砌复杂机制。

13.3 对网络架构演进的影响

CQRD的工作对数据中心网络架构演进具有深远影响：

  1. 推动了交换机内细粒度队列管理的研究
  2. 展示了交叉点队列在DCN中的可行性和价值
  3. 启发了随机丢弃在网络中的应用
  4. 为简单可部署的DCN方案提供了范例

14.2 关键公式汇总

  1. 随机丢弃概率： $$P_{drop}(f) = \frac{n_f}{N}$$
  2. 每XB大小： $$L_{XB} = \frac{M_{total}}{N^2}$$
  3. 小流入队概率（路径争用情况）： $$P_{enter}(f_{small}) = \frac{r}{1+r}$$ 其中 $r = n_{small} / n_{large}$
  4. 总缓冲内存： $$M_{total} = N^2 \times L_{XB} \times S_{cell}$$
  5. 调度公平性（理论）： $$P_{select}(I_i \to O_j) = \frac{1}{N_{active}(O_j)}$$ 其中 $N_{active}(O_j)$ 是有数据要发往 $O_j$ 的非空XB数
  6. OQ vs CQRD的干扰概率： $$P_{conflict}^{OQ}(f_1, f_2) = 1, \quad P_{conflict}^{CQRD}(f_1, f_2) = 0$$ （当 $f_1, f_2$ 来自不同输入）
  7. 大流吞吐量损失： $$\eta_{loss} = 1 - \frac{Goodput_{CQRD}}{Goodput_{OQ}} \approx 5-10\%$$
  8. 短流FCT减少： $$\Delta_{FCT} = \frac{FCT_{OQ} - FCT_{CQRD}}{FCT_{OQ}} \approx 20-44\%$$

14.6 实现参考

14.6.1 NS-2仿真代码框架

// CQRD队列管理模块
class CQRDQueue : public Queue {
public:
    CQRDQueue(int n_ports, int xb_capacity);
    void enqueue(Packet* p);
    Packet* dequeue();

private:
    int n_ports_;
    int xb_capacity_;

    // N×N 交叉点缓冲数组
    PacketQueue*** xb_;

    // 调度器状态
    int* last_input_;

    // 随机数生成器
    LFSR random_gen_;

    // 入队辅助函数
    bool tryEnqueue(int input, int output, Packet* p);
    void randomDrop(int input, int output, Packet* p);

    // 调度辅助函数
    Packet* scheduleOutput(int output);
};

void CQRDQueue::enqueue(Packet* p) {
    int input = p->input_port();
    int output = p->output_port();

    PacketQueue* xb = xb_[input][output];

    if (xb->length() < xb_capacity_) {
        // 直接入队
        xb->enque(p);
    } else {
        // 随机丢弃
        int n = xb->length();
        int random_idx = random_gen_.next() % n;

        // 删除随机选中的包
        Packet* dropped = xb->remove_at(random_idx);
        drop(dropped);

        // 新包入队
        xb->enque(p);
    }
}

Packet* CQRDQueue::dequeue() {
    // 对每个输出端口进行调度
    for (int output = 0; output < n_ports_; output++) {
        Packet* p = scheduleOutput(output);
        if (p != NULL) {
            return p;
        }
    }
    return NULL;
}

Packet* CQRDQueue::scheduleOutput(int output) {
    int start = (last_input_[output] + 1) % n_ports_;

    for (int i = 0; i < n_ports_; i++) {
        int current = (start + i) % n_ports_;
        PacketQueue* xb = xb_[current][output];

        if (xb->length() > 0) {
            last_input_[output] = current;
            return xb->deque();
        }
    }
    return NULL;
}

14.7 常见问题

Q1：CQRD为什么使用随机丢弃而不是其他策略？

随机丢弃有以下优点：

  • 实现简单，硬件友好
  • 自动按数据包占比公平分配丢弃
  • 不需要复杂的状态维护
  • 有助于小流获得公平的入队机会

Q2：CQRD对TCP性能有何影响？

CQRD对TCP的影响是：

  • 大流：会有更多丢包，但通常仍能保持良好的拥塞控制
  • 小流：丢包率显著降低，FCT显著改善
  • 总体：TCP的自适应性确保系统不会陷入崩溃

Q3：CQRD能否与ECN结合？

是的，CQRD可以与ECN结合：

  • 在缓冲区接近满时，标记ECN而非丢弃
  • 这样大流接收到ECN信号会降速
  • 可以进一步减少丢包，提升整体性能

Q4：CQRD适合什么规模的数据中心？

CQRD适合：

  • 小型到中型数据中心（数千服务器）
  • 大型数据中心需要分层CQRD设计
  • 边缘数据中心和私有云

Q5：CQRD的实施成本如何？

CQRD的成本相对较低：

  • 不需要修改主机，主要成本在交换机硬件
  • FPGA实现可以增量部署
  • ASIC实现需要更长时间
  • 主要成本是新交换机的研发和部署

14.9 与后续工作的关系（详细版）

CQRD是数据中心网络队列管理研究的一个里程碑，启发了许多后续工作：

14.9.1 CQRD的演进

CQRD自2014年LCN发表后，作者团队和其他研究者进行了多项扩展：

  • CQRD-CN15：发表于Computer Networks 2015，进一步分析了CQRD在多种流量模式下的性能
  • 智能丢弃CQRD：结合机器学习预测哪些包应该被丢弃
  • 异构CQRD：不同交叉点使用不同的缓冲大小
  • 多优先级CQRD：支持多个优先级队列

14.9.3 工业界采纳

CQRD的思想被多家网络设备厂商采纳：

  • 部分高端数据中心交换机采用了类似交叉点队列的设计
  • 开源软件交换机（如Open vSwitch）增加了细粒度队列管理选项
  • P4可编程交换机使CQRD的灵活实现成为可能
  • SmartNIC厂商开始集成类似的队列管理功能

14.10 实践建议

对于希望部署CQRD的工程团队，提供以下实践建议：

14.10.1 评估阶段

  2. 性能基线：建立当前系统的性能基线
  3. 瓶颈识别：识别现有系统的性能瓶颈

  3. 对比分析：与传统OQ设备进行A/B对比
  4. 问题排查：及时发现并解决部署中的问题

14.10.3 推广阶段

  1. 分阶段推广：按照网络层级（ToR→Agg→Core）逐步推广
  2. 培训运维：培训运维团队掌握新设备的管理
  3. 文档完善：建立完整的运维文档和故障处置预案
  4. 持续优化：根据实际运行情况调整参数

14.13 数据中心网络流量特征分析

为了更深入理解CQRD的设计动机，本节详细分析了数据中心网络的流量特征。这些特征是CQRD能够有效工作的基础。

14.13.1 流大小分布

数据中心流量呈现典型的"重尾分布"特征：

  • 约90%的流为短流（< 10KB），但它们仅占总字节数的不到10%
  • 约3%的流为巨流（> 1MB），但它们占据总字节数的80%以上
  • 这种分布在搜索、电商、社交等应用中普遍存在

14.13.2 流到达模式

时间段新流到达率突发性
低峰期1000-5000 flows/sec低
正常时段10000-50000 flows/sec中
高峰时段100000+ flows/sec高
突发事件500000+ flows/sec极高

14.13.3 应用特征

不同应用展现不同的流量模式：

  • 搜索查询：大量小流，对延迟极敏感
  • 视频流：少量长流，对吞吐量敏感
  • 数据库复制：大流为主，对带宽利用率敏感
  • 分布式计算（MapReduce）：混合模式，shuffle阶段产生大量并发流

14.14 CQRD扩展性研究

14.14.1 大规模交换机

对于大型交换机（如128端口、256端口），CQRD面临扩展性挑战：

14.14.2 分层CQRD

为了支持大型交换机，提出分层CQRD设计：

  1. 将端口分组为多个虚拟交换机
  2. 每个虚拟交换机内部使用CQRD
  3. 虚拟交换机之间使用VOQ连接
  4. 分组大小通常为16-32个端口

14.14.3 ECMP兼容性

CQRD与等价多路径路由（ECMP）兼容：

  • ECMP在L3层选择路径，与CQRD的L2层调度独立
  • 不同路径上的CQRD独立运行
  • 结合使用可以同时获得负载均衡和流隔离的好处

14.15 研究方法学总结

CQRD研究项目采用了严谨的方法论，确保了研究结果的可靠性和实用价值：

  2. 覆盖单交换机和多级网络场景
