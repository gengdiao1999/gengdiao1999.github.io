二、背景（Background）

Wi-Fi 是当前最主要的 Internet接入方式。802.11ac 等 PHY 标准已经能把物理层峰值速率推到1.3 Gbps，但 MAC 层时延却始终难以压下去；在会议厅、教室、宿舍这类"密集 Wi-Fi"场景下，文献 [1] 测得90 分位 Wi-Fi跳时延能超过25 ms。Web 浏览这种业务，网络层几十毫秒的尾延迟会被上层重传、连接握手等放大到秒级（多至100 倍）。这种"最后一跳"已成为端到端时延的最弱一环。MAC 层时延的来源主要有两类：(a)主机内部排队（IP 层、driver、PHY队列等）；(b)802.11 DCF 在共享信道上造成的"分布式队列"——所有节点 HOL 的帧在"空中"轮流争抢信道。文献 [3] 表明主机排队可以用端侧 QoS绕开，但分布式队列受 DCF约束不可绕过。IEEE802.11e/EDCA试图用更小的 CW 和 IFS 给高优先级帧"插队"，然而一旦密集部署中话音流数量增多，小 CW 会导致严重冲突，性能反而下降；论文 [15] 中的图3 也直接证实了这一点。集中式调度（PCF、CENTAUR、DOMINO 等）虽然能彻底避免冲突，但需要中心控制器与严格时间同步，几乎没有在消费级产品中落地。因此"既不修改 AP / PHY，又不依赖集中控制器、又能在密集场景下显著降低 MAC 层延迟"就成了一个长期未解的难题。

三、目的（Purpose / Problems Solved）

  • 痛点1：分布式队列无法绕过：高优先级流即使在 driver 层享受本地 QoS，也仍要在共享信道排队等待，被最差信道节点"拖死"。
  • 痛点2：802.11e/EDCA不可扩展：高优先级帧共用一个小 CW，节点增多后碰撞概率暴增，时延反而恶化。
  • 痛点3：集中式方案难部署：PCF、Soft-TDMAC、CENTAUR 等都需要中心调度或 PHY 层改动，几乎没有商业 AP 支持。
  • 痛点4：现有拥塞控制方案只盯吞吐量：Idle Sense 等虽能改善 goodput，但对每帧延迟（对实时流体验至关重要）几乎没有作用。
  • 解决方案：用每帧时延作为信道竞争激烈程度的实时信号，分布式地让每个节点估算当前竞争者数量 β，并在 driver 内引入一个 Rate Regulator，把"信道上同时竞争的节点数"从 β 控制到接近1，从而把分布式队列变回主机队列，让延迟敏感流能"插队"。

四、核心原理（Principles）

QAir 的核心思想非常直观：如果存在一个集中调度器，它就可以调节 driver→PHY 的下发速率，从而控制分布式队列的长度。QAir 没有中心调度器，但模拟了这一行为：每个节点根据本地测得的每帧延迟估计 β，再把"每秒允许下发到 PHY层的帧数"控制到只剩 β≈1 的水平。延迟敏感的小流自然能绕过几乎所有排队。

定义 Wi-Fi 一帧的延迟：

t = t_{\text{const}} + t_{\text{cont}} + t_{\text{tr}}

其中 t_{\text{const}} = \text{DIFS} + \text{SIFS} + T_{\text{ack}}，t_{\text{tr}} 是帧本身的发送时间，t_{\text{cont}} 是 DCF退避时间。当有 β 个节点争抢信道时，退避时间随 β线性增长：

t_{\text{max}} \approx t_{\text{const}} + \text{SLOT} \cdot CW_{\text{max}} + \beta \cdot \frac{\text{avgframe}}{\text{avgphyrate}}

也就是说 β越大，每帧最坏延迟越大。QAir 的目标就是让 β ≈1。

为了稳定地得到 β 的代理指标，QAir 采用两层滑动平均：先在每个 checkInterval 时窗内对帧延迟做算术平均得到 T_i，再跨时窗做指数滑动平均得到 \hat{T}：

\hat{T}_{\text{new}} = \alpha \cdot \hat{T}_{\text{old}} + (1-\alpha) \cdot T_i

其中 checkInterval 的选取原则是"至少能发出 CW_min 个包"，在802.11n、1500B 包、100Mbps真实速率下大约1.8 ms。算法据此控制速率：

\hat{T} < T_{\text{target}} \Rightarrow R_{\text{tc}} \leftarrow \min(R_{\text{tc}}+\delta, R_{\text{max}})

\hat{T} \ge T_{\text{target}} \Rightarrow R_{\text{tc}} \leftarrow \max\!\Big(\frac{T_{\text{target}}}{\hat{T}} \cdot R_{\text{tc}},\; R_{\text{min}}\Big)

这里的"加性增、乘性减"借鉴自 TCP/AIMD，区别在于：(1) 上界 R_{\text{max}}防止恢复期抖动下速率飙升；(2) 下界 R_{\text{min}}防止"死锁"（R_{\text{tc}}=0 后永远等不到新的 \hat{T}反馈）；(3) 控制目标 T_{\text{target}} 是用户可配的延迟-吞吐权衡点。

五、算法详解（Algorithm）

输入：每帧延迟 t_k（内核打时间戳），控制目标 T_{\text{target}}。 输出：每秒下发到 PHY层的最大帧速率 R_{\text{tc}}，由 Rate Regulator强制执行。 核心模块：

  • Delay Monitor：在 ath9k驱动中打三段时间戳 $t_w(p)$、$t_h(p)$、$t_e(p)$。其中 $t_h(p)$ 通过前一个包 ACK时刻 $t_e(p-1)$推断：

    • t_h(p) = \max\{t_w(p),\; t_e(p-1)\}

  • Control Algorithm：每收到一次 ACK 调用 F(t_k)，累积 checkInterval窗口内的延迟，窗口结束更新 $\hat{T}$，按 AIMD规则更新 $R_{\text{tc}}$。
  • Rate Regulator：在 ath9k driver 与 PHY 之间插入一个 token bucket，按 $R_{\text{tc}}$速率发放 token。数据包到来时若无 token 则缓存。

伪代码：

def F(tk): #每次 ACK触发
 global sum, ntrans, That
 sum += tk
 ntrans +=1
 if (now - last_time) > checkInterval:
 Ti = sum / ntrans
 That = alpha * That + (1 - alpha) * Ti
 last_time = now
 sum =0; ntrans =0
 if That < Ttarget:
 Rtc = min(Rtc + delta, Rmax)
 else:
 Rtc = max((Ttarget / That) * Rtc, Rmin)
 return Rtc

关键数学：见 §四的方程。 复杂度：每个 ACK 仅一次常数时间更新，复杂度 O(1)；token bucket 占 O(1)内存。 推理流程：ACK→ t_k累积→窗口结束→ 更新 \hat{T}→ 更新 R_{\text{tc}}→ token bucket决定下一个包能否下发到 PHY。

示例：4 个节点 S1–S4 同时向 PHY 各塞1帧。在 DCF 下 β=4，延迟正比于4；QAir 把 S1、S3 在 Rate Regulator 层缓存后 β=2，每个竞争节点的延迟下降到一半；缓存包的额外延迟远小于原信道上叠加的退避延迟。

八、关键创新点（Key Innovations）

  • +分布式队列转主机队列：放弃"如何在共享信道排队"的思路，转而把排队拉回到本地 driver缓存，从而让本地 QoS真正生效。
  • + 自适应 AIMD 控制：借鉴 TCP拥塞控制但针对无线特性定制 $R_{\min}$/$R_{\max}$，既避免"死锁"，也避免剧烈抖动。
  • + 双层滑动平均的延迟估计：直接对原始帧延迟做 EWMA会被极端值带飞（差100 倍），先做时间桶内算术平均再做跨桶 EWMA，稳态估计与瞬时值兼顾。
  • + 三时间戳精确测每帧延迟：在 ath9k 中利用 $t_h(p)=\max(t_w(p),t_e(p-1))$准确推断 PHY队列 HOL时刻，无需修改硬件。
  • + 完全分布式、无需集中调度：单台路由器上修改驱动即可上线，可与现有802.11e 共存，不需时间同步。

九、实验与结果（Experiments）

  • 测试床：20 台 NETGEAR WNDR4300 双频路由器（OpenWrt15.05、560 MHz CPU、128 MB RAM），5 GHz802.11n、20 MHz、固定在同一信道；10 条 AP–client链路。
  • Baseline：原生 IEEE802.11 DCF（即 QAir关闭）。
  • 指标：聚合吞吐量 + ping 平均 RTT。
  • 结果：
  • UDP背景流：在1–4 条并发时 ping RTT下降 50%–75%；9 条并发时下降更明显，吞吐量与 baseline持平。
  • TCP背景流：在1–5 条并发时 ping RTT下降 50%–60%，吞吐量损失 ≤15%。

    • -直播实测：把两部手机面对面做实时时钟直播，100 s 内 baseline出现 7 次缓冲，端到端时延2.15–9.76 s；QAir开启后 0 次缓冲，时延压缩到2.15–2.23 s。

  • 消融：本论文没有逐组件的 ablation，但 $\alpha$、$T_{\text{target}}$ 的取值都对收敛速度与延迟有可观测影响；增大 $T_{\text{target}}$提升吞吐量、减小则进一步压延迟。
  • 效率：CPU几乎无开销；token bucket 在 Linux hrtimer 上以 µs 级精度调度。

##十二、术语表（Glossary）

  • DCF：Distributed Coordination Function，802.11 的基本 CSMA/CA退避机制。
  • CW：Contention Window，竞争窗口，节点在 [0, CW] 内选退避计数。
  • DIFS / SIFS：分布式帧间隔 /短帧间隔，802.11 时序常量。
  • ECMP / EDCA：IEEE802.1ad 多路径等价路由 / IEEE802.11e增强分布式信道访问。
  • AIMD：Additive Increase Multiplicative Decrease，TCP拥塞控制经典规则。
  • PSM：Power Save Mode，节能模式下 beacon/DTIM调度。
  • MSDU / A-MPDU：MAC Service Data Unit /聚合 MAC协议数据单元。
  • ath9k：Atheros802.11n Linux 开源驱动，本文的实现对象。
