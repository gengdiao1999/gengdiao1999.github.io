1.1 核心贡献

TraceAnomaly是首个专门针对微服务轨迹异常检测的深度贝叶斯网络方法，首次实现了时延和调用路径的统一建模，在大规模微服务系统上取得SOTA效果。
  1. 统一建模：首次将轨迹的时延（数值维度）和调用路径（离散维度）统一在深度贝叶斯网络框架下建模
  2. 无监督：无需人工标注，完全从正常Trace中学习模式，可检测未知异常
  3. 可解释：贝叶斯网络的概率推理提供可解释性，能定位异常发生的具体服务
  4. 工业验证：在微众银行的4个大型在线服务和测试床上进行了充分验证，检测精度、召回率都超过97%

2. 研究背景

2.1 微服务架构的普及

近年来，微服务架构在大型Web应用中越来越流行。这种架构将Web服务解耦成许多独立的微服务，每个微服务都可以被单独升级，支持更敏捷的软件开发和部署。

但微服务也给故障排查带来了巨大的挑战：

  • 服务数量众多：单个业务系统可能包含数百个微服务
  • 调用关系复杂：服务间调用链复杂，依赖关系难以梳理
  • 问题定位困难：单个服务故障可能引发连锁反应，根因难以追溯

2.2 分布式追踪的现状

为了监控微服务系统，业界普遍采用了分布式追踪系统（如Google Dapper、OpenTracing、Jaeger、SkyWalking等），其工作原理如图1所示：

graph LR subgraph "外部请求" EXT1[外部服务1] EXT2[外部服务2] end subgraph "RPC框架" IN[入口服务S1] --> A[微服务a] IN --> B[微服务b] A --> C[微服务c] B --> C C --> D[微服务d] D --> E[微服务e] E --> OUT[响应返回] end subgraph "Trace记录" MSG1[消息1: UUID-1, a→b, 时间戳] MSG2[消息2: UUID-1, b→c, 时间戳] MSG3[消息3: UUID-1, c→d, 时间戳] MSG4[消息4: UUID-1, d→e, 时间戳] end EXT1 --> IN EXT2 --> IN

在RPC框架中，每条消息都有一个唯一的UUID，所有具有相同UUID的消息构成一个Trace（调用链）。每条消息记录了：

  • 调用发起方（发送方）
  • 调用接收方（接收方）
  • 发送时间戳
  • 接收时间戳
  • 返回码（成功/失败）

2.3 轨迹异常检测的挑战

给定一个Trace，判断该Trace是否异常，需要同时考虑两个维度：

维度 含义 挑战
时延维度 每个RPC调用的响应时间 不同服务的时延分布差异极大，异常的判断标准差异大
路径维度 调用的拓扑结构、调用顺序 服务数量多（数百个），路径组合爆炸，规则无法覆盖

此外，还要应对以下挑战：

  1. 时延与路径的相互依赖：一个服务的时延异常可能影响后续服务的时延，需要建模这种依赖关系
  2. 标签稀缺：微服务系统中的Trace数量巨大，人工标注几乎不可能
  3. 规模性：系统每天产生百万级Trace，检测算法必须高效
  4. 可解释性：运维人员不仅需要知道"异常"，还需要知道"哪里异常"、"为什么异常"

2.4 现有方法的不足

现有工业方案：基于返回码的规则

当前工业界的主流方案是基于服务调用的返回码设计规则：

  • 如果某个RPC返回码非0（表示失败），则认为该Trace有异常
  • 维护大量的固定阈值规则

但这种方案存在严重问题：

  • 规则维护成本高：服务数从61到344个，规则需要频繁更新
  • 召回率低：大量异常是时延异常，而非返回码异常，如"虽然成功但响应极慢"的情况
  • 规则滞后：新出现的异常类型无法及时检测

现有学术方案

  • 难以端到端学习两者关系，性能不足
    • 方案 思路 局限性
    Multimodal LSTM [3] 用LSTM同时学习时延和调用序列
    AEVB [4] 仅关注时延异常 完全忽略路径维度，无法检测异常路径
    TraceAnomaly（本文） 深度贝叶斯网络，统一建模时延和路径 ✓ 同时处理时延和路径

    3. 论文目的与解决的问题

    3.1 研究目标

    TraceAnomaly的设计目标是：

    1. 统一建模：在同一个框架下同时建模Trace的时延和路径两个维度
    2. 无监督：不需要任何人工标注，直接从正常Trace中学习模式
    3. 高精度：检测精度和召回率必须高于现有方法
    4. 可解释：能定位异常发生的具体服务，辅助根因分析
    5. 高性能：能处理百万级Trace的检测需求

    3.2 要解决的核心问题

    问题1：时延与路径的异质性

    时延是连续数值变量，路径是离散的拓扑结构，如何在同一个模型中统一处理这两类异质数据？

    问题2：服务间的依赖关系

    调用链中前序服务的状态会影响后续服务的状态，如何建模这种依赖传播？

    问题3：无监督学习的挑战

    在没有标签的情况下，如何学习正常模式的分布，并检测偏离正常的Trace？

    3.3 核心思想

    将Trace看作一个随机过程：每个服务的时延是一个随机变量，调用关系对应变量间的依赖，用深度贝叶斯网络建模这个联合分布，通过后验概率检测异常。

    4. 核心原理（含数学公式）

    4.1 Trace的概率建模

    首先，将一条Trace形式化定义为：

    \[ T = \{ (s_i, d_i, p_i) \}_{i=1}^{N} \]

    其中：

    • \(s_i\) 表示第i个调用的服务对（source, destination）
    • \(d_i\) 表示该调用的时延
    • \(p_i\) 表示该调用的父调用（即调用发起方）
    • \(N\) 是该Trace中的调用总数

    整个Trace的联合概率可以分解为条件概率的乘积：

    \[ p(T) = \prod_{i=1}^{N} p(d_i | \text{Parents}(d_i), s_i, \theta) \cdot p(s_i | \text{Parents}(s_i), \phi) \]

    其中：

    • \(\theta\) 是时延分布的参数
    • \(\phi\) 是路径分布的参数
    • \(\text{Parents}(\cdot)\) 表示调用链中的父节点（前序调用）

    4.2 时延的贝叶斯建模

    每个服务的时延服从混合高斯分布，但分布参数由贝叶斯网络的隐状态决定：

    \[ d_i \sim \sum_{k=1}^{K} \alpha_k \cdot \mathcal{N}(\mu_k(h_i), \sigma_k(h_i)^2) \]

    其中：

    • \(K\) 是混合成分数（实验中K=5）
    • \(h_i\) 是第i个服务的隐状态向量（从深度网络得到）
    • \(\mu_k(h_i)\) 是第k个成分的均值，由隐状态通过神经网络计算得到
    • \(\sigma_k(h_i)\) 是第k个成分的标准差，由隐状态通过神经网络计算得到
    • \(\alpha_k\) 是混合权重，满足 \(\sum_{k=1}^{K} \alpha_k = 1\)

    隐状态的更新通过后验流网络实现：

    \[ h_i = \text{RNN}(h_{\text{parent}(i)}, d_{\text{parent}(i)}, s_i) \]

    这里使用循环神经网络捕捉调用链上的依赖传播。

    4.3 路径的贝叶斯建模

    路径建模的核心是服务对的条件概率。给定父调用的服务对，当前调用的服务对服从多项式分布：

    \[ s_i \sim \text{Categorical}(\pi(h_i)) \]

    其中 \(\pi(h_i)\) 是各服务对的概率，由隐状态 \(h_i\) 通过softmax层计算得到：

    \[ \pi_k(h_i) = \frac{\exp(w_k^T h_i + b_k)}{\sum_{j=1}^{M} \exp(w_j^T h_i + b_j)}, \quad k = 1,...,M \]

    M是所有可能服务对的数量（由系统中的服务数决定）。

    4.4 深度贝叶斯网络的结构

    深度贝叶斯网络的推断过程可以概括为：

    1. 前向传播：从根节点开始，按照调用链的拓扑顺序，依次计算每个节点的隐状态、时延分布参数、路径分布参数
    2. 似然计算：对实际观测到的时延和服务对，计算其对数似然
    3. 反向传播：最小化负对数似然，更新网络参数

    整个网络的训练目标是最大化所有正常Trace的对数似然：

    \[ \max_{\theta, \phi} \sum_{T \in \mathcal{T}_{\text{normal}}} \log p(T | \theta, \phi) \]

    其中 \(\mathcal{T}_{\text{normal}}\) 是训练集中的正常Trace集合。

    4.5 异常得分计算

    对于一条待检测的新Trace T，异常得分定义为其对数似然的负值：

    \[ \text{Score}(T) = -\log p(T) = -\sum_{i=1}^{N} \left[ \log p(d_i | \text{Parents}(d_i), s_i) + \log p(s_i | \text{Parents}(s_i)) \right] \]

    异常得分越高，表示该Trace越偏离正常模式，越可能是异常。

    当异常得分超过某个阈值 \(\tau\) 时，判定该Trace为异常：

    \[ \text{IsAnomaly}(T) = \begin{cases} \text{True} & \text{Score}(T) > \tau \\ \text{False} & \text{otherwise} \end{cases} \]

    阈值 \(\tau\) 可以通过正常Trace的得分分布的分位数确定（如99%分位数）。

    4.6 异常定位机制

    TraceAnomaly不仅能检测整个Trace是否异常，还能精确定位到异常发生的具体调用。第i个调用的异常得分是：

    \[ \text{LocalScore}(i) = - \left[ \log p(d_i | \cdot) + \log p(s_i | \cdot) \right] \]

    对异常Trace，LocalScore最高的几个调用就是最可能的异常发生点。这为运维人员提供了直接的根因线索。

    4.7 后验流网络

    论文提出了后验流（Posterior Flow）的概念，用于高效推断隐变量的后验分布：

    \[ q(h | T) = \prod_{i=1}^{N} q(h_i | h_{\text{parent}(i)}, d_i, s_i) \]

    每个条件后验由一个归一化流（Normalizing Flow）网络建模，实现精确而非近似的贝叶斯推断。

    6.1 Trace标准化与预处理

    伪代码

    算法1: Trace预处理与标准化
输入: 原始Trace集合 RawTraces
输出: 标准化的Trace集合 NormalizedTraces

1: // 步骤1: Trace结构解析
2: for raw_trace in RawTraces:
3:     // 按UUID分组
4:     messages = group_by_uuid(raw_trace)
5:     // 按时间戳排序
6:     messages = sort_by_timestamp(messages)
7:
8:     // 重建调用树结构
9:     root = build_call_tree(messages)
10:    // 层序遍历，得到调用序列
11:    call_sequence = bfs_traversal(root)
12:
13:    // 步骤2: 服务对编码
14:    for call in call_sequence:
15:        service_pair = (call.source, call.destination)
16:        if service_pair not in service_pair_vocab:
17:            service_pair_vocab.append(service_pair)
18:        call.service_pair_id = index_of(service_pair_vocab, service_pair)
19:
20:    // 步骤3: 时延标准化
21:    for call in call_sequence:
22:        // 对数变换处理长尾分布
23:        call.log_delay = log(call.delay + 1e-8)
24:        // Z-score标准化（按服务对）
25:        mean_delay = mean_of_service_pair(service_pair)
26:        std_delay = std_of_service_pair(service_pair)
27:        call.normalized_delay = (call.log_delay - mean_delay) / max(std_delay, 1e-6)
28:
29:    // 步骤4: 去除异常短的Trace
30:    if len(call_sequence) >= min_calls:
31:        NormalizedTraces.append(call_sequence)

32: return NormalizedTraces

    伪代码

    算法2: 深度贝叶斯网络训练
输入: 标准化的正常Trace集合 Trainset

      训练轮数 epochs = 200

输出: 训练好的贝叶斯网络模型 model

1: // 步骤1: 模型初始化
2: model = DeepBayesianNetwork(
    service_pair_vocab_size = M,
    hidden_dim = 256,
    num_mixture = 5,
    rnn_type = 'GRU'
)

3: optimizer = Adam(model.parameters(), lr=1e-3)
4: scheduler = ReduceLROnPlateau(optimizer, patience=10)

5: // 步骤2: 训练循环
6: best_val_loss = +∞
7: for epoch in 1 to epochs:
8:     total_loss = 0.0
9:
10:    for batch in DataLoader(Trainset, batch_size, shuffle=True):
11:        optimizer.zero_grad()
12:        batch_loss = 0.0
13:
14:        for trace in batch:
15:            // 按拓扑顺序遍历
16:            h = [zero_initial_state()]
17:            trace_log_likelihood = 0.0
18:
19:            for i, call in enumerate(trace):
20:                // 获取父节点信息
21:                parent_idx = call.parent_index
22:                h_parent = h[parent_idx]
23:                d_parent = trace[parent_idx].normalized_delay
24:                s_parent = trace[parent_idx].service_pair_id
25:
26:                // 后验流更新隐状态
27:                h_i = model.posterior_flow(h_parent, d_parent, call.service_pair_id)
28:                h.append(h_i)
29:
30:                // 计算时延的对数似然
31:                mu, sigma, alpha = model.delay_head(h_i)
32:                log_p_d = log_gaussian_mixture(
33:                    call.normalized_delay, mu, sigma, alpha
34:                )
35:
36:                // 计算路径的对数似然
37:                pi = model.path_head(h_i)
38:                log_p_s = log_categorical(call.service_pair_id, pi)
39:
40:                // 累加该调用的似然
41:                trace_log_likelihood += log_p_d + log_p_s
42:
43:            // 负对数似然作为损失
44:            batch_loss += - trace_log_likelihood / len(trace)
45:
46:        // 反向传播
47:        batch_loss.backward()
48:        // 梯度裁剪防止爆炸
49:        torch.nn.utils.clip_grad_norm_(model.parameters(), 5.0)
50:        optimizer.step()
51:
52:        total_loss += batch_loss.item()
53:
54:    // 验证阶段
55:    val_loss = evaluate(model, Valset)
56:    scheduler.step(val_loss)
57:
58:    // 早停机制
59:    if val_loss < best_val_loss:
60:        best_val_loss = val_loss
61:        save_model_checkpoint(model)
62:    else if no_improve > 20:
63:        break  // 早停

64: // 步骤3: 确定异常阈值
65: all_scores = compute_anomaly_scores(model, Valset)
66: anomaly_threshold = percentile(all_scores, 99)  // 99%分位数

67: return model, anomaly_threshold

    6.3 在线检测算法

    算法3: Trace在线异常检测
输入: 待检测Trace test_trace
      训练好的模型 model
      异常阈值 τ
输出: 是否异常 is_anomaly, 异常定位 anomaly_positions

1: // 步骤1: 预处理标准化
2: normalized_trace = preprocess_single_trace(test_trace)

3: // 步骤2: 前向传播计算似然
4: h = [zero_initial_state()]
5: total_score = 0.0
6: call_scores = []

7: for i, call in enumerate(normalized_trace):
8:     parent_idx = call.parent_index
9:     h_parent = h[parent_idx]
10:    d_parent = normalized_trace[parent_idx].normalized_delay
11:
12:    h_i = model.posterior_flow(h_parent, d_parent, call.service_pair_id)
13:    h.append(h_i)
14:
15:    // 计算该调用的异常得分
16:    mu, sigma, alpha = model.delay_head(h_i)
17:    log_p_d = log_gaussian_mixture(call.normalized_delay, mu, sigma, alpha)
18:
19:    pi = model.path_head(h_i)
20:    log_p_s = log_categorical(call.service_pair_id, pi)
21:
22:    call_score = -(log_p_d + log_p_s)
23:    call_scores.append(call_score)
24:    total_score += call_score

25: // 步骤3: 异常判定
26: avg_score = total_score / len(normalized_trace)
27: is_anomaly = (avg_score > τ)

28: // 步骤4: 异常定位（仅对异常Trace）
29: anomaly_positions = []
30: if is_anomaly:
31:     // 找出得分最高的前k个调用
32:     top_k_indices = argsort(call_scores)[-3:]
33:     for idx in top_k_indices:
34:         anomaly_positions.append({
35:             'call': normalized_trace[idx],
36:             'score': call_scores[idx],
37:             'rank': idx
38:         })

39: return is_anomaly, anomaly_positions

    6.4 复杂度分析

    阶段 单Trace时间复杂度 说明
    训练（单步） \(O(N \cdot D \cdot H)\) N是Trace长度，D是隐状态维度，H是RNN隐层单元
    检测 \(O(N \cdot D \cdot H)\) 仅一次前向传播
    实测检测速度 单Trace平均检测时间约0.3ms，单GPU每秒可处理约3000条Trace

    12. 局限性与未来工作

    12.1 当前局限性

    1. 仅考虑调用树结构：当前模型假设Trace是树结构，但真实Trace可能有更复杂的DAG结构（如并行分支）
    2. 未考虑Trace长度异常：长度异常（过短或过长）也是一种重要异常，但当前模型未专门处理
    3. 对新服务冷启动：新上线的服务缺乏历史数据，模型需要时间积累
    4. 异常类型分类：当前仅区分正常/异常，无法具体分类异常类型（如超时、重试、限流等）
    5. 多Trace关联分析：当前孤立分析每条Trace，未考虑Trace之间的关联（如同一批次请求的集体异常）

    12.2 未来工作方向

    方向1：图神经网络（GNN）增强

    用GNN替代当前的RNN，更好地建模任意拓扑结构的Trace：

    • 支持并行分支的DAG结构
    • 更好地捕捉跨分支的依赖关系
    • 利用图注意力机制自动学习重要的调用路径

    方向2：异常类型细粒度分类

    从二分类（正常/异常）扩展到多分类：

    • 时延异常（突发/持续/渐变）
    • 路径异常（不该调用的/缺少调用/循环调用）
    • 返回码异常
    • 长度异常

    方向3：跨Trace的异常模式

    学习多条Trace之间的关联模式：

    • 同一时段多条Trace集体异常
    • 异常传播模式学习
    • 基于Trace异常的故障预测

    方向4：统一的可观测性模型

    将Trace、Metric、Log三者在同一个模型中联合建模：

    • 多模态异常检测
    • 跨模态根因推断
    • 统一的可观测性表示学习

    权重衰减 \(\lambda\) 1e-5 L2正则化系数
    训练轮数 \(E\) 200 带早停
    异常阈值分位数 \(q\) 99% 正常Trace得分的99%分位数
    Trace最小长度 \(L_{min}\) 3 过滤过短的Trace
    训练数据窗口 \(W_{train}\) 7天 训练数据的时间跨度

    14.3 关键公式速查

    公式 用途
    \(p(T) = \prod_{i=1}^{N} p(d_i | \cdot) p(s_i | \cdot)\) Trace联合概率分解
    \(d_i \sim \sum_{k=1}^{K} \alpha_k \mathcal{N}(\mu_k(h_i), \sigma_k(h_i)^2)\) 时延混合高斯分布
    \(s_i \sim \text{Categorical}(\pi(h_i))\) 服务对类别分布
    \(h_i = \text{RNN}(h_{\text{parent}(i)}, d_{\text{parent}(i)}, s_i)\) 后验流隐状态更新
    \(\text{Score}(T) = -\log p(T)\) Trace异常得分
    \(\text{LocalScore}(i) = -[\log p(d_i|\cdot) + \log p(s_i|\cdot)]\) 单调用异常定位得分
