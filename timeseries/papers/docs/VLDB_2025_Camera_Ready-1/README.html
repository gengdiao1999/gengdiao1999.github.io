二、背景

时序数据广泛存在于电力、医疗、流量、气象、金融以及工业 AIOps 场景。运维人员经常需要基于多变量监控时序回答诸如"成功率是否有异常抖动""哪些指标同步出现了波动""系统发生了什么故障"这类开放性问题。传统时序异常检测模型只能给出"是否有异常"的二值或分数结论，无法解释，且缺乏可对话性。

近年来 MLLM（多模态大模型）在视觉-语言任务上取得突破，但时序模态一直缺失。现有三类方法各有缺陷：(1) Text-based 把数值塞进 prompt，受 token 长度限制，全局特征理解差；(2) Vision-based 把时序画成图像交给 GPT-4o/QWen-VL，受图像分辨率和细节解析能力限制；(3) Agent-based 让 LLM 调用 STL 分解、AD/Rocket 等工具，token 消耗大、易产生幻觉。

此外，要让 LLM 真正"理解"时序，必须有大量"时序-文本"对齐数据，而现有公开语料几乎不存在。TimeLLM 等工作仅聚焦于预测任务，并未解决"理解"与"推理"问题。论文指出，工业 AIOps 场景下，多变量时序之间的相关性是诊断故障的关键，因此需要把多变量时序作为一等模态，类比图像让 LLM 能端到端读图。

三、目的

  • 痛点 1：缺乏高质量"语言+时序"对齐数据 → 方案：基于属性的合成时序生成方法。
  • 痛点 2：现有方法无法保证对齐精度和属性多样性 → 构建包含 4 类 Trend、7 类 Seasonality、3 类 Noise、19 类 Local Fluctuation 的属性集合。
  • 痛点 3：缺乏多样化 QA 训练数据 → TSEvol 演化算法。
  • 痛点 4：无法保留多变量上下文与数值精度 → Context-Aware TS-MLLM 编码与 Value-Preserved 双通道归一化。
  • 痛点 5：缺乏面向 TS-MLLM 的评测基准 → 构建 A/B/MCQ2 三套数据集。

四、核心原理

ChatTS 由四部分组成：属性驱动的合成时序生成器、TSEvol QA 演化器、上下文感知的 TS-MLLM、两阶段训练。整体思路是"先合成、再训、最后用真实数据评估"。模型输入包括多变量时序数组与自然语言查询，输出自然语言答案。

关键概念：

  • Attribute Set：4 大类（Trend/Periodicity/Noise/Local Fluctuation）共 33 种细粒度属性，按规则拼装可生成无限多样时序。
  • Attribute Selector：由 GPT 根据真实业务场景选取属性子集，使生成时序具备物理意义。
  • Time Series Patches：每个时序列切成固定大小 patch，由 5 层 MLP 编码。
  • Context-Aware Encoder：保持多变量时序在原始输入中的相对位置，将 patch 嵌入到文本 token 之间。
  • Value-Preserved Normalization：除 0-1 归一化外，将 scaling 和 offset 参数以文本形式附加到 prompt，使 LLM 可还原数值。
  • TSEvol：从种子 QA 通过七个方向逐步演化。

对数值任务定义相对准确率：

$$\text{relative\_accuracy} = \max\left(1.0 - \frac{|V_{\text{answer}} - V_{\text{label}}|}{|V_{\text{label}}|},\, 0.0\right)$$

五、算法详解

  1. 输入 / 输出：多变量时序 $T$ 与自然语言查询 $Q$，输出文本答案 $A$。
  2. 核心模块：Attribute Selector、Attribute Sampler、Time Series Generator、TSEvol、Context-Aware Encoder、Value-Preserved Normalization。
  3. 伪代码： 
    # 1) 属性合成
attrs = GPT_Selector.select(metric_set, all_attribute_set)
sample = AttributeSampler.sample(attrs)
ts = TimeSeriesGenerator.generate(sample)

# 2) QA 演化
qa_pairs = []
for qa_seed in seed_qas:
    new_qa = TSEvol.evolve(qa_seed, attribute_pool,
                            types=["Breadth","Depth","Reasoning","Situation"])
    qa_pairs.append(new_qa)

# 3) 上下文编码
patches = mlp_encoder(patchify(ts_arrays))
text_tokens = text_embedding(tokenize(prompt))
x = insert_patches_at_position(patches, text_tokens)

# 4) 推理
answer = qwen25_14b.generate(x)
  4. 关键数学：相对准确率（见上式）。
  5. 复杂度：单 patch 编码 $O(p \cdot d)$，推理 token 量比 text-based 方法少 5× 以上。
  6. 训练与推理：基模型为 QWen2.5-14B-Instruct，8×A800 GPU，DeepSpeed + LLaMA-Factory。
  7. 示例：ChatTS 能正确识别"哪些指标同步波动"，相比 GPT-4o 多变量对齐任务 F1 提升 46.0%，推理任务提升 25.8%。

八、关键创新点

  • 属性驱动的合成时序生成：数值与属性文本精确对齐。
  • TSEvol：首个面向时序的多模态 Evol-Instruct 算法。
  • Context-Aware TS-MLLM：token 级别保留多变量相对位置。
  • Value-Preserved 双通道归一化：0-1 缩放 + scaling/offset 文本参数。
  • 三套真实-合成混合评测：覆盖 6 类对齐与 4 类推理子任务。

九、实验与结果

  • 数据集：Dataset A (525 题)、Dataset B (1,616 题)、MCQ2 (100 题)；训练用约 17 万合成样本。
  • Baseline：Text-based（GPT-4o、GPT-4-Turbo、QWen2.5-14B）、Vision-based（GPT-4o/mini）、Agent-based（ReAct + STL/AD/Rocket）。
  • 主要指标：F1、相对准确率、RAGAS、accuracy。
  • 关键数字：
    • 对齐任务平均：ChatTS 在 Dataset A 上 categorical/numerical 比 GPT-4o 提升 46.0% 和 80.7%；Dataset B 提升 75.9% 与 112.7%。
    • 推理任务平均 0.667，远超 GPT-4o (0.530)。
    • 多变量任务相关性 F1 达 0.905。
    • Token 成本仅 0.08M-0.34M，是 GPT-4o 文本基线 1.3M-4.5M 的约 1/15。
  • 消融：w/o Attribute-Based、w/o TSEvol、w/o TS Modality 均显著下降；多变量任务 text-only 几乎完全失效。
  • 效率：每 1.3M token GPT-4o 需 $3.25，ChatTS 仅 $0.02。

十二、术语表

  • Attribute：时序的可量化文本属性。
  • TSEvol：Time Series Evol-Instruct。
  • Context-Aware Encoder：把时序 patch 按文本位置嵌入的编码器。
  • Value-Preserved Normalization：0-1 归一化 + scaling/offset 文本形式。
  • Alignment Task / Reasoning Task：分类-数值对齐任务与归纳-演绎-因果-对比推理任务。
