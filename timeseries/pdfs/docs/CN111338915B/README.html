二、背景（Background）

大型在线服务系统由很多组件组成来支持大量并发用户。为保证服务质量和用户体验，需要从各个组件中收集各种监控数据（指标、日志、调用链等），并手动设置多种告警规则。一旦监控数据违反告警规则（CPU 利用率超过 80%、日志文件出现 "fail" 关键字等）就会生成告警，并发送给工程师检查。如果告警比较严重，工程师会创建工单来进行排查和诊断。告警数据可以包含多个属性，例如告警时间、告警内容、告警类型、告警来源系统、告警来源机器、告警级别、告警关闭时间等。

由于在线服务的复杂性和动态性，系统可能并发生成大量告警，超出工程师的处理能力。因此，实际中常采用手动方式定义定级规则，将告警分为不同优先级（P1-error、P2-warning、P3-info；CPU 利用率超过 90% 是 P1，超过 70% 是 P2）。工程师主要关注最高级别的告警，即严重告警。然而即便如此，严重告警的触发量仍然很大。此外：

  • 手工定义和维护规则很难有统一标准，切换非常消耗人力；
  • 基于规则的告警定级方法准确性不高，难以反映告警的真实严重程度；
  • 无法适应系统的动态变化——大促期间、版本迭代后，原有的"固定阈值 + 固定级别"映射就会失效。

三、目的（Purpose / Problems Solved）

  • 用机器学习替代固定规则：把"动态告警定级"问题建模为基于机器学习的排序问题，而不是阈值匹配。
  • 充分利用历史数据：从历史的工单、告警、指标数据中自动提取"严重性标注"和"特征"，训练排序模型。
  • 数据融合 + 特征融合：同时利用告警数据和指标数据，从多角度刻画告警的严重性。
  • 在线自适应：模型在线推理，给实时到达的并发大量告警一个严重性定级（一个连续分数），引导工程师优先处理严重告警。
  • 可解释的特征：特征都有物理含义（文本主题、文本熵、时序频率、周期、KPI 异常分数等），便于排障时理解。

四、核心原理（Principles）

4.1 系统总览

整个方案分为两个阶段：

  1. 离线训练阶段：从历史工单、告警、指标数据中提取严重性标注和特征向量，训练一个排序模型；
  2. 在线排序阶段：对实时到达的告警数据，提取同样的特征，输入训练好的模型，得到每条告警的严重性定级。

4.2 关键概念

  • 告警定级（Alert Severity Ranking）：本发明用机器学习模型给每条告警输出一个连续的严重性分数，分数越高越严重。
  • BTM（Biterm Topic Model）双词主题模型：一种针对短文本的主题模型，比 LDA 更适合告警这种短文本。
  • IDF（Inverse Document Frequency）逆文档频率：衡量单词稀有度的统计量。
  • ACF（Auto Correlation Function）自相关函数：衡量时间序列周期性的统计量。
  • LSTM（Long Short Term Memory）长短期记忆网络：一种能捕获时序长期依赖的循环神经网络。
  • 工单标注：从工单处置记录中提取的严重性分数，0 = 无、0.1 = 白名单、0.2 = 自动恢复、0.4 = 联系应用管理员确认业务影响、0.6 = 已知原因已修复、0.8 = 联系应用管理员已修复、1.0 = 创建事件单持续跟进。

4.3 数学原理

4.3.1 严重性打标（基于 TF-IDF + k-means 聚类）

对工单的告警处置记录用 TF-IDF 表示，用 k-means 聚类得到若干簇，每个簇对应一种处置情况（白名单、自动恢复、联系管理员、创建工单……），给每个簇打一个严重性分数，作为该簇内所有告警的标注。

4.3.2 主题特征（BTM）

对告警内容用 BTM 提取主题特征。给定主题个数 n，BTM 能发现隐藏的主题和每个主题对应的关键字。对于一条告警，BTM 给出其属于每个主题的概率，长度为 n 的向量即主题特征。

实际 BTM 学习到的主题示例（n=14）：

  • T#1：oracle, 连接, 数据库, 空间, pool, 进程, 锁, ...
  • T#2：syslog, 告警, error, 堆栈, 记录, 硬件, 告警, ...
  • T#3：监控, 环境, 主机, 电池, 湿度, 机房, 电压, ...
  • ...
  • T#13：系统, 交易量, 响应, 阈值, 时间, 业务, 值为, ...
  • T#14：交换机, 虚拟, 通信, 连接, 响应, ping, 网络, ...

对于告警"oracle表空间使用率达到78%，超过阈值"，主题特征为 [0.78, 0.05, 0.02, ..., 0.19, 0.04]（长度 14）。

4.3.3 文本熵（IDF 权重）

对于单词 ω，单词熵为：

$$ \text{IDF}(\omega) = \log \frac{N}{N_\omega} $$

其中 N 是所有告警的个数，Nω 是包含单词 ω 的告警的个数。一条告警的文本熵由其包含的所有单词熵的平均值计算得到。单词熵越高越稀有，越可能出现在严重告警中。

4.3.4 时序特征

  • 频率：告警在历史上发生的频率越高，异常分数越低；越罕见越严重。
  • 周期：用 ACF 检测告警的周期性。

对于一个告警 a，时间序列 C(a) = {c₁(a), c₂(a), …, cₖ(a)}，自相关函数：

$$ \text{ACF}(l) = \frac{\sum_{i=1}^{N-l} (x_i - \bar{x})(x_{i+l} - \bar{x})}{\sum_{i=1}^{N} (x_i - \bar{x})^2} $$

如果时间序列是周期的且周期为 T，那么 ACF(l) 会在 T, 2T, 3T… 处表现出尖峰。用 ACF(l) 的最大值来表征告警的周期性，周期性越强，告警越可能是冗余的（夜里批处理那种），严重性越低。

  • 告警数量：短时间内大量告警 → 可能严重故障。
  • 间隔时间：长时间无告警后突然出现一个告警 → 可能严重。

4.3.5 指标特征（LSTM 多变量异常检测）

用基于 LSTM 的多时间序列异常检测算法捕获每个关键业务指标和机器指标对应的异常分数、整体业务系统异常分数、整体机器系统异常分数，作为指标特征。

4.3.6 排序模型

把告警的"严重性分数"作为标签，特征向量作为输入，用监督学习排序模型（如 LambdaMART、GBDT）训练，推理时输出在线告警的严重性定级。

五、算法详解（Algorithm）

5.1 输入 / 输出

  • 离线输入：历史工单、历史告警、历史指标
  • 离线输出：训练好的排序模型
  • 在线输入：实时告警 + 实时指标
  • 在线输出：每条告警的严重性定级（连续分数）

5.2 伪代码

def offline_train(workorders, hist_alerts, hist_metrics):
    # === 阶段 1: 工单标注 ===
    # 对工单处置记录做 TF-IDF + k-means 聚类
    wo_vectors = tfidf(workorders.resolution_text)
    clusters = kmeans(wo_vectors, k=7)
    severity_labels = {  # 7 个处置类别对应 7 个分数
        "None": 0.0, "白名单": 0.1, "自动恢复": 0.2,
        "联系管理员": 0.4, "已知原因已修复": 0.6,
        "联系管理员已修复": 0.8, "创建工单": 1.0
    }
    for alert in hist_alerts:
        if alert.workorder:
            alert.label = severity_labels[cluster_of(alert.workorder)]

    # === 阶段 2: 特征提取 ===
    # 告警特征
    btm = BitermTopicModel(n_topics=14)
    btm.fit([preprocess(a.text) for a in hist_alerts])  # 训练 BTM
    for a in hist_alerts:
        topic_feat = btm.transform(a.text)              # 主题特征
        text_entropy = mean([idf(w) for w in a.words])   # 文本熵
        ts_feat = [
            history_freq(a),                            # 频率
            max_acf(history_series(a)),                 # 周期性
            alert_count_in_window(a),                   # 数量
            interval_to_prev(a)                         # 间隔
        ]
        rule_severity = a.severity                       # 告警自带
        a.alert_features = concat([topic_feat,
                                    [text_entropy],
                                    ts_feat,
                                    [rule_severity, ...]])

    # 指标特征 (LSTM 多变量异常检测)
    biz_anom = lstm_multivariate_error(hist_metrics, "biz_kpis")
    srv_anom = lstm_multivariate_error(hist_metrics, "server_kpis")
    a.metric_features = [biz_anom[a.time], srv_anom[a.time]]

    # 组合
    a.features = concat([a.alert_features, a.metric_features])

    # === 阶段 3: 训练排序模型 ===
    ranker = LambdaMART()  # 也可以用 GBDT / XGBoost
    ranker.fit([(a.features, a.label) for a in hist_alerts])
    return ranker


def online_rank(ranker, online_alerts, online_metrics):
    biz_anom = lstm_multivariate_error(online_metrics, "biz_kpis")
    srv_anom = lstm_multivariate_error(online_metrics, "server_kpis")
    scored = []
    for a in online_alerts:
        f = extract_features(a, btm, biz_anom, srv_anom)
        score = ranker.predict(f)
        scored.append((a, score))
    return sorted(scored, key=lambda x: -x[1])

5.3 复杂度分析

  • 离线：BTM 训练 O(N × n_topics × iter)；LSTM 异常检测 O(N × T × d²)。
  • 在线：每条告警的特征提取 O(L) + 模型推理 O(d × log(nodes))。

八、关键创新点（Key Innovations）

  • + 首创动态告警定级 = 机器学习排序问题：用有监督的排序模型替代手工阈值规则，让定级结果可以自适应。
  • + 工单自动打标：从历史工单聚类中自动给每条历史告警打严重性分数，避免人工标注的巨大成本。
  • + BTM 短文本主题模型：相比 LDA，BTM 适合告警这种短文本，提取的主题更具可解释性。
  • + IDF 文本熵：用"稀有词更可能严重"的直觉来设计文本特征，简单有效。
  • + ACF 表征告警周期性：把"周期性告警 = 冗余告警"的领域知识形式化为一个可计算的 ACF 特征。
  • + LSTM 多变量异常分数作为指标特征：把 KPI 的多变量异常检测结果作为告警定级的高级特征。
  • + 数据融合 + 特征融合：同时利用告警数据（文本 + 时序）和指标数据（LSTM 异常分数），从多角度评估严重性。

十、应用场景（Use Cases）

  1. 金融交易系统告警定级：在支付链路报警时，自动识别哪些告警是"真的严重"（如数据库连接池耗尽）而哪些是"重复冗余"（如每分钟的常规 GC 告警）。
  2. 云原生微服务运维：K8s 上百个 Pod 同时告警时，自动把"Pod 失败 → 流量切换 → 更多 Pod 失败"这一级联中的关键告警排在前面。
  3. 电商大促保障：大促期间告警量级暴增，本发明按严重性排序让 SRE 把精力集中在最关键的告警上。
  4. 银行核心系统：银行日终批处理期间大量周期性告警，定级模型能识别"周期性 = 低严重"避免打扰工程师。
  5. 告警风暴：与 CN111309565B 联动，定级后只把最严重的告警推送给工程师。

十二、术语表（Glossary）

  • 告警定级（Alert Severity Grading/Ranking）：给告警一个连续的严重性分数。
  • BTM（Biterm Topic Model）：双词主题模型，针对短文本设计。
  • LDA（Latent Dirichlet Allocation）：传统主题模型，对短文本效果不佳。
  • IDF（Inverse Document Frequency）：逆文档频率，衡量单词稀有度。
  • ACF（Auto Correlation Function）：自相关函数，衡量时序周期性。
  • LSTM（Long Short Term Memory）：长短期记忆网络，一种 RNN 变体。
  • LambdaMART：一种基于 GBDT 的监督学习排序模型。
  • 工单（Workorder / Ticket）：告警处理记录，本发明用其作为严重性标注来源。
