2. 研究背景

云系统（AWS、Azure、阿里云）提供计算、存储、电商等多种服务。故障会严重影响稳定性、可用性，导致客户流失、收入下降。批服务器宕机（Batch Servers Outage）是云基础设施最致命的故障之一，会让所有上游服务完全不可用。

真实案例：2022-12-18 阿里云香港数据中心批服务器宕机，导致 ECS、PolarDB、Networking 等服务中断 12 小时以上。

云基础设施包含多个域：

  • 互联网数据中心 (IDC)：服务器机柜、UPS、温度湿度控制。
  • 云网络 (Cloud Networking)：交换机、负载均衡器。
  • 云服务器 (Cloud Servers)：CPU/GPU/SSD 集群。

现有方法的问题：

  1. 微服务 vs 云基础设施：基于 metric/log/trace 的方法不适用于云基础设施（数据格式差异大）。
  2. 故障关联测量不准确：图方法对层级/冗余结构失效。
  3. 缺乏可解释性：仅定位根因，未给出故障传播路径。
论文对阿里云基础设施的实证研究揭示：单源粗粒度数据不足以全面刻画故障，且故障通常由跨域并发故障引起。必须从全局视角建模故障相关性。

3.论文目的与解决的问题

本文提出 BSODiag，目标是无监督、轻量的批服务器宕机诊断框架，从全局视角建模多源监测数据中的故障相关性。

核心挑战：

  1. 低质量故障监测数据（粗粒度）。
  2. 故障相关性复杂（跨域 + 跨层级）。
  3. 根因定位缺乏可解释性（仅有 root cause，不含 propagation path）。

核心贡献：

  1. 首次系统研究大规模云基础设施的批服务器宕机问题，并进行真实系统实证。
  2. 无监督、轻量级诊断框架 BSODiag，含 3 个核心模块：多源故障检测、故障相关性挖掘、宕机根因分析。
  3. 真实工业数据验证：阿里云基础设施 1 年数据，BSODiag 在 PR@3 上 87.5%、PCR 46.3%，显著优于基线。

4.核心原理

4.1 问题形式化

故障快照（Outage Snapshot）：

\[ U \equiv \{A_{-L:T'}, L_{-T:T'}, C_{-T:T'}\} \]

其中 \(A\) 是 alerts，\(L\) 是 incidents，\(C\) 是 changes；\(-L, T, T'\) 分别为前/后时间窗。

两个子问题：

  1. 故障检测子问题：\(F: U \mapsto E = \{e_1, ..., e_n\}\) 识别 outage-related events。
  2. 根因分析子问题：\(M: E \mapsto \{e_r, p_U\}\) 定位 top-k 根因并推断传播路径。

4.2 多源故障检测（Multi-source Failure Detection, MFD）

三个组件：

  1. Alert2TS：将 alerts 转换为多变量时间序列。Alert 子集 \(A_s = \{a_{s,t}^{k,t} : k \in \mathcal{K}, t \in [-L, T']\}\)，每条 alert 序列在长度为 \(\delta\) 的时隙内聚合成 \(X_s \in \mathbb{R}^{K \times N}\)。
  2. Failure Detection：使用 SPOT（基于极值理论的无监督异常检测）识别真异常。SPOT 在初始窗口 \([-L, -T']\) 学习正常 alert 模式，在诊断窗口 \([-T, T']\) 检测异常。
  3. Change Filter：基于专家白名单过滤无关的 changes，只保留 proactive changes。
  4. Event Merge：将 alerts/incidents/changes 合并为统一事件。

4.3 故障相关性挖掘（Failure Correlation Mining, FCM）

基于高频失败关联挖掘（来自 Apriori 算法）：

支持度：

\[ \text{support}(\langle e_a, e_b \rangle) = P(e_a, e_b) = \frac{\text{num}(\langle e_a, e_b \rangle)}{\text{num}(\text{failure pairs})} \]

置信度：

\[ \text{confidence}(\langle e_a, e_b \rangle) = P(e_b | e_a) = \frac{\text{num}(\langle e_a, e_b \rangle)}{\text{num}(e_a)} \]

约束：仅保留满足故障规则树（hierarchical relationships）的高频对，得到 failure knowledge graph \(G_f\)。

4.4 宕机根因分析（Outage Root Cause Analysis, ORCA）

事件因果图构建：

  • 节点：事件 \(e_i\)。
  • 边权重 \(w_{ij} = p_{ij}.\text{conf} \cdot \text{dist}(e_i, e_j)\)，其中 \(\text{dist}(e_i, e_j) = |e_i.\text{sn} \cap e_j.\text{sn}| / |e_i.\text{sn}|\)。
  • 结合历史故障相关性（来自 \(G_f\)）和当前设备连接强度（来自 CMDB）。

根因定位：定制 MAPR（Multi-Attribute Personalized Random Walk）算法：

  1. 初始化每个节点个性化分数 \(u_i = \exp(-l) / \text{dist}(e_i, e_o)\)。
  2. 使用故障相关性 \(w_{ij}\) 作为转移概率。
  3. 多次随机游走，更新 \(u_i\)。
  4. 对每个 outage-related node \(e_i\)，加 incoming edge \(e_o \to e_i\)，权重 \(w_{oi} = 1 / |E^* - 1|\)。
  5. 迭代 L 次后收敛。
  6. 移除 outage 节点 \(e_o\)，按 \(u_i\) 排序得 top-k 根因 \(S_U\)。

故障传播路径推断：基于 PPI（Propagation Probability Inference）：

\[ p_U = \arg\max_{p \in \mathbb{P}} \text{TransPrt}(p_i) = \arg\max_{p \in \mathbb{P}} \prod_{j \in \mathbb{P}(p_i)} u_j \]

6.关键算法详解

6.1 Alert2TS 告警转时序

对每个设备 \(s\)，将 alert 按时间窗口 \(\delta\) 聚合为多变量时序 \(X_s\)。每列对应一种 alert 类型（高 CPU 利用率、磁盘满、内存异常等）。

聚合策略：

  • 数值型 alert：取窗口内的聚合值（sum/avg）。
  • 非数值型 alert：取该时段内出现过的 alert 类型集合（\(r_s^{k,t} = 1\) if 出现过）。
  • 缺失值：填 0（表示该时段内未出现）。

6.2 SPOT 异常检测

SPOT（Streaming Peaks over Threshold）算法：

  1. 在初始窗口 \([-L, -T']\) 学习正常 alert 模式，建立极端值边界。
  2. 在诊断窗口 \([-T, T']\) 动态更新边界。
  3. 超阈值点视为真异常。

6.3 故障关联挖掘（FPM）

基于 Apriori 的高频故障对挖掘：

  1. 从历史数据中统计每个故障事件的频率。
  2. 筛选频率 ≥ \(\alpha \cdot |\mathcal{Q}_1|\) 的事件作为高频单事件。
  3. 对候选对，检查 hierarchical relationship（\(e_i.\text{level} > e_j.\text{level}\)）和支持度。
  4. 输出高频对集合 \(\mathcal{Q}_2\)。

6.4 MAPR 随机游走

MAPR（Multi-Attribute Personalized Random Walk）结合时空信息：

  1. 对每个非 outage 节点，初始化个性化分数 \(u_i\)。
  2. 使用历史故障相关性 + 当前设备连接强度作为转移概率。
  3. 添加 outage 节点到所有相关节点的 incoming edge。
  4. 迭代更新 \(u_i\) 直到收敛。
  5. 移除 outage 节点后，按 \(u_i\) 排序 top-k 节点。

6.5 伪代码

伪代码 1：FPM 故障对挖掘

Algorithm 1: Failure Pairs Mining (FPM)
Input: Event Set T = {T_1, T_2, ...}, Support Threshold α
Output: Frequent Failure Pairs Set Q_2

1: Q_1 ← ∅; Q_2 ← ∅
2: for each event e in T do
3:    if e in Q_1 then e.count ← e.count + 1
4:    else
5:       Q_1 ← Q_1 ∪ {e}
6:       e.count ← 0
7: Q_1 ← {e ∈ Q_1 | e.count ≥ |Q_1| · α}
8: // frequent failure pair mining
9: for i = 1...|Q_1| do
10:   for j = i...|Q_1| do
11:      p_ij ← ⟨e_i, e_j⟩
12:      if e_i.level > e_j.level then
13:         Q_2 ← p_ij
14:         for each failure group T_l in T do
15:            if p_ij ∈ T_l then
16:               p_ij.count ← p_ij.count + 1
17: Q_2 ← {p_ij ∈ Q_2 | p_ij.count ≥ |Q_2| · α}
18: return Q_2

伪代码 2：BSODiag 主流程

Algorithm 2: BSODiag Main Pipeline
Input: Outage Snapshot U
Output: Top-k Root Causes S_U, Failure Propagation Path p_U

1. // 多源故障检测
2. A_ts ← Alert2TS(U.alerts)
3. E_alert ← FailureDetection(A_ts, [-T, T'])
4. E_change ← ChangeFilter(U.changes, whitelist)
5. E_incident ← U.incidents
6. E ← EventMerge(E_alert, E_change, E_incident)
7. // 故障相关性挖掘
8. F ← FailureCorrelationMining(historical_data)
9. G_f ← build_knowledge_graph(F)
10. // 根因分析
11. G_e ← build_event_causal_graph(E, G_f, CMDB)
12. S_U ← MAPR_random_walk(G_e, E_outage_node)
13. p_U ← PPI_path_inference(G_e, S_U[1])
14. return S_U, p_U

7.2 关键阈值

  • 初始窗口：\([-4, -2]\) 小时（outage 前 4-2 小时）。
  • 诊断窗口：\([-2, 0.25]\) 小时（outage 前 2 小时到后 15 分钟）。
  • Alert 聚类窗口 \(\delta = 1\) 分钟。
  • Event merge 窗口 \(\eta = 5\) 分钟。
  • FPM 支持度阈值 \(\alpha = 0.001\)。
  • MAPR 迭代次数 \(L = 100\)。

7.3 评估指标

  • PR@k：Top-k 中包含真实根因的比例，\(\text{PR}@k = \frac{1}{|\mathcal{U}|} \sum_{U \in \mathcal{U}} \mathbb{I}[r_U \in S_U[1:k]]\)
  • MAP：\(\text{MAP} = \frac{1}{k} \sum_{i \leq i \leq k} \text{PR}@i\)
  • PCR（Path Coverage Rate）：传播路径覆盖率，\(\text{PCR} = \frac{1}{|\mathcal{U}|} \sum_{U \in \mathcal{U}} \frac{|p_U \cap \tilde{p}_U|}{|p_U|}\)

8.4 效率分析

BSODiag 完整诊断 24.5 秒（手动 30 分钟），节约 99%+ 时间。最耗时的部分是 MFD（故障检测），因为要做 SPOT、change filter、event merge 等多步操作。

9.消融实验

9.1 Failure Knowledge Graph 消融

去掉 FKG 后 PR@3 80.0% vs 87.5%，PCR 38.4% vs 46.3%，说明历史故障知识图是根因分析的关键。

9.2 CMDB 消融

去掉 CMDB（设备连接）后 PR@3 84.5% vs 87.5%，说明当前设备连接度也有贡献，但不如 FKG 显著。

9.3 MAPR 消融

去掉 MAPR 后 PR@3 82.7%，比 baseline 高，比完整 BSODiag 低 4.8%，说明随机游走机制对根因定位很重要。

  2. 训练：无监督（FCM 挖掘历史数据 + SPOT 在线学习）。

  4. 集成：嵌入阿里云基础设施诊断平台。

11.2 实际效果

BSODiag 在阿里云基础设施系统部署：完整诊断 24.5 秒（手动诊断典型 30 分钟，节约 99%+ 时间）。通过 PR@3 87.5% 和 PCR 46.3% 提升，SRE 能更准确、完整地定位根因并修复故障传播路径中的所有故障点。

12.局限性与未来工作

  1. 数据规模差异大：不同数据中心数据量差异大，影响泛化。
  2. 依赖 CMDB：CMDB 维护不及时会降低效果。
  3. 不处理瞬时告警：偶发告警可能在 SPOT 中被忽略。
  4. 故障类型有限：仅考虑 5 大模块（Storage, Resource Scheduler, Host, Network, Other）。

12.1未来工作

  • 扩展到更多故障类型和模块。
  • 结合 LLM 做根因解释。
  • 支持多数据中心的联邦诊断。
  • 实时增量学习 FKG。

13.总结与启示

BSODiag 是首个面向大规模云基础设施的批服务器宕机全局诊断框架，在阿里云基础设施系统部署并验证有效。核心贡献：

  1. 多源数据整合：MFD 模块利用 alerts/incidents/changes 三源数据。
  2. 故障关联挖掘：FCM 模块基于 Apriori + 专家规则挖掘历史故障关联。
  3. 全局根因分析：ORCA 模块用 MAPR 随机游走 + CMDB 连接强度定位 top-k 根因。
  4. 故障传播路径：PPI 推断完整路径，覆盖率 46.3%。

对实践者的启示：

  1. 多源数据是云基础设施 RCA 的胜负手——单源告警会遗漏关键故障。
  2. 历史故障关联（FKG）比单点告警重要——故障模式具有跨事件相关性。
  3. 设备连接（CMDB）是根因分析的空间先验。
  4. 可解释的传播路径是 OSE 信任系统的关键。

对研究者的启示：

  1. 云基础设施 vs 微服务的 RCA 数据格式差异大，需独立设计。
  2. 无监督方法在故障样本稀缺时是更优选择。
  3. 结合专家规则（hierarchical tree）+ 数据驱动（FPM）是工业界实用范式。
